# Nexus 智能体协同驾驭平台 - 平台架构设计

**版本**: v3.1  
**作者**: 麻子  
**最后更新**: 2026-04-02  

---

## 1. 概述

Nexus 是一个智能体协同驾驭平台，通过"五兄弟"Agent（悟/Grasp、御/Reins、化/Evo、达/Reach、鉴/Vigil）的协同工作，实现复杂的任务处理和智能服务。

---

## 2. 整体架构

### 2.1 分层架构

Nexus 平台采用四层分层架构设计，各层职责明确、边界清晰：

**层级结构**：

- **统一界面层**：作为用户交互的统一入口，聚合所有功能模块，提供 UI、API、CLI 和 Webhook 四种交互方式
- **协作编排层**：负责 Agent 间的协同工作，包括工作流引擎、任务分配、消息路由和 Agent 间通信中枢
- **Agent 执行层**：五兄弟 Agent 的核心执行层，包括认知、工作流、进化、方案和审计五个 Agent
- **基础设施层**：提供基础技术支撑，包括数据库、缓存、消息队列、对象存储和 AI 服务

**分层原则**：
- 各层之间通过标准接口通信
- 下层为上层提供服务，上层调用下层能力
- 层与层之间保持松耦合，便于独立演进

### 2.2 核心组件

**统一界面层组件**：

- **Nexus Console**：Web 管理控制台，提供图形化操作界面
- **Nexus API Gateway**：RESTful 和 gRPC API 网关，统一对外接口
- **Nexus CLI**：命令行工具，支持脚本化自动化操作
- **Nexus Webhook**：外部系统集成接口，支持事件推送

**协作编排层组件**：

- **Reins 工作流引擎**：任务编排和执行控制
- **A2A Hub**：Agent 间消息路由中枢

**Agent 执行层组件**：

- **Grasp（悟）**：认知理解和知识管理
- **Reins（御）**：工作流和执行控制
- **Evo（化）**：经验学习和能力进化
- **Reach（达）**：方案库、复用
- **Vigil（鉴）**：安全和审计

## 3. 统一界面层（Unified Interface Layer）

### 3.1 定位说明

统一界面层**不是新功能层**，而是**五兄弟能力的聚合出口**。所有用户交互都通过这个层级进行统一管理和路由。

**核心价值**：
- **统一管理入口**：用户只需与一个界面交互，无需了解内部五兄弟分工
- **权限统一控制**：统一的认证、授权和审计
- **体验一致性**：统一的 UI/UX 设计语言
- **接口标准化**：对外提供一致 API 格式

### 3.2 UI 聚合

#### 3.2.1 Nexus Console（Web 控制台）

Nexus Console 提供统一的管理界面，聚合五兄弟的所有功能。界面采用模块化的设计理念，将五个 Agent 的功能分别封装为独立的功能模块：

**功能模块布局**：

界面顶部为主功能导航栏，包含四个核心模块：认知中心（Grasp）、工作流中心（Reins）、进化中心（Evo）、方案中心（Reach），底部为安全审计中心（Vigil）。

**各模块功能**：

- **认知中心**：提供知识库管理、认知回流审核、知识图谱可视化等功能
- **工作流中心**：提供工作流编排、执行监控、版本管理等功能
- **进化中心**：提供经验学习、能力评估、模型版本管理等功能
- **方案中心**：提供方案存储、检索、实例化等功能
- **审计中心**：提供操作日志查询、安全审计、异常告警等功能

#### 3.2.2 界面交互流程

用户交互遵循标准流程：用户请求首先到达 Nexus Console，然后通过 API Gateway 路由到对应的 Agent 进行处理，处理完成后将结果返回给用户。

**流程特点**：
- 用户无需了解五兄弟内部职责，统一界面屏蔽了内部复杂度
- 统一的认证和授权机制，所有请求都经过身份验证和权限校验
- 统一的错误处理和反馈格式，确保用户体验一致

### 3.3 API 统一出入口

#### 3.3.1 API Gateway 设计

Nexus API Gateway 作为所有 API 请求的统一入口，提供认证授权、限流控制、日志记录、路由转发和协议转换等核心功能。

**网关架构**：

API Gateway 采用前置代理模式，所有外部请求首先到达网关，网关完成认证授权和限流检查后，根据请求路径路由到对应的 Agent API 服务。各 Agent（Grasp、Reins、Evo、Reach、Vigil）分别提供独立的 API 接口，但对外统一通过网关暴露。

**网关核心能力**：
- **认证授权**：验证用户身份，检查访问权限
- **限流控制**：防止请求过载，保障系统稳定
- **日志记录**：记录所有请求和响应，支持审计
- **路由转发**：根据路径将请求分发到对应的 Agent
- **协议转换**：支持 RESTful 和 gRPC 协议转换

#### 3.3.2 API 设计规范

**统一响应格式**：

所有 API 响应采用统一的结构格式，包含成功状态标识、数据主体、提示信息、时间戳和请求追踪 ID。

**统一错误格式**：

所有错误响应遵循统一的结构，包含失败状态标识、错误对象（含错误码、错误描述、详细信息）、时间戳和请求追踪 ID。

#### 3.3.3 API 标准对标

**对标国际和行业标准**：

- **OpenAPI 3.0**：所有 API 提供 OpenAPI 规范文档，支持接口描述和参数定义
- **RESTful 最佳实践**：资源命名和 HTTP 方法符合 RFC 7231 标准
- **JSON:API**：错误格式和分页格式参考 https://jsonapi.org/ 规范
- **OpenTelemetry**：支持请求追踪和日志格式，实现分布式追踪能力
- **OAuth 2.0**：认证授权支持 JWT 和 Bearer Token 方式
- **Microsoft GraphRAG**：Grasp Agent 的知识查询接口兼容 GraphRAG API 标准

**接口一致性保障措施**：
- 统一使用 application/json 内容类型
- 统一错误码格式，参考 RFC 7807 Problem Details 标准
- 统一时间格式，采用 ISO 8601 标准
- 统一分页格式，包含页码、每页大小和总数三个字段

## 3.5 使用场景

### 3.5.1 场景一：自动化部署流程

**业务场景描述**：用户需要通过工作流自动完成应用的部署流程。

**流程说明**：

1. 用户通过 Nexus Console 或 API 创建部署工作流
2. 界面层验证请求、进行身份鉴权、将请求路由到 Reins 工作流引擎
3. Reins 将部署任务分解为多个子任务：
   - 调用 Reach 的 Git 拉取技能获取源代码
   - 调用 Reach 的测试运行技能执行代码测试
   - 调用 Reach 的云部署技能将应用部署到云服务器
   - 调用 Vigil 进行安全合规检查
4. 所有子任务通过 A2A Hub 进行协调，执行状态实时推送到前端界面
5. 部署完成后，Grasp 将本次部署的经验记录到知识库中
6. Evo 分析部署效率数据，优化未来的部署策略

**统一界面层的价值**：
- 用户只需与统一 API 交互，无需了解内部五兄弟的职责分工
- 统一的认证机制和错误处理方式
- 实时状态推送通过 WebSocket 协议实现

### 3.5.2 场景二：智能客服问答

**业务场景描述**：用户询问产品相关问题，系统需要智能生成回答。

**流程说明**：

1. 用户通过 Web 界面或 API 提交问题
2. 界面层将请求路由到 Grasp 认知引擎
3. Grasp 分析用户意图、查询知识库、调用知识图谱推理
4. 如需要获取外部数据，Grasp 通过 Reins 调用 Reach 的技能
5. Vigil 审计问答记录，确保合规性
6. Evo 学习新的问答模式，优化回答策略
7. 返回统一格式的回答结果

**统一界面层的价值**：
- 提供统一的问答 API 接口，支持 Web、Mobile 和第三方集成
- 统一的认证和配额控制机制
- 支持多语言翻译服务的集成
- 审计日志集中管理

### 3.5.3 场景三：数据分析和报告生成

**业务场景描述**：用户需要生成业务分析报告。

**流程说明**：

1. 用户通过 CLI 工具或 API 请求生成报告
2. 界面层验证请求参数、检查用户配额
3. Reins 创建工作流执行引擎：
   - 调用 Reach 的数据库查询技能提取业务数据
   - 调用 Grasp 分析数据趋势和模式
   - 调用 Evo 选择最佳的图表类型
   - 调用 Reach 的文件系统技能保存报告文件
4. Vigil 检查数据访问权限，确保数据安全
5. 实时推送处理进度到前端界面
6. 完成后通过消息通知用户

**统一界面层的价值**：
- 支持多种输入方式（CLI、Webhook、API）
- 统一的任务状态查询接口
- 支持异步处理和回调通知机制
- 统一的文件下载接口

## 3.6 CLI 命令行入口

### 3.6.1 Nexus CLI 工具

Nexus CLI 提供命令行界面，支持脚本化和自动化操作。命令采用分层设计：

**命令格式**：`nexus <agent> <command> <args>`

**功能分组**：

- **认证模块**：支持令牌登录和登出
- **认知管理（Grasp）**：支持知识列表、添加、查询等操作
- **工作流管理（Reins）**：支持工作流创建、执行、状态查询
- **能力管理（Evo）**：支持能力列表、评估操作
- **方案管理（Reach）**：支持方案列表、检索、实例化操作
- **审计查询（Vigil）**：支持日志过滤、审计搜索

### 3.6.2 CLI 设计原则

- **统一入口**：通过 `nexus` 命令统一访问所有功能
- **子命令分层**：按五兄弟功能分组，对应各 Agent 的职责划分
- **输出格式**：支持 JSON、表格、人类可读等多种输出格式
- **离线模式**：部分功能支持离线缓存，减少网络依赖

## 3.7 Webhook 集成

### 3.7.1 Webhook 事件类型

系统提供五种主要的事件类型，用于外部系统的事件驱动集成：

| 事件类型 | 触发条件 | 数据内容 |
|---------|---------|---------|
| `workflow.started` | 工作流开始执行 | 工作流 ID、触发人、参数 |
| `workflow.completed` | 工作流完成 | 工作流 ID、结果、耗时 |
| `agent.executed` | Agent 执行任务 | Agent 名称、任务 ID、结果 |
| `audit.alert` | 安全告警触发 | 告警级别、描述、详情 |
| `knowledge.update` | 知识更新 | 知识 ID、操作类型、版本 |

### 3.7.2 Webhook 配置

Webhook 配置包含目标 URL、订阅的事件类型列表、安全密钥和启用状态。系统支持为不同的外部系统配置不同的事件订阅组合。

## 3.8 统一界面层设计原则

1. **透明性**：用户无需了解五兄弟 Agent 的内部分工，统一界面屏蔽内部复杂度
2. **一致性**：提供统一的认证、授权和错误处理机制
3. **可扩展性**：新增 Agent 时只需在网关层添加路由规则，不影响现有功能
4. **可观测性**：提供统一的日志记录和监控指标，便于问题排查和性能分析

### 4.2 A2A Hub（Agent-to-Agent Hub）

#### 4.2.1 功能定位

A2A Hub 是五兄弟 Agent 之间通信的核心枢纽，负责消息路由、状态同步、事件分发等关键功能。

#### 4.2.2 部署架构选项

系统提供三种部署架构方案，适应不同规模和需求的场景：

**方案一：集中式部署**（适合中小型系统）

集中式部署采用单实例架构，所有 Agent 连接到一个中央 Hub 实例及其关联数据库。

**特点**：
- 架构简单、部署成本低
- 适合并发 Agent 数量小于 100 的场景
- 存在单点故障风险，需要定期备份

**方案二：集群部署**（推荐，适合中大型系统）

集群部署采用负载均衡 + 多实例 Hub + 分布式缓存和数据库的架构。负载均衡器将请求分发到多个 Hub 实例，使用 Redis 进行消息缓存，MySQL 进行数据持久化。

**特点**：
- 高可用设计，支持自动故障转移
- 支持水平扩展，可根据负载增加实例
- 使用 Redis 做消息缓存、MySQL 做持久化

**方案三：事件驱动架构**（适合超大规模系统）

事件驱动架构采用消息队列（如 Apache Kafka 或 RabbitMQ）作为核心通信基础设施，多个 Hub 实例订阅和发布消息，各 Agent 通过事件总线进行通信。

**特点**：
- 完全解耦，支持异步通信
- 支持事件溯源，便于审计和恢复
- 适合分布式、多租户场景

#### 4.2.3 部署建议

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 个人/小型项目 | 集中式部署 | 架构简单、成本低廉 |
| 企业级应用 | 集群部署 | 高可用性、易于扩展 |
| 超大规模/多租户 | 事件驱动架构 | 完全解耦、弹性伸缩、高性能 |

#### 4.2.4 配置核心要素

集群部署配置主要包含以下核心要素：

- **集群配置**：集群名称、节点列表（包含主机、端口、优先级）
- **存储配置**：缓存存储（Redis 主机列表、密码、数据库号）、持久化存储（MySQL 连接信息）
- **消息配置**：队列大小、消息生存时间、重试次数、重试延迟

### 4.3 消息路由协议

#### 4.3.1 消息格式

Agent 间通信采用统一的消息格式，包含消息 ID、消息类型、发送方、接收方、消息负载、时间戳、关联 ID 和生存时间等字段。

**消息字段说明**：

- **ID**：消息唯一标识符
- **类型**：消息类别（任务请求、任务响应、状态更新、事件）
- **发送方**：消息来源 Agent（Grasp/Reins/Evo/Reach/Vigil）
- **接收方**：目标 Agent
- **负载**：消息具体内容和参数
- **时间戳**：消息发送时间
- **关联 ID**：用于关联请求和响应的追踪 ID
- **生存时间**：消息有效时长（秒）

#### 4.3.2 路由规则

A2A Hub 根据消息类型采用不同的路由策略：

| 消息类型 | 路由策略 | 示例说明 |
|---------|---------|---------|
| `task_request` | 基于任务类型路由到对应 Agent | 工作流任务路由到 Reins Agent |
| `task_response` | 根据关联 ID 路由回发起方 | 响应消息返回给 Grasp Agent |
| `status_update` | 广播到所有订阅者 | 工作流状态更新广播给所有相关方 |
| `event` | 基于事件类型订阅/分发 | 知识更新事件分发给 Evo Agent |

## 7. 安全与审计

### 7.1 认证授权

- **认证方式**：JWT / OAuth2 / API Key
- **授权模型**：RBAC（基于角色的访问控制）
- **多租户支持**：租户隔离、资源配额

### 7.2 审计日志

见 `05-vigil-architecture.md` 详细设计。

**日志分级策略**：

| 级别 | 保留时间 | 用途 |
|------|---------|------|
| 关键审计日志 | 3 年 | 合规、追责 |
| 操作日志 | 1 年 | 问题排查 |
| 访问日志 | 30 天 | 性能分析 |
| 调试日志 | 7 天 | 开发调试 |

**日志脱敏**：
- 敏感字段（密码、Token、个人信息）自动脱敏
- 支持动态脱敏配置

### 8.2 生产环境部署

#### 8.2.1 Kubernetes 部署

Nexus 采用容器化部署方式，推荐使用 Kubernetes 作为容器编排平台。每个 Agent 作为独立的 Deployment 进行部署，通过 Service 暴露内部服务。

**部署配置要点**：

- **副本数量**：建议每个 Agent 至少部署 3 个副本，保证高可用性
- **资源请求和限制**：为每个容器设置 CPU 和内存的请求值和限制值
- **端口暴露**：通过容器端口暴露服务接口
- **镜像管理**：使用版本化镜像标签，便于回滚和追踪

#### 8.2.2 高可用配置

生产环境高可用配置包括以下核心要素：

- **多副本部署**：每个 Agent 至少部署 2 个副本，避免单点故障
- **负载均衡**：使用 Nginx 或 HAProxy 进行请求分发
- **自动扩缩容**：基于 CPU 使用率、内存使用率、消息队列积压情况自动调整副本数
- **故障自动转移**：通过健康检查机制 + 自动重启策略实现故障恢复

### 8.3 云原生部署

针对主流云厂商的推荐部署方案：

| 云厂商 | 推荐方案 | 备注说明 |
|--------|---------|---------|
| AWS | EKS + RDS + ElastiCache | 成熟稳定，生态完善 |
| Azure | AKS + Azure SQL + Cache for Redis | 与企业系统集成良好 |
| 阿里云 | ACK + RDS + Redis | 国内访问速度快 |
| 腾讯云 | TKE + TDSQL + Redis | 性价比高，适合国内使用 |

## 四、核心信念

理解这四个信念，就理解了 Nexus 的设计哲学：

### 信念一：智能体不好用的根因是认知，不是记忆

```
❌ 主流做法：用户不断聊 → 智能体记住偏好 → 看起来变好了
✅ 我们的做法：悟提供基础认知 → 智能体在工作中产生新认知 → 回流悟 → 所有智能体共享
```

**认知不是一次性装好的静态配置，而是在使用中持续生长的活体。**

### 信念二：驾驭是能力，数据是飞轮

编排让智能体协同，数据让智能体持续变强。

```
智能体在平台上协同 → 产生运行数据 → Evo 提取经验 → 成熟方案沉淀到 Reach
→ 吸引更多用户 → 更多运行数据 → 飞轮越转越快
```

### 信念三：自举是灵魂

双螺旋上升：任务产出质量 ↑ ↔ 智能体自身能力 ↑

三个自举闭环：
1. **执行反馈闭环**：执行 → 对比预期 → 发现差距 → 调整策略
2. **能力沉淀闭环**：好的策略 → 提取为胶囊 → 固化到能力库 → 下次直接调用
3. **跨智能体传播闭环**：智能体 A 学到经验 → 胶囊共享 → 智能体 B 也获得这个能力

### 信念四：安全是一切的基石

```
❌ 主流做法：智能体跑起来 → 出安全问题 → 打补丁
✅ 我们的做法：每个操作都有审计 → 每次进化都有扫描 → 每条认知都有验证
```

Vigil 核心理念：
- 信任需要积累：智能体从低信任起步，通过可靠行为逐步获得更高权限
- 全程可审计：所有操作有完整日志，可追溯、可回放
- 实时防护：执行时实时拦截越权操作
- 安全与效率并重：高信任智能体获得更大自由度

## 五、五大核心能力

Nexus 必须提供的五个核心能力：

| 能力 | 描述 | 衡量标准 |
|------|------|----------|
| **高效使用** | 把任务给到智能体，它就能理解并执行 | 任务首次理解正确率 ≥ 90%，从下达目标到开始执行 ≤ 30 秒 |
| **自动化** | 人给一个目标，智能体自己拆解、分配、执行、验收 | 端到端自动化率 ≥ 80%，异常自愈率 ≥ 70% |
| **持续工作** | 7×24 持续运行 | 系统可用性 ≥ 99.9%，平均故障恢复时间 ≤ 5 分钟 |
| **自举进化** | 每完成一次任务，智能体能力提升 | 同类任务第 N 次执行质量 > 第 1 次，经验复用率 ≥ 60% |
| **可信安全** | 每个智能体的行为都可审计、可追溯、可控制 | 越权操作拦截率 100%，敏感信息泄露率 0% |

## 六、MVP 落地节奏

### 四个阶段

| 阶段 | 时间 | 落地内容 | 目标 |
|------|------|---------|------|
| **Phase 1** | MVP | Grasp P0 + Reins P0 + Vigil P0 | 认知 + 驾驭 + 安全基础，核心体验跑通 |
| **Phase 2** | MVP+1 | Evo P0 + 各层 P1 | 自举闭环形成，数据飞轮启动 |
| **Phase 3** | V1.0 | Reach P1 + 各层 P2 | 生态层上线，方案可复用 |
| **Phase 4** | V2.0 | 全量 P2/P3 | 全功能，规模化 |

### 核心原则

1. **认知和驾驭不能分开**：没有认知的驾驭是"管一群不懂行的新兵"
2. **安全从第一天就在**：Vigil 的 P0 功能随 Phase 1 一起上线
3. **进化是闭环的关键**：Phase 2 补上 Evo 后，数据飞轮才开始转
4. **生态需要数据支撑**：Reach 等飞轮转起来、有真实运行数据后再上线

## 七、技术原则

### 协议拥抱标准

- **MCP**（Model Context Protocol）：智能体 ↔ 工具/数据源
- **A2A**（Agent-to-Agent）：智能体 ↔ 智能体
- **不造轮子**，确保广泛适应性

### 不改 OpenClaw 代码

OpenClaw 是外部依赖，更新太快。Nexus 只调用 OpenClaw API，不修改其源码。

### Paperclip 是 Reins 的主体

要改要扩展的是 Paperclip（作为 Reins Server 原型），这是自己的产品。

### Evo 引擎可借鉴代码

`@evomap/evolver` npm 包可用，但文档中不出现 "Evolver" 名字，统一称 "Nexus Evo" 或 "Evo 引擎"。

## 十一、常见问题

### Q: Nexus 和 OpenClaw 是什么关系？

A: OpenClaw 是智能体运行时（类似操作系统），Nexus 是驾驭平台（类似任务调度器 + 能力市场）。Nexus 调用 OpenClaw API 来管理和调度智能体。

### Q: Nexus 和 CrewAI、AutoGen 有什么区别？

A: CrewAI/AutoGen 是智能体编排框架，Nexus 是完整的驾驭平台。差异在于：
- Nexus 有认知层（Grasp），让智能体从根上懂行
- Nexus 有进化层（Evo），让智能体持续变强
- Nexus 有安全层（Vigil），全程可审计可控制
- Nexus 有生态层（Reach），成熟方案可复用
