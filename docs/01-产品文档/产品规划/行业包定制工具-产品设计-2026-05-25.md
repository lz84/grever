## 一、产品定位

### 是什么

> **将行业包定制从手艺活变成生产线的 B2B 内部工具。**

### 用户是谁

**不是终端客户，是咨询团队。**

| 用户角色 | 使用场景 | 使用频率 |
|---------|---------|---------|
| 咨询顾问（初期是你自己） | 帮客户定制行业包 | 每次定制项目 |
| 客户 IT 管理员 | 日常维护，微调参数 | 偶尔 |
| 合作伙伴（未来） | 用你的工具帮他们的客户定制 | 每次定制项目 |

### 解决什么问题

| 现状 | 问题 | 工具解决 |
|------|------|---------|
| 手动改 YAML/JSON/DB | 效率低，容易出错 | 可视化编辑，自动生成 |
| 每个园区都不一样 | 无法复用，每次从头来 | 标准骨架 + 参数化定制 |
| 只有你能做 | 你忙了就做不了 | 工具化，别人也能用 |

## 二、核心概念

### 标准骨架 vs 定制皮肤

```
标准骨架（标准行业包提供）：
  - 通用流程（泄漏→检测→评估→响应→恢复）
  - 通用决策规则（Ⅰ/Ⅱ/Ⅲ 级响应条件）
  - 通用角色定义（总指挥/现场指挥/技术专家）
  - 通用能力标签库

定制皮肤（定制工具生成）：
  - 客户具体流程（他们的预案怎么写）
  - 客户决策参数（泄漏量 > ? 吨启动 Ⅰ 级 → 客户自己的阈值）
  - 客户组织架构（具体是谁）
  - 客户设施信息（储罐位置、危化品种类、敏感目标）
```

### 定制的本质

**定制 = 标准骨架 + 参数覆盖 + 流程扩展**

不是从零开始写，而是在标准骨架上做差异化。

## 三、五个核心模块

### 模块 1：预案解析器

**目标**：把客户的 Word/PDF 预案文档变成结构化场景草案。

#### 输入

| 类型 | 格式 | 示例 |
|------|------|------|
| 应急预案 | .docx / .pdf | 《XX 化工园区危化品事故应急预案》 |
| 组织手册 | .docx / .xlsx | 《XX 园区应急组织架构及联系方式》 |
| 设施清单 | .xlsx / .csv | 《XX 园区危化品存储设施清单》 |

#### 处理流程

```
1. 文档上传
  ↓
2. LLM 智能解析（GPT-4/Claude）
  - 提取事故分级标准
  - 提取响应流程步骤
  - 提取角色定义和职责
  - 提取联系方式和通知链
  - 提取特殊规则和条件
  ↓
3. 结构化输出（YAML 草案）
  ↓
4. 人工审核 + 修正
```

#### 输出

```yaml
# 解析结果草案（需要人工审核）
scenario_draft:
  name: "XX园区危化品泄漏处置"
  source_document: "XX园区危化品事故应急预案.docx"
  confidence: 0.85  # LLM 解析置信度
  
  phases:
    - name: "事故发现"
      tasks:
        - name: "发现泄漏"
          description: "巡检人员发现储罐区有异味"
          role: "巡检员"
          
    - name: "事故报告"
      tasks:
        - name: "上报班长"
          description: "立即报告当班班长"
          role: "巡检员"
          condition_type: human_input
          
    - name: "事故评估"
      tasks:
        - name: "评估泄漏等级"
          description: "根据泄漏量和物质类型评估响应级别"
          role: "技术专家"
          decision_rules:
            - condition: "泄漏量 > 1 吨"
              action: "启动 Ⅰ 级响应"
            - condition: "泄漏量 0.1-1 吨"
              action: "启动 Ⅱ 级响应"
            - condition: "泄漏量 < 0.1 吨"
              action: "启动 Ⅲ 级响应"
    
    - name: "应急响应"
      tasks:
        - name: "启动应急预案"
          description: "总指挥审批后启动相应级别预案"
          role: "总指挥"
          condition_type: human_decision
    
    - name: "恢复"
      tasks:
        - name: "事故调查"
          description: "调查事故原因，形成报告"
          role: "安全部门"
```

#### 置信度标注

解析结果每个部分标注置信度，低置信度的部分高亮提醒人工审核：

| 置信度 | 含义 | 处理 |
|--------|------|------|
| > 0.9 | 高置信 | 自动采用 |
| 0.7-0.9 | 中置信 | 建议审核 |
| < 0.7 | 低置信 | 必须人工确认 |

### 模块 2：场景编辑器

**目标**：在标准场景骨架上进行可视化定制编辑。

#### 界面布局

```
┌─────────────────────────────────────────────────────────┐
│  场景编辑器 - XX园区危化品泄漏处置                         │
├──────────────┬──────────────────────────┬───────────────┤
│  标准场景模板 │      定制编辑区           │  属性面板      │
│              │                          │               │
│  ☑ 泄漏发现  │  ┌─────────────────┐     │  当前选中:     │
│  ☑ 事故报告  │  │ 事故发现          │     │  评估泄漏等级   │
│  ☑ 事故评估  │  │     ↓            │     │               │
│  ☑ 应急响应  │  │ 事故报告          │     │  能力标签:     │
│  ☑ 恢复      │  │     ↓            │     │  ☑ hazmat-id   │
│              │  │ 事故评估          │     │  ☑ msds-parse  │
│  差异对比:    │  │     ↓            │     │               │
│  + 新增 2 个  │  │ 应急响应          │     │  HITL:         │
│  ~ 修改 3 个  │  │     ↓            │     │  ☑ human_dec   │
│  - 删除 1 个  │  │ 恢复              │     │               │
│              │  └─────────────────┘     │  决策规则:     │
│              │                          │  泄漏量 > ? 吨   │
│              │  工具栏:                  │  → Ⅰ 级响应    │
│              │  [添加节点][删除][连线]    │               │
│              │  [HITL][绑定能力]         │               │
├──────────────┴──────────────────────────┴───────────────┤
│  底部状态栏: 已保存 | 验证通过 | 差异: +2 ~3 -1           │
└─────────────────────────────────────────────────────────┘
```

#### 核心操作

| 操作 | 说明 |
|------|------|
| **选择标准场景** | 从标准行业包选一个基础场景作为起点 |
| **添加/删除节点** | 增删 Task，标准场景没有的节点标注为"新增" |
| **修改节点属性** | 改描述、改角色、改决策规则 |
| **调整依赖关系** | 拖拽连线，调整 Task 执行顺序 |
| **设置 HITL 节点** | 标记哪些环节需要人审批 |
| **绑定能力标签** | 每个 Task 需要什么能力，从行业标签库选 |
| **对比标准版** | 实时显示与标准场景的差异 |

#### 验证规则

编辑过程中实时验证：

| 验证项 | 规则 | 错误提示 |
|--------|------|---------|
| 连通性 | 场景流程是连通图 | "存在孤立节点：XX" |
| 无环 | 依赖关系无环 | "存在循环依赖：A→B→A" |
| 能力匹配 | 所有 Task 的能力标签都有 Agent 可匹配 | "Task XX 需要 hazmat-id，无匹配 Agent" |
| HITL 完整性 | 所有 human_decision/human_input 都有对应角色 | "HITL 节点 XX 没有绑定人类角色" |
| 决策规则一致性 | 不会出现同时触发多个互斥条件 | "Ⅰ 级和 Ⅱ 级响应条件有重叠" |

### 模块 4：设施参数配置器

**目标**：录入客户的设施信息，作为场景运行的参数。

#### 参数分类

| 分类 | 参数 | 用途 |
|------|------|------|
| 危化品存储 | 储罐编号、物质名称、存储量、危险等级 | 泄漏场景计算基础 |
| 地理信息 | 园区坐标、边界、周边居民区/学校/医院 | 扩散范围计算 + 疏散决策 |

| 气象站 | 风速/风向传感器位置 | 实时扩散计算 |
| 敏感目标 | 学校、医院、居民区、水源地 | 疏散优先级 |

#### 数据关联

设施参数不是孤立数据，会被场景引用：

```
场景 Task "计算扩散范围":
  - 读取泄漏物质 → 从危化品存储表获取物质属性
  - 读取气象数据 → 从气象站获取实时风速风向
  - 读取敏感目标 → 计算是否在扩散范围内
  - 输出疏散建议 → 基于敏感目标人数和距离
```

## 五、定制工具的商业模式

### 三种使用方式

| 方式 | 用户 | 定价 |
|------|------|------|
| **自用（内部工具）** | 你/你的团队用来服务客户 | 计入定制项目成本，不单独定价 |
| **合作伙伴授权** | 咨询公司/集成商用它做定制 | ¥2-5 万/年（工具授权费，服务费另算） |
| **企业版内置** | 大型企业客户自己定制行业包 | 包含在企业版价格中 |

> **"自用免费"的矛盾已解决**：自用不是"免费"，而是将工具成本计入定制服务费。合作伙伴授权只收工具授权费，是因为他们还买了你的定制咨询服务（服务费覆盖了你投入的人力）。

### 为什么这是护城河

| 别人能抄 | 别人抄不走 |
|---------|-----------|
| 开源代码（DAG 调度、能力匹配） | 定制工具 + 标准骨架 + 定制流程 |
| 场景文件格式 | 预案解析的 LLM prompt + 经验 |
| 标签体系 | 行业知识积累（哪个场景对应什么规则） |

**定制工具的本质竞争力不是代码，是"怎么把一个非结构化的预案变成结构化场景"的经验和流程。**

## 六、开发路线图

### Phase 1：手动定制（现在 - 1 个月）

**不开发工具，先手动做。**

| 任务 | 产出 |
|------|------|
| 手动帮 1-2 个客户定制行业包 | 摸清定制流程 |
| 记录手动定制的每一步 | 提炼标准流程 |
| 打磨标准骨架 | 确保骨架能覆盖 80% 的客户 |

**这是最重要的阶段。** 不知道定制流程就开发工具，做出来的工具一定不对。

### Phase 2：半自动工具（1-3 个月）

**先做最常用的两个模块。**

| 模块 | 优先级 | 原因 |
|------|--------|------|
| 场景编辑器 | 🔴 最高 | 定制工作量的 60% |
| 设施参数配置器 | 🟡 高 | 定制工作量的 20% |

| 功能 | 状态 |
|------|------|
| 可视化场景编辑 | ✅ |
| 设施参数录入 | ✅ |
| 差异对比 | ✅ |
| 验证 | ✅ |
| 导出 .nexus | ✅ |

**这个阶段不用 LLM 解析，客户的东西还是手动录入。**

### Phase 3：智能解析（3-6 个月）

**加入 LLM 预案解析。**

| 模块 | 功能 |
|------|------|
| 预案解析器 | Word/PDF → YAML 草案 |
| 组织架构映射器 | 支持 Excel 导入 |
| 置信度标注 | 自动标记需要人工审核的部分 |

### Phase 4：全自动流水线（6-12 个月）

**上传文档 → 生成行业包。**

| 功能 | 说明 |
|------|------|
| 一键导入 | 上传预案文档 → 自动解析 → 自动生成场景草案 → 人工微调 → 发布 |
| 批量定制 | 一次定制多个场景 |
| 模板市场 | 社区贡献的定制模板 |

## 七、数据格式规范

### .nexus 文件格式

实际是 ZIP 压缩包，包含：

```
custom-pack-v1.0.nexus
├── manifest.json          # 包元数据
├── tags.yaml              # 能力标签定义（含定制扩展）
├── scenarios/
│   ├── leak-response.yaml  # 定制后的场景
│   └── fire-response.yaml
├── skills/
│   └── ...                 # 定制技能
├── knowledge/
│   └── ...                 # 行业知识
├── agent-profiles/
│   └── ...                 # 定制 Agent 方案
├── facilities/
│   └── params.yaml         # 设施参数
└── org-chart/
    └── mapping.yaml        # 组织架构映射
```

### manifest.json

```json
{
  "format_version": "1.0",
  "pack_id": "custom-chem-park-xx-2026",

    "scenarios_modified": 3,
    "scenarios_removed": 0,
    "tags_added": 5,
    "agents_configured": 5
  }
}
```

## 八、风险与对策

| 风险 | 可能性 | 影响 | 对策 |
|------|--------|------|------|
| 预案文档格式千奇百怪 | 高 | 高 | Phase 1 手动积累，建立文档模板库 |
| 客户不愿提供详细设施信息 | 中 | 高 | 提供信息脱敏方案，只录关键参数 |
| 定制后场景与标准版差异太大 | 中 | 中 | 记录差异，定期更新标准骨架 |
| LLM 解析错误导致安全隐患 | 中 | 极高 | 所有解析结果必须人工审核，置信度 < 0.9 必须确认 |
| 工具依赖 Nexus 核心功能未开发 | 高 | 高 | 工具独立开发，通过 .nexus 格式交互 |
