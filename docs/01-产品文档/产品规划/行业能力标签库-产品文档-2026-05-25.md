## 一、定位：能力标签匹配体系的能力定义模块

### 一句话定义

> **行业能力标签库 = 能力标签匹配体系的能力定义模块。**

它不是一个独立的系统。它和 Agent 的 `capability_tags` 字段、Task 的 `capability_tags` 字段、匹配引擎，**共同构成一个完整的能力标签匹配体系**。

### 体系结构

```
┌─────────────────────────────────────────────────────────┐
│              能力标签匹配体系（一个系统）                    │
│                                                         │
│  ┌──────────────────────────┐    ┌──────────────────┐   │
│  │  行业能力标签库（定义层）   │    │  匹配引擎（使用层） │   │
│  │                          │    │                  │   │
│  │  定义：什么是"危化品识别"   │───→│  Agent 有这个能力  │   │
│  │  定义：什么是"扩散建模"     │───→│  Task 需要这个能力 │   │
│  │  定义：维度/依赖/工具      │    │  匹配引擎找交集    │   │
│  │                          │    │                  │   │
│  └──────────────────────────┘    └──────────────────┘   │
│                                                         │
│  两者通过标签 ID 字符串关联，不是两个系统，是一个系统的两  │
│  个模块：一个管"标准定义"，一个管"运行时匹配"。              │
└─────────────────────────────────────────────────────────┘
```

### 模块分工

| 模块 | 职责 | 数据结构 | 谁写的 |
|------|------|---------|--------|
| **能力定义层**（行业标签库） | 定义标签是什么、属于哪个维度、有什么依赖、关联什么工具 | `industry_capability_tags` 表 | 行业专家 |
| **能力引用层**（Agent/Task） | Agent 有哪些能力、Task 需要什么能力 | `capability_tags`（JSON object） | 场景作者 / 匹配引擎 / 心跳 |
| **匹配引擎** | 用 Agent 的能力交集打分 | 读 `capability_tags` 字符串 | 调度器 tick |

### 关键约束

1. **行业标签库不直接参与匹配打分**。匹配只看 `capability_tags` 字符串的交集。
2. **标签库提供标准 ID**。避免拼写不一致（`hazmat-id` vs `hazmat_identification`）。
3. **标签 ID 映射到四维**。`chem:hazmat-identification` → `professional` 维度。
4. **一个系统，不是两个**。行业标签库是能力标签匹配体系的组成部分，不是外部字典。

## 二、什么是行业能力标签库（能力定义模块）

行业能力标签库是 Nexus 行业包的核心组件之一，定义了**一个行业内所有 Agent 能力的标准化标签体系**。

### 解决什么问题

没有标签库时：

```
场景说：这个 Task 需要"危化品识别"能力
Agent 说：我有"hazmat"能力
系统说：这两个是一个东西吗？不知道。→ 匹配失败
```

有了标签库后：

```
标签库定义：
  "hazmat-identification" = 危化品识别

场景声明：requires: ["hazmat-identification"]
Agent 声明：has: ["hazmat-identification"]
系统：精确匹配 → 派发成功
```

### 一句话定义

> **行业能力标签库是行业的"AI 协作语言"。** 场景用它说"需要什么"，Agent 用它说"能做什么"，系统用它来做匹配。

## 二、标签体系架构

### 两层标签体系

```
┌──────────────────────────────────────────┐
│           通用四维标签（已有）              │
│  business / professional / technical /    │
│  management                               │
│  ── 所有行业共用，定义能力的"大类"         │
└────────────────────┬─────────────────────┘
                     │
┌────────────────────┴─────────────────────┐
│           行业能力标签库（新增）            │
│  每个行业一套，定义具体的能力标签          │
│  ── 行业包的核心交付物                    │
└──────────────────────────────────────────┘
```

### 标签格式

```
{行业缩写}:{能力标识符}

示例：
  chem:hazmat-identification      （化工：危化品识别）
  chem:diffusion-modeling         （化工：扩散建模）
  chem:emergency-response-level   （化工：应急响应分级）
  dev:python-code-review          （开发：Python 代码审查）
  dev:java-security-audit         （开发：Java 安全审计）
```

### 标签的完整结构

每个标签在标签库中是一个实体：

```yaml
tag_id: "chem:hazmat-identification"
name: "危化品识别"
description: "识别危险化学品类型、危害类别、处置方法"
dimension: "professional"          # 映射到通用四维之一
level: "basic"                      # basic / intermediate / advanced
prerequisites: []                   # 依赖的前置标签
tools: ["msds-parser", "hazmat-db"] # 关联的工具
examples:                           # 示例场景
  - "识别泄漏物质为液氯，Ⅱ类危险品"
  - "根据 MSDS 判断泄漏处置方案"
```

## 三、通用四维 × 行业标签映射

### 四维定义

| 维度 | 含义 | 标签前缀规则 |
|------|------|-------------|
| **business** | 业务领域知识 | `biz:` |
| **professional** | 专业技能/行业专长 | `prof:` 或直接行业前缀 |
| **technical** | 技术工具/框架能力 | `tech:` |
| **management** | 管理/协调能力 | `mgmt:` |

### 映射规则

行业标签必须映射到四维之一，这样跨行业的 Agent 才能被部分匹配。

```
示例：化工应急行业

business 维度：
  biz:chemical-park-operations    化工园区运营知识
  biz:emergency-management        应急管理体系

professional 维度：
  chem:hazmat-identification      危化品识别
  chem:diffusion-modeling         扩散建模
  chem:emergency-response-level   应急响应分级
  chem:evacuation-planning        疏散规划

technical 维度：
  chem:msds-parsing               MSDS 文档解析
  chem:leak-rate-calculation      泄漏速率计算
  chem:weather-analysis           气象数据分析

management 维度：
  chem:incident-command           事故指挥
  chem:resource-allocation        资源调配
  chem:multi-agency-coordination  多部门协调
```

### 跨行业复用的能力

有些能力跨行业通用，不需要重新定义：

```
通用能力（所有行业可用）：
  - code-review         （代码审查）
  - document-analysis   （文档分析）
  - data-validation     （数据校验）
  - report-generation   （报告生成）
```

## 四、体系内部结构：定义层与引用层

### 一个系统，两个层级

```
┌─────────────────────────────────────────────────┐
│  定义层（行业标签库）                              │
│  ─ 标签的标准定义：ID / 名称 / 描述 / 维度 / 工具  │
│  ─ 类比：化学品安全数据表（MSDS）                   │
│  ─ 谁写的：行业专家                                │
└──────────────────────┬──────────────────────────┘
                       │ 提供标准标签 ID
                       ▼
┌─────────────────────────────────────────────────┐
│  引用层（Agent/Task 的 capability_tags）           │
│  ─ 标签的使用实例：字符串 ID 的引用               │
│  ─ 类比：采购清单（写 MSDS 编号，不需要抄全文）     │
│  ─ 谁写的：场景作者 / 匹配引擎 / Agent 心跳        │
└─────────────────────────────────────────────────┘
```

### 匹配引擎如何工作

匹配引擎读引用层，不读定义层：

```
Task.capability_tags:
  professional: ["chem:hazmat-identification", "chem:msds-parsing"]

Agent.capability_tags:
  professional: ["chem:hazmat-identification", "chem:diffusion-modeling", ...]

匹配逻辑：
  交集 = ["chem:hazmat-identification"]  ← 匹配
  缺失 = ["chem:msds-parsing"]           ← 不匹配

定义层（标签库）的作用：
  1. 提供标准 ID（避免拼写不一致）
  2. 提供标签的维度/级别/工具信息（前端展示）
  3. 提供标签的依赖关系（校验 Task 是否配齐了 prerequisites）
```

### 权重如何存储

| 存储位置 | 内容 | 生效时机 |
|---------|------|---------|
| `agent_tag_weights` 表 | Agent × 标签 → 权重 | 匹配打分 |
| `industry_capability_tags` 表 | 标签的元数据（描述/工具/示例） | 展示，不参与打分 |

## 五、标签库的数据模型（DB Schema）

```sql
-- 行业能力标签库
CREATE TABLE industry_capability_tags (
    id              TEXT PRIMARY KEY,         -- "chem:hazmat-identification"
    industry        TEXT NOT NULL,             -- "chemical-emergency"
    tag_name        TEXT NOT NULL,             -- "危化品识别"
    tag_name_en     TEXT,                      -- "Hazardous Material Identification"
    description     TEXT NOT NULL,             -- 能力描述
    dimension       TEXT NOT NULL,             -- business/professional/technical/management
    level           TEXT DEFAULT 'basic',      -- basic/intermediate/advanced
    prerequisites   TEXT,                      -- JSON 数组：依赖的前置标签
    tools           TEXT,                      -- JSON 数组：关联的工具名称
    examples        TEXT,                      -- JSON 数组：使用示例
    created_at      INTEGER,
    updated_at      INTEGER
);

-- 行业包元数据
CREATE TABLE industry_packs (
    id              TEXT PRIMARY KEY,
    name            TEXT NOT NULL,             -- "化工应急行业包"
    industry        TEXT NOT NULL,             -- "chemical-emergency"
    version         TEXT NOT NULL,             -- "1.0.0"
    description     TEXT,
    tags_count      INTEGER DEFAULT 0,         -- 标签数量
    scenarios_count INTEGER DEFAULT 0,         -- 场景数量
    skills_count    INTEGER DEFAULT 0,         -- 技能数量
    status          TEXT DEFAULT 'draft',      -- draft/published/deprecated
    pack_type       TEXT DEFAULT 'standard',  -- standard / custom（Phase 2 实现）
    base_pack_id    TEXT,                    -- 定制包基于哪个标准包（Phase 2 实现）
    created_at      INTEGER,
    updated_at      INTEGER
);

-- 行业包内容关联
CREATE TABLE industry_pack_contents (
    pack_id         TEXT NOT NULL,
    content_type    TEXT NOT NULL,             -- tag/scenario/skill/knowledge/agent
    content_id      TEXT NOT NULL,             -- 对应实体的 ID
    PRIMARY KEY (pack_id, content_type, content_id)
);
```

## 五、标签的生命周期

### 创建流程

```
1. 定义行业 → 确定行业缩写（如 chem:）
2. 梳理能力 → 罗举该行业所有需要的 AI 能力
3. 归类维度 → 每个标签映射到 business/professional/technical/management
4. 定义依赖 → 哪些能力是前置的（如先会 MSDS 解析，才能做危化品识别）
5. 关联工具 → 哪些工具/脚本提供这个能力
6. 写示例 → 2-3 个具体使用场景
7. 审核发布 → 标签库版本化发布
```

### 版本管理

| 版本类型 | 条件 | 影响 |
|---------|------|------|
| **MAJOR** | 标签删除、维度变更 | 可能破坏已有场景和 Agent 匹配 |
| **MINOR** | 新增标签 | 向后兼容，已有场景不受影响 |
| **PATCH** | 描述/示例/工具更新 | 完全兼容 |

### 标签的状态

| 状态 | 含义 |
|------|------|
| `active` | 正常使用 |
| `deprecated` | 不推荐新建使用，但已有场景继续支持 |
| `replaced_by` | 被新标签替代，指向新标签 ID |

## 六、标签与场景/技能/Agent 的关系

### 场景声明所需标签

```yaml
# 场景：危化品泄漏处置
scenario:
  name: "危化品泄漏处置"
  tasks:
    - name: "评估泄漏等级"
      requires:
        - chem:hazmat-identification
        - chem:msds-parsing
      condition_type: auto_eval
    
    - name: "计算扩散范围"
      requires:
        - chem:diffusion-modeling
        - chem:weather-analysis
      condition_type: human_decision  # 需要人审批
    
    - name: "制定疏散方案"
      requires:
        - chem:evacuation-planning
        - chem:incident-command
      condition_type: auto_eval
```

### 技能声明提供标签

```yaml
# 技能：MSDS 解析器
skill:
  name: "MSDS Parser"
  provides:
    - chem:msds-parsing
  tools:
    - name: "msds-parser"
      type: python
      entry_point: "msds_parser.py:parse"
  verification:
    - type: "unit_test"
      script: "test_msds_parser.py"
```

### Agent 声明拥有标签

```yaml
# Agent：现场评估 Agent
agent_profile:
  name: "现场评估 Agent"
  role: "现场应急处置"
  has:
    - chem:hazmat-identification    # 权重 1.0
    - chem:diffusion-modeling       # 权重 0.8
    - chem:weather-analysis         # 权重 0.6
    - mgmt:incident-command         # 权重 0.5
```

### 匹配逻辑

```
场景 Task 声明 requires: [A, B, C]
↓
系统查找所有 Agent 的 has 标签
↓
匹配度 = (Agent 拥有的标签 ∩ Task 需要的标签) 的权重之和 / 总权重
↓
匹配度最高的 Agent 获得任务
```

## 七、多行业包共存设计

### 核心原则

**同一 Nexus 实例可以同时安装多个不同行业的包，无需隔离。**

标签表是共享的，通过 `industry` 字段自然区分。不同行业的标签 ID 前缀不同（`chem:` vs `dev:` vs `medical:`），匹配引擎根据 Task 需要的标签自动路由到对应行业的 Agent。

### 示例：同时安装化工应急 + 软件开发两个行业包

```
DB 标签表（共存，无隔离）：
  chem:hazmat-identification        ← 来自化工应急包
  chem:diffusion-modeling           ← 来自化工应急包
  chem:emergency-response-level     ← 来自化工应急包
  dev:python-code-review           ← 来自软件开发包
  dev:java-security-audit          ← 来自软件开发包
  dev:ci-cd-pipeline             ← 来自软件开发包

DB 行业包表（共存）：
  pack-chemical-emergency-v1       化工应急行业包（standard）
  pack-software-dev-v1             软件开发行业包（standard）

匹配引擎工作方式：
  Task 需要 chem:hazmat-identification → 查所有 Agent → 匹配化工类 Agent
  Task 需要 dev:python-code-review      → 查所有 Agent → 匹配开发类 Agent
  两个行业完全独立，互不干扰
```

### Agent 可以跨行业拥有标签

同一个 Agent 可以同时持有不同行业的标签：

```yaml
# Agent：全栈助手
agent_profile:
  name: "全栈助手"
  has:
    - chem:hazmat-identification   # 来自化工应急包
    - chem:diffusion-modeling       # 来自化工应急包
    - dev:python-code-review        # 来自软件开发包
    - dev:java-security-audit       # 来自软件开发包
```

这样的 Agent 可以同时响应两个行业的任务，由调度引擎根据 Task 的 `requires` 标签自动路由。

### 定制包的共存

客户定制包（Phase 2）与标准包共存：

```
标准包（预置）：
  pack-chemical-emergency-v1（化工应急）
    → 标签：chem:xxx

客户定制包（基于标准包定制）：
  pack-xxpark-chemical-v1（XX化工园区定制）
    → 标签：chem:xxpark:xxx（客户专属扩展）
    → 场景：XX园区特殊预案
    → 设施参数：XX园区具体设施信息
```

**定制包不修改标准包，扩展新标签/场景/知识，共享标准包的通用能力。**

### 未来扩展：包类型字段

当前 `industry_packs` 表暂无 `pack_type` 字段。未来支持后应区分：

| pack_type | 说明 |
|-----------|------|
| `standard` | 标准包（预置，行业通用） |
| `custom` | 定制包（客户专属，基于某标准包扩展） |

此字段在 Phase 2（行业包定制工具）实现时增加。

## 八、标签库的商业价值

### 为什么标签库值钱

| 层面 | 价值 |
|------|------|
| **降低门槛** | 客户不用自己想"我需要哪些 AI 能力"，标签库告诉 ta |
| **标准化** | 一个园区用了这套标签，其他园区想对接就得用同一套 |
| **锁定生态** | 标签越多，迁移成本越高 |
| **持续收入** | 新标签随行业包更新，客户需要持续订阅获取最新标签 |

### 标签库的定价策略

| 内容 | 价格 |
|------|------|
| 开源版标签库（基础标签，10-15 个） | ¥0 |
| 标准行业包标签库（完整标签，30-50 个） | 包含在行业包价格中 |
| 定制标签（客户专属能力定义） | 包含在定制咨询中 |

## 十、与现有能力标签体系的兼容

### 现状

Nexus 已有四维标签体系（`capability_tags`），用于 Agent 匹配。

### 兼容策略

| 原则 | 说明 |
|------|------|
| **不破坏现有** | 现有 `capability_tags` 字段继续使用 |
| **行业标签是扩展** | 行业标签格式 `行业前缀:能力标识符`，兼容现有系统 |
| **双向映射** | 行业标签必须映射到四维之一，四维标签可细化为行业标签 |
| **渐进迁移** | 现有 Agent 的通用标签保留，新增行业标签逐步细化 |

### 映射表示例

| 现有标签 | 化工应急行业标签 | 匹配行为 |
|---------|----------------|---------|
| `professional` | `chem:hazmat-identification` | 专业标签匹配时，行业标签优先 |
| `technical` | `chem:msds-parsing` | 技术标签匹配时，行业标签优先 |
| `business` | `biz:chemical-park-operations` | 业务标签匹配时，行业标签优先 |
| `management` | `chem:incident-command` | 管理标签匹配时，行业标签优先 |
