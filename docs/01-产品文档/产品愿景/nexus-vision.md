## 一、Nexus 是什么

### 一句话定义

**智能体 协同驾驭平台(代号 Nexus)--让每个 智能体 发挥最大价值,协同完成复杂任务,在工作中持续自我进化,全程安全可信。**

### 驾驭平台的边界

| Nexus 不是 | 那是谁的事 |
|---|---|
| 智能体 开发框架 | LangChain、CrewAI |
| 智能体 运行时 | OpenClaw |
| 智能体 底层认知与本体建模 | Nexus Grasp(原 GLG) |
| 又一个聊天机器人 | ChatGPT、Claude |

### Nexus 架构图

Nexus 是一个五层架构的产品家族。

五个产品以东方哲学命名,自身就构成一个完整叙事:

```
悟 · 御 · 化 · 达 · 鉴
Grasp · Reins · Evo · Reach · Vigil
```

> **先悟道,再执缰,在实践中蜕化,最终通达众人--明镜高悬,始终可信。**

**五兄弟各司其职**:

| 产品 | 代号 | 定位 | 核心职责 |
|---|---|---|---|
| **悟** (Grasp) | 悟 | 认知层 | 领域本体建模、基础认知输出、认知共享与更新 |
| **御** (Reins) | 御 | 驾驭层 | 任务拆解、智能体 编排、协同执行、质量闭环 |
| **化** (Evo) | 化 | 进化层 | 经验提取、能力沉淀、跨 智能体 传播、自举闭环 |
| **达** (Reach) | 达 | 场景库 | 场景实例化、数据背书、反馈回收、方案沉淀 |
| **鉴** (Vigil) | 鉴 | 安全层 | 行为审计、权限执行、信任评分、异常检测、数据脱敏 |

**五者之间的协作关系**:

```
悟(Grasp)提供认知 → 御(Reins)驾驭 智能体 协同工作 → 产生执行数据
                                                      ↓
御(Reins)工作中产生新认知 → 回流 悟(Grasp)           化(Evo)提取经验让 智能体 进化
                                                      ↓
                                              成熟场景沉淀到 达(Reach)→ 服务更多用户

                    Vigil 贯穿全程 ↕
          认知安全审查 · 执行行为审计 · 经验投毒检测 · 方案安全扫描
```

**Vigil 贯穿所有层,是整个架构的安全织网**:

```
悟(Grasp)← Vigil 审查:认知投毒检测,防止错误/恶意认知污染本体
御(Reins)← Vigil 监控:行为审计 + 权限执行,实时拦截越权操作
化(Evo)  ← Vigil 扫描:胶囊 安全扫描,危险代码/敏感信息过滤
达(Reach)← Vigil 背书:场景安全评分,只有通过安全审查的场景才能实例化
```

**达(Reach)与化(Evo)的动态共生--场景与能力的关系**:

场景不是静态的场景文档,而是从认知中被抽取、被实例化、被用户根据实际情况修改后执行的活体方案。执行过程中产生反馈数据,最终被回收到场景库,成为该场景的新版本或新分支。

```
     悟(认知层)
         │
         │ 场景从认知中抽取
         ▼
    达(场景库)
         │
         │ 场景实例化
         ▼
    御(驾驭层)
    用户根据实际修改场景 → 执行 → 产生数据
         │                              │
         │ 反馈回收                      │ 执行数据
         ▼                              ▼
  达(场景库)更新 ←── 成功率/耗时/质量计算
         │
         │ 成熟经验提取
         ▼
    化(Evo)能力沉淀
         │
         │ 能力传播
         ▼
    所有智能体受益
```

- 悟(Grasp)提供基础领域认知
- 场景从认知中被抽取,进入场景库(达)
- 用户实例化场景,根据实际情况修改后执行
- 执行数据回流:成功率、耗时、冲突数
- 反馈回收:成功执行的场景更新统计,有用户修改则提示保存为新版本或新分支
- 成熟经验提取为胶囊,通过化(Evo)在智能体之间传播
- **最终形成:场景市场 + 能力共同体**

## 二、理论基础:Harness Engineering 与控制论

### 一个反复出现的模式

George 在《Harness Engineering Is Cybernetics》中指出了一个跨越 240 年的模式[^1]:

| 时代 | 之前 | 之后 | 人类角色转变 |
|------|------|------|------------|
| 1780s 瓦特蒸汽机 | 工人手动调阀门 | 飞球调速器自动调节 | 从转阀门 → 设计调速器 |
| Kubernetes | 运维手动重启服务 | 声明期望状态,控制器自动调谐 | 从重启服务 → 写 spec |
| 现在(智能体) | 工程师写代码 | 设计环境+反馈循环+架构约束,智能体 写代码 | 从写代码 → 驾驭 智能体 |

**同一个模式**:你不再转阀门,你**驾驭**。Wiener 1948 年命名的控制论(Cybernetics),希腊语 κυβερνήτης = 舵手--Kubernetes 的词根也是这个。

### 三个关键洞察

**洞察一:反馈循环必须在正确的层级闭合**

编译器、测试、linter 是低层反馈循环。但"这个架构对不对?这个抽象会不会随着代码库增长出问题?"--以前只有人类能判断。LLM 第一次让反馈循环能在**高层决策层**闭合。

→ 平台的回应:**化**(Evo)不是 retry with feedback,是在决策层闭合反馈循环--能力沉淀和跨 智能体 传播。

**洞察二:知识必须外化,否则 智能体 永远学不会**

> "智能体 不好用"的诊断几乎总是错的。不是 智能体 能力不够,是你脑子里的知识没有外化。智能体 不会通过耳濡目染学习。如果你不写下来,智能体 在第一百次犯的错跟第一次一模一样。

→ 平台的回应:**悟**(Grasp)就是知识外化的系统化方案--领域本体建模,让认知成为可查询的资产。

**洞察三:生成-验证不对称**

生成正确方案比验证方案更难。人类不需要比机器更能写代码,但需要比机器更能**评判**代码:定义"正确"长什么样,识别输出哪里偏了,判断方向对不对。

→ 平台的回应:**御**(Reins)让人类成为驾驭者--看得见、管得住、放得开。鉴(Vigil)是校准传感器--信任评分和权限策略定义了"正确"的边界。

### 代价的不对称性

> 以前不写文档、不写测试的代价是缓慢的质量下降。现在 智能体 会以机器速度、全天候地按照错误的方式工作。不校准的话,制造问题的机器也解决不了问题。

这就是为什么平台必须存在。没有悟(Grasp),智能体 没有认知;没有御(Reins),协同是混乱;没有化(Evo),错误会重复;没有鉴(Vigil),失控以机器速度扩散。

[^1]: George (@odysseus0z), "Harness Engineering Is Cybernetics", 2026. https://x.com/odysseus0z/status/2030416758138634583

## 三、核心信念

### 信念一:智能体 不好用的根因是认知,不是记忆

业界普遍做法是让用户不断跟 智能体 聊天,通过积累对话记忆来"变好用"。

**我们认为这是治标不治本。**

```
❌ 主流做法:
用户不断聊 → 智能体 记住偏好 → 看起来变好了
(本质还是记忆。实习生记住你爱喝美式,但他依然不懂咖啡。)

✅ 我们的做法:
悟(Grasp)提供基础领域认知 → 智能体 在工作中产生新认知 → 新认知回流 悟(Grasp)
→ 所有 智能体 共享更新 → 认知共同体持续生长
```

**认知不是一次性装好的静态配置,而是在使用中持续生长的活体。** 悟(Grasp)提供基础认知让 智能体 起步就好用,智能体 在 御(Reins)中工作时产生的新认知又反哺 悟(Grasp),让整个系统越来越聪明。最终所有 智能体 形成**认知共同体 + 能力共同体**。

### 信念二:驾驭是能力,数据是飞轮

"让 智能体 协同"这件事不新鲜,CrewAI、AutoGen 都能做。平台的价值不止于编排,而是让数据驱动整个系统持续变强。

**数据飞轮让平台越转越快**:

```
智能体 在平台上协同
        ↓
产生运行数据(成功率、耗时、质量分)
        ↓
化(Evo)从数据中提取经验,智能体 进化
        ↓
成熟场景沉淀到 达(Reach)(带数据背书)
        ↓
达(Reach)吸引更多用户实例化
        ↓
更多运行数据
        ↓
飞轮越转越快
```

**编排让 智能体 协同,数据让 智能体 持续变强。**

### 信念三:自举是灵魂

> "只要我把工作给到 智能体,他就能不断地迭代提升--提升产品的质量,同时提升自己的能力。"

这句话定义了平台的终极目标--**双螺旋上升**:

```
任务产出质量 ↑
      ↕ 相互促进
智能体 自身能力 ↑
```

### 信念四:安全是一切的基石

智能体 越强大,越需要约束。没有安全的自举是失控,没有审计的协同是黑箱。

**业界现状**:大多数 智能体 框架把安全当"后期加固"--先跑起来,出了问题再补。

**我们的做法**:安全从第一天就内建于架构,Vigil 是骨架的一部分。

```
❌ 主流做法:
智能体 跑起来 → 出了安全问题 → 打补丁 → 再出问题 → 再打补丁
(永远在追赶,永远晚一步。)

✅ 我们的做法:
每个操作都有审计 → 每次进化都有扫描 → 每条认知都有验证
→ 信任需要积累,智能体 用行为证明自己值得信赖
```

**Vigil 的核心理念**:

- **信任需要积累**:智能体 从低信任起步,通过可靠行为逐步获得更高权限
- **全程可审计**:所有 智能体 行为有完整日志,可追溯、可回放
- **实时防护**:执行时实时拦截越权操作
- **安全与效率并重**:高信任 智能体 获得更大自由度,安全与效率同等重要

## 四、五大核心能力

### 能力一:高效使用

> 把任务给到 智能体,它就能理解并执行。不需要手把手教,不需要反复沟通。

**具体含义**:
- 用户给一个自然语言目标,平台自动拆解为可执行任务
- 智能匹配最合适的 智能体 执行每个子任务
- 任务描述不清时,智能体 主动澄清而非猜测
- 不需要用户懂 智能体 内部机制

**衡量标准**:

- 任务首次理解正确率 ≥ 90%
- 从下达目标到开始执行 ≤ 30 秒
- 用户介入频率:简单任务 0 次,复杂任务 ≤ 2 次

### 能力二:自动化

> 人给一个目标,智能体 自己拆解、分配、执行、验收。

**具体含义**:
- 目标 → 自动拆解为项目和任务
- 任务 → 自动匹配和分配给 智能体
- 执行 → 多 智能体 自动协同,无需人工调度
- 验收 → 自动对照标准验证,不合格自动返工
- 异常 → 自动检测、自动修复、修复不了才升级

**衡量标准**:
- 端到端自动化率 ≥ 80%(不需要人工干预)
- 异常自愈率 ≥ 70%
- 人工介入只在关键决策点

### 能力三:持续工作

> 7×24 持续运行。

**具体含义**:
- 智能体 可以处理长期目标(跨天、跨周的项目)
- 有阻塞时不停下来等人,尝试绕过或升级
- 有异常时自动恢复,不需要人工重启
- 支持定时任务、事件驱动、持续监控

**衡量标准**:
- 系统可用性 ≥ 99.9%
- 单任务阻塞不影响其他任务
- 平均故障恢复时间 ≤ 5 分钟

### 能力四:自举进化

> 每完成一次任务,智能体 能力提升。产品质量提升 + 智能体 自身能力提升。

**具体含义**:
- 任务完成后自动复盘:哪里做得好,哪里可以改进
- 好的策略自动固化为可复用经验
- 经验在 智能体 之间流动,一个学到的所有人受益
- 不只是"记住了",而是"能力真的变强了"

**三个自举闭环**:

```
闭环 1:执行反馈闭环
┌──────────────────────────────────────────────┐
│  执行任务 → 产出结果 → 对比预期 → 发现差距   │
│      ↑                              │        │
│      └──────── 调整策略 ←───────────┘        │
│                                              │
│  大多数产品做到这一步。                       │
│  大多数产品止步于此。真正的进化在后面。        │
└──────────────────────────────────────────────┘

闭环 2:能力沉淀闭环
┌──────────────────────────────────────────────┐
│  好的策略 → 提取为 胶囊 → 固化到能力库     │
│      ↑                              │        │
│      └──── 下次类似任务直接调用 ←────┘        │
│                                              │
│  这是 Evo 的核心。                          │
│  从"这次有效"到"以后都有效"。                  │
└──────────────────────────────────────────────┘

闭环 3:跨 智能体 传播闭环
┌──────────────────────────────────────────────┐
│  智能体 A 学到经验 → 胶囊 共享 → 智能体 B   │
│  也获得这个能力                               │
│                                              │
│  目前没人做好。                                │
│  这是 Nexus 的差异化价值。                    │
└──────────────────────────────────────────────┘
```

**三个闭环都转起来,才是真正的自举。**

**衡量标准**:
- 同类任务第 N 次执行质量 > 第 1 次
- 经验复用率 ≥ 60%(不重复犯同样的错)
- 跨 智能体 经验传播成功率 ≥ 80%

### 能力五:可信安全

> 每个 智能体 的行为都可审计、可追溯、可控制。信任需要积累,用行为证明。

**具体含义**:
- 所有 智能体 操作有完整审计日志,谁、何时、做了什么、结果如何
- 权限基于角色和信任评分动态调整,越可靠权限越大
- 实时异常检测:越权尝试、频率异常、数据泄露模式自动拦截
- 经验共享前必须安全扫描:代码静态分析、敏感信息脱敏、危险操作检测
- 认知投毒防护:恶意/错误认知无法污染本体

**衡量标准**:
- 越权操作拦截率 100%
- 敏感信息泄露率 0%
- 异常行为检测延迟 ≤ 1 秒
- 安全事件可追溯率 100%

## 五、达(Reach):场景库

### 达(Reach)的核心使命

**找到有价值的场景,让它服务更多人。**

### 什么样的场景有价值

```
❌ 没价值:我有一份场景文档,从来没执行过
✅ 有价值:这套场景在 12 次执行中成功率 94.3%,平均 28 分钟完成
```

有价值的场景是**真正被验证过的**。达(Reach)要做的是把这些场景找出来,用运行数据证明它的价值,然后让更多人实例化使用。

### 场景库与模板分享的区别

| 纯配置分享(GitHub) | 平台的场景库(Reach) |
|---|---|
| 这是我的场景文档 | 这套场景在 12 次执行中跑过 |
| 你自己试试看吧 | 平均任务成功率 94.3% |
| 效果好不好我不知道 | 平均耗时 28 分钟,冲突数 1.2 次 |
| 没人维护 | 持续执行、持续优化、持续进化 |

**差异**:场景库里的场景是**活的、被验证过的、带数据的**。

### Reach 的价值飞轮

```
发现有价值的场景
        ↓
让用户实例化使用(运行数据积累)
        ↓
数据证明价值(成功率、效率、质量)
        ↓
更多用户信任并实例化
        ↓
场景在使用中持续优化(反馈回收)
        ↓
价值越来越大
```

### 场景的生命周期

```
从认知中抽取 → 进入场景库 → 用户实例化 → 根据实际修改
    → 执行 → 反馈回收 → 更新为新版本或新分支
```

- **抽取**:从 Grasp 的认知中提取标准化的流程,形成场景
- **实例化**:用户选择场景,根据实际情况修改后执行
- **反馈**:执行完成后,数据自动回写到场景库
- **回收**:成功执行的场景更新统计数据,有修改则生成新版本或新分支

### 冷启动策略

1. **从团队实践中提取**:把 Nexus 团队自身验证过的流程打包进场景库
2. **解决"每次用户都要从零开始"的问题**:新用户直接用成熟场景,不从零开始
3. **自己用自己的平台**:Nexus 团队在 Nexus 上工作,产生的成熟场景第一批入库
4. **数据积累到一定量级后开放第三方提交**

### Reach 与 Evo 的关系

```
Evo   = 微观复制(一个经验、一个技巧在智能体之间流动)
Reach = 宏观复制(整套成功场景对外输出)

Reach 给你一个能用的起点(成熟场景)
        ↓
Evo 让它越用越聪明(持续进化)
```

## 六、信任边界与 Vigil(鉴)

### 核心问题

> 智能体 自己改自己,改到什么程度需要人批准?智能体 之间共享经验,怎么防止"一颗老鼠屎坏了一锅粥"?

这两个问题直接决定了平台的产品设计。鉴(Vigil)就是答案--明镜高悬,让 智能体 可信赖。

### 信任分层模型

```
┌──────────────────────────────────────────────────┐
│  Level 0:完全自主                                │
│  智能体 可以自行决定,无需人类批准                  │
│                                                   │
│  适用:执行策略微调、参数优化、经验记录            │
│  风险:低                                         │
├──────────────────────────────────────────────────┤
│  Level 1:事后报告                                │
│  智能体 先做,做完告诉人类                          │
│                                                   │
│  适用:任务执行方式调整、工具选择变更              │
│  风险:中低                                       │
├──────────────────────────────────────────────────┤
│  Level 2:事前审批                                │
│  智能体 提出方案,人类批准后才执行                  │
│                                                   │
│  适用:角色变更、权限调整、对外交互                │
│  风险:中高                                       │
├──────────────────────────────────────────────────┤
│  Level 3:人类主导                                │
│  人类发起,智能体 辅助执行                          │
│                                                   │
│  适用:关键业务决策、资金操作、安全策略变更        │
│  风险:高                                         │
└──────────────────────────────────────────────────┘
```

### 自举进化的信任边界

| 自举行为 | 信任级别 | 说明 |
|---|---|---|
| 记录执行经验 | Level 0 | 随便记,不影响行为 |
| 优化自己的执行策略 | Level 0 | 在效果可验证的前提下 |
| 把经验封装为 胶囊 | Level 1 | 做完报告,人可以撤回 |
| 向其他 智能体 共享 胶囊 | Level 1 | 做完报告,人可以撤回 |
| 修改自己的角色定义 | Level 2 | 必须人批准 |
| 修改协同编排流程 | Level 2 | 必须人批准 |
| 上架 场景库 | Level 3 | 人类主导 |

### 防止"学歪了"的安全护栏

| 护栏 | 机制 |
|---|---|
| **效果验证** | 新策略必须在沙箱里验证通过才能固化 |
| **回滚能力** | 任何进化都可以一键回退到之前版本 |
| **异常检测** | 如果进化后效果反而变差,自动回滚 |
| **人类否决权** | 人类可以随时冻结任何 智能体 的进化能力 |
| **进化速率限制** | 不能一次改太多,小步迭代 |

### Vigil 的三维信任评分

Vigil 为每个 智能体 维护三维信任评分,决定其权限和自由度:

```
信任评分 = 能力 × 可靠性 × 安全

能力(Capability):任务完成率、质量分、领域覆盖
可靠性(Reliability):稳定性、异常率、超时率
安全(Security):越权次数、脱敏合规、异常行为
```

| 信任等级 | 评分 | 权限 |
|---|---|---|
| **受限** | < 50 | 沙箱执行,所有操作需人工确认 |
| **标准** | 50-79 | 正常权限,高风险操作需确认 |
| **信赖** | 80-95 | 扩展权限,仅关键决策需确认 |
| **自主** | > 95 | 最大自由度,仅 Level 3 操作需人类主导 |

信任评分不是静态的--好行为加分,坏行为扣分,严重违规直接降级。**智能体 的自由度是用可靠行为挣来的。**

### Vigil 的四道防线

```
第一道:权限执行(事前拦截)
  → 智能体 发起操作前,Vigil 检查权限策略
  → 越权操作直接拒绝,记录告警

第二道:行为审计(事中监控)
  → 全量记录 智能体 的文件操作、网络请求、代码执行
  → 实时异常检测:频率异常、模式异常

第三道:安全扫描(内容审查)
  → 胶囊 上报前:代码静态分析 + 敏感信息脱敏
  → 认知上报前:投毒检测 + 来源验证

第四道:事后追溯(审计回放)
  → 完整审计日志,支持任意时间段回放
  → 安全合规报告自动生成
```

## 九、MVP 落地节奏

### 接入模式:Skill 优先

**核心决策**:MVP 阶段优先开发 **nexus Skill**,而非 nexus SDK。

| 模式 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **Skill 模式** | OpenClaw、Claude Code 等现有 Agent | 上手快、好推广、无需改源码 | 轮询延迟、监控粒度有限 |
| **SDK 模式** | 新开发的 Agent | 深度集成、实时推送、完整监控 | 需要改代码、推广成本高 |

**为什么 Skill 优先**:
- ✅ 已有 evolver 技能验证(跨 Agent 进化已实现)
- ✅ OpenClaw 原生支持 Skill 机制
- ✅ Claude Code 等外部 Agent 可快速接入
- ✅ 不要求 Agent 改源码,推广成本低
- ⏳ SDK 模式(长期):适合第三方开发者,等平台成熟后再推广

**nexus Skill 功能模块**:
- `nexus-core`:任务轮询、结果上报、心跳
- `nexus-grasp`:认知查询、认知缓存
- `nexus-evo`:Capsule 提取、Gene 应用、A2A 传播(复用 evolver)
- `nexus-vigil`:本地审计日志、权限拦截、指标上报

### 为什么不是"先可控再提效"

业界常见建议是先上编排+安全(先管住),再补认知+进化(再提效)。听起来合理,但忽略了我们的���心信念:**智能体 好不好用的根因在认知。**

```
❌ "先可控再提效"的问题:
先上 Reins+Vigil → 智能体 有纪律但不懂行 → 用户体验:"这东西不好用"
→ 后面再补认知也晚了,第一印象已经坏了

✅ 我们的节奏:
Grasp+Reins 同步上 → 智能体 既懂行又有组织 → 用户体验:"这东西真好用"
→ Vigil P0 同步跟进 → 安全不缺席
→ Evo 紧随其后 → 形成自举闭环
→ Reach 最后 → 有了数据再谈生态
```

### 四个阶段

| 阶段 | 时间 | 落地内容 | 目标 |
|------|------|---------|------|
| **Phase 1** | MVP | Grasp P0 + Reins P0 + Vigil P0 | 认知+驾驭+安全基础,核心体验跑通 |
| **Phase 2** | MVP+1 | Evo P0 + 各层 P1 | 自举闭环形成,数据飞轮启动 |
| **Phase 3** | V1.0 | Reach P1(场景库上线)+ 各层 P2 | 场景可实例化,带数据背书 |
| **Phase 4** | V2.0 | 全量 P2/P3 | 全功能,规模化 |

### 核心原则

1. **认知和驾驭不能分开**:没有认知的驾驭是"管一群不懂行的新兵",没有驾驭的认知是"一群专家各干各的"
2. **安全从第一天就在**:Vigil 的 P0 功能(行为记录、权限执行、数据脱敏)随 Phase 1 一起上线
3. **进化是闭环的关键**:Phase 2 补上 Evo 后,数据飞轮才开始转
4. **场景库需要数据支撑**:Reach 等飞轮转起来、有真实运行数据后再上线

### 冷启动:种子数据

空系统上线,Grasp 和 Evo 都是空的。智能体 从零摸索会产生大量垃圾数据,飞轮转不起来。

**解决方案**:预置高质量种子数据,让系统"出生即可用"。

| 种子类型 | 示例 | 衰减特性 |
|---------|------|---------|
| **安全红线**(永恒知识) | "不要删除用户数据"、"密钥不能明文存储" | 不衰减 |
| **业务场景 SOP**(中频变) | Docker 部署最佳实践、Python 编码规范 | 半衰期 1-3 年 |
| **工具调用模板**(高频变) | vLLM 配置参数、特定 API 调用方式 | 半衰期 3-6 个月 |

**来源**:
1. 团队专家手动编写第一批种子认知和种子 胶囊
2. 从 Nexus 团队自身实践中提炼(自己用自己的平台)
3. 种子数据必须经过人工审核,初始信任评分 90+

### 单场景验证

MVP 阶段选择一个具体场景(如"部署 vLLM 到 DGX Spark")完整验证五层 P0 功能:
- Grasp 注入 GPU/vLLM 领域认知 → 减少试错
- Reins 分解任务、分配 智能体、协同执行 → 自动化完成
- Vigil 审计全程、拦截越权 → 安全可控
- Evo 提取部署经验为 胶囊 → 下次复用
- 定义成功指标:认知注入后试错率降低 X%、胶囊 复用后耗时减少 Y%

## 十、开放问题(待后续讨论)

1. **智能体 Protocol 标准**:已确认--拥抱现成标准(MCP 用于 智能体↔工具/数据源,A2A 用于 智能体↔智能体),不造轮子,确保广泛适应性。Nexus 是协议之上的编排层。
2. **Evo 经验泛化**:怎么判断一个经验是"普遍有效"还是"特定场景有效"?思路:不提前判断,在使用中验证,用置信度谱代替非黑即白的标签。
3. **场景库价值衡量**:怎么定义和证明一个场景"有价值"?运行数据(成功率、效率、质量)是核心证据,但具体指标体系待定义。
4. **Grasp 集成深度**:Reins 和 Grasp 双向数据流的接口规范待定义(认知输出 + 认知回流)。
5. **信任边界个性化**:不同企业对信任级别的要求不同,通过三维矩阵(操作类型×agent角色×场景上下文)+ 策略模板实现灵活配置,具体模板待设计。
