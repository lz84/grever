## 一、概述

### 1.1 文档目的

本文档是智能体注册与心跳系统的详细设计文档，涵盖 Agent 注册流程、心跳机制、状态机、负载计算、能力更新以及孤儿 Agent 清理。是架构设计文档 [20-agent-heartbeat-system-design.md](../../02-架构设计/20-agent-heartbeat-system-design.md) 和 [06-agent-sdk-architecture.md](../../02-架构设计/06-agent-sdk-architecture.md) 的实现层补充。

### 1.2 核心概念

| 概念 | 一句话解释 |
|------|-----------|
| **Agent 注册** | Agent 连接 Nexus 平台并上报自身能力、模型、连接参数的过程 |
| **心跳** | Agent 与 Nexus 之间的双向通信通道，用于报活、领任务、更新状态 |
| **能力标签** | 四维标签（business/professional/technical/management）描述 Agent 能力 |
| **负载** | Agent 当前任务饱和度，0-100 数值，影响任务分配决策 |
| **孤儿 Agent** | 已注册但长时间无心跳、与平台断开连接的 Agent |

## 二、Agent 注册流程

### 2.1 注册概述

Agent 注册是 Agent 首次连接 Nexus 平台时的初始化过程。注册完成后，Agent 进入 idle 状态，开始参与任务调度。

```
注册流程：
SDK 初始化 → 连接 Nexus → 上报注册信息 → 平台绑定能力标签 → Agent 上线（idle）
```

### 2.2 注册前置条件

Agent 在注册前需要满足以下条件：

| 条件 | 说明 |
|------|------|
| **Nexus 平台可达** | Agent 能访问 Nexus 的 WebSocket 或 HTTP 端点 |
| **Agent SDK 已初始化** | SDK 完成本地存储、配置文件加载 |
| **模型服务可用** | Agent 使用的模型 API（vLLM/OpenAI）可连通 |
| **唯一标识** | Agent 有全局唯一的 agent_id（UUID 或平台分配） |

### 2.4 注册 API 详细设计

```python
# POST /api/v1/agents/register
class AgentRegisterRequest(BaseModel):
    agent_id: str              # 全局唯一标识
    name: str                  # Agent 名称（中文）
    model: ModelInfo           # 模型信息
    capability_tags: CapabilityTags  # 四维能力标签
    config: AgentConfig        # 配置参数

class ModelInfo(BaseModel):
    type: Literal["vllm", "openai", "anthropic"]
    endpoint: str              # 模型 API 地址
    model_name: str            # 模型名称

class CapabilityTags(BaseModel):
    business: list[str] = []      # 业务能力
    professional: list[str] = []   # 专业能力
    technical: list[str] = []     # 技术能力
    management: list[str] = []    # 管理能力

class AgentConfig(BaseModel):
    max_concurrent_tasks: int = 3   # 最大并发任务数
    heartbeat_interval: int = 60     # 心跳间隔（秒）
    labels: dict[str, float] = {}   # 初始标签权重（可选）

async def register_agent(
    request: AgentRegisterRequest,
    db: Session
) -> AgentRegisterResponse:
    # 1. 检查 agent_id 唯一性
    existing = await db.query(Agent).filter(
        Agent.id == request.agent_id
    ).first()

    if existing:
        # 如果是孤儿（旧记录），覆盖；否则拒绝
        if existing.status == "offline" and \
           (now() - existing.last_heartbeat).seconds > ORPHAN_THRESHOLD:
            # 孤儿，覆盖旧记录
            await _overwrite_agent(existing, request, db)
            return AgentRegisterResponse(ok=True, status="idle")
        else:
            raise HTTPException(409, "Agent already registered")

    # 2. 创建新 Agent 记录
    agent = Agent(
        id=request.agent_id,
        name=request.name,
        model_type=request.model.type,
        model_endpoint=request.model.endpoint,
        model_name=request.model.model_name,
        capability_tags=request.capability_tags.model_dump(),
        status="idle",
        load=0,
        max_concurrent_tasks=request.config.max_concurrent_tasks,
        heartbeat_interval=request.config.heartbeat_interval,
        labels={k: 0.3 for k in _flatten_tags(request.capability_tags)},
        last_heartbeat=now()
    )
    db.add(agent)
    await db.commit()

    return AgentRegisterResponse(
        ok=True,
        agent_id=agent.id,
        status="idle",
        registered_at=agent.created_at
    )
```

### 2.5 重复注册处理

| 场景 | 处理策略 |
|------|---------|
| 相同 agent_id 重新注册，状态为 idle/busy | 返回 409 冲突，要求先注销 |
| 相同 agent_id 重新注册，状态为 offline 且超过孤儿阈值 | 视为孤儿，覆盖旧记录 |
| agent_id 已存在但平台认为在线，Agent 认为离线 | Agent 应先调用 `/agents/{id}/unregister` 再注册 |

## 三、心跳机制

### 3.1 心跳概述

心跳是 Agent 与 Nexus 平台之间的双向通信通道，每隔固定时间（默认 30-300 秒）由 Agent 主动发起。

```
一次心跳完成四件事：
1. 报活：Agent 上报当前状态、负载、正在执行的任务
2. 领活：Nexus 检查是否有分配给该 Agent 的任务，返回待执行任务列表
3. 更新：Nexus 更新 Agent 的 last_heartbeat、status、load
4. 通知：Nexus 返回配置变更、裁决结果、停止指令等
```

### 3.2 心跳流程

```
Agent 端定时器触发（每 heartbeat_interval 秒）
     │
     ▼
┌─────────────────────────────────────────────────────────────┐
│ Agent 构造心跳请求                                          │
│                                                              │
│ GET /api/v1/agents/{agent_id}/heartbeat                    │
│ 或 WebSocket 发送心跳消息                                    │
│                                                              │
│ 请求内容：                                                  │
│ {                                                           │
│   "current_status": "idle",                                │
│   "load": 45,                                              │
│   "running_task_ids": ["task-001", "task-002"],           │
│   "completed_task_ids_since_last_heartbeat": ["task-000"], │
│   "model_health": "ok",                                    │
│   "timestamp": "2026-05-27T10:05:00Z"                     │
│ }                                                           │

┌─────────────────────────────────────────────────────────────┐
│ Nexus 处理心跳请求                                          │
│                                                              │
│ 1. 记录心跳时间（last_heartbeat = now）                    │
│ 2. 更新 Agent 状态（如果与上次不同）                        │
│ 3. 更新 Agent 负载（基于 running_task_ids）                  │
│ 4. 验证模型连通性（可选，轻量级检测）                       │
│ 5. 分配待执行任务（根据能力匹配 + 负载）                    │

│ 7. 检查裁决结果（需要通知 Agent 的裁决）                   │
│ 8. 记录心跳日志                                            │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ Nexus 返回心跳响应                                          │
│                                                              │
│ {                                                           │
│   "ok": true,                                               │
│   "pending_tasks": [                                        │
│     { "task_id": "task-003", "priority": "high", ... }     │
│   ],                                                        │
│   "pending_stops": ["task-001"],  ← 需要停止的任务         │
│   "rulings": [                                              │
│     { "task_id": "task-xxx", "type": "disputed",           │
│       "ruling": "pass", "comment": "通过" }                │
│   ],                                                        │
│   "config_updates": {},                                     │
│   "server_time": "2026-05-27T10:05:01Z"                   │
│ }                                                           │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
Agent 收到响应后：
- 领取 pending_tasks 中的新任务
- 停止 pending_stops 中的任务
- 处理 rulings 中的裁决结果
- 重置心跳定时器
```

### 3.3 心跳请求与响应数据结构

```python
# 心跳请求（Agent → Nexus）
class HeartbeatRequest(BaseModel):
    current_status: AgentStatus          # 当前状态
    load: int                            # 当前负载 0-100
    running_task_ids: list[str] = []     # 正在执行的任务 ID 列表
    completed_task_ids_since_last_heartbeat: list[str] = []  # 上次心跳后完成的任务
    model_health: Literal["ok", "degraded", "unavailable"] = "ok"
    timestamp: datetime

# 心跳响应（Nexus → Agent）
class HeartbeatResponse(BaseModel):
    ok: bool
    pending_tasks: list[PendingTask] = []   # 待执行任务列表
    pending_stops: list[str] = []           # 需要停止的任务 ID 列表
    rulings: list[RulingNotification] = []   # 需要通知的裁决结果
    config_updates: dict = {}               # 配置变更
    server_time: datetime

class PendingTask(BaseModel):
    task_id: str
    title: str
    description: str
    priority: str
    capability_tags: CapabilityTags
    context: dict  # Task 上下文

class RulingNotification(BaseModel):
    task_id: str
    type: Literal["disputed", "waiting_human", "human_input"]
    ruling: Literal["pass", "reject", "request_changes"]
    comment: str | None
```

### 3.4 心跳间隔配置

| 参数 | 默认值 | 可配置范围 | 说明 |
|------|--------|-----------|------|
| `heartbeat_interval` | 60 秒 | 30-300 秒 | Agent 发送心跳的间隔 |
| `stale_threshold` | 300 秒 | — | 5 分钟无心跳标记 stale |
| `offline_threshold` | 900 秒 | — | 15 分钟无心跳标记 offline |
| `max_consecutive_failures` | 3 | — | 超过此次数心跳失败标记 offline |

不同场景的推荐心跳间隔：

| 场景 | 推荐间隔 | 理由 |
|------|---------|------|
| 高频任务执行 | 30 秒 | 快速响应任务完成/失败 |
| 常规任务执行 | 60 秒 | 平衡响应速度与负载 |
| 低频/待机状态 | 300 秒 | 减少不必要的网络请求 |

### 3.5 心跳异常处理

| 异常类型 | 处理策略 | Agent 端行为 |
|---------|---------|------------|
| HTTP/WebSocket 连接失败 | 重试 3 次，间隔 5/10/30 秒 | 进入 degraded 状态 |
| 心跳响应超时（30 秒无响应） | 重试 3 次 | 降低任务领取优先级 |
| 心跳返回 error | 记录错误，等待下次心跳 | 按响应中的指示行动 |
| 模型服务不可用 | 标记 model_health="unavailable" | Nexus 暂停派发新任务 |

## 四、Agent 状态机

### 4.1 状态定义

Agent 有五种状态：

| 状态 | 值 | 说明 | 可接收任务 |
|------|-----|------|----------|
| **离线** | `offline` | Agent 未连接或心跳超时 | ❌ 不派发 |
| **空闲** | `idle` | Agent 已连接，无正在执行的任务 | ✅ 正常派发 |
| **忙碌** | `busy` | Agent 正在执行任务（负载 > 0） | ✅ 可派发（受负载限制） |
| **暂停** | `paused` | Agent 被管理员或自身暂停 | ❌ 不派发 |
| **错误** | `error` | Agent 发生错误（如模型故障） | ❌ 不派发 |

### 4.2 状态转换图

### 4.3 状态转换规则表

| 当前状态 | 触发事件 | 目标状态 | 动作 |
|---------|---------|---------|------|
| — | `register_agent()` | `idle` | 初始化 Agent 记录 |
| `idle` | 有任务开始执行 | `busy` | 更新 load > 0 |
| `idle` | `pause()` | `paused` | 停止接收新任务 |
| `idle` | 心跳超时 | `offline` | 停止派发，标记孤儿 |
| `idle` | 模型故障 | `error` | 停止派发 |
| `busy` | 所有任务完成 | `idle` | load = 0 |
| `busy` | `pause()` | `paused` | 发送停止信号给所有任务 |
| `busy` | 心跳超时 | `offline` | 挂起所有任务，等待重新分配 |
| `busy` | 任务连续失败 > 阈值 | `error` | 暂停派发，通知管理员 |
| `paused` | `resume()` | `idle` 或 `busy` | 恢复接收任务 |
| `error` | 模型恢复 / 人工恢复 | `idle` | 恢复正常状态 |
| `offline` | 重新心跳 | `idle` | 恢复派发 |

## 五、负载计算

### 5.1 负载概述

负载（Load）是 Agent 当前任务饱和度的量化指标，范围 0-100。负载由正在执行的任务数和 Agent 的最大并发任务数决定。

```
负载 = (正在执行的任务数 / 最大并发任务数) × 100
```

### 5.2 calc_dynamic_load 公式

```python
def calc_dynamic_load(
    running_task_count: int,
    max_concurrent_tasks: int,
    task_weights: dict[str, float] | None = None
) -> int:
    """
    计算 Agent 的动态负载

    参数：
        running_task_count: 当前正在执行的任务数
        max_concurrent_tasks: Agent 配置的最大并发任务数
        task_weights: 各任务的权重（可选，默认全为 1.0）

    返回：
        负载值，范围 0-100
    """
    if max_concurrent_tasks <= 0:
        return 100  # 除零保护

    if task_weights:
        # 加权模式：考虑任务的复杂度
        weighted_sum = sum(task_weights.get(tid, 1.0)
                         for tid in range(running_task_count))
        base_load = (weighted_sum / max_concurrent_tasks) * 100
    else:
        # 简单模式：直接按任务数计算
        base_load = (running_task_count / max_concurrent_tasks) * 100

    # 上限 100
    return min(100, int(base_load))
```

### 5.3 任务权重配置

不同优先级的任务可以配置不同的权重：

```python
# 任务权重配置
TASK_WEIGHTS = {
    "critical": 2.0,   # 紧急任务权重翻倍
    "high": 1.5,       # 高优先级任务
    "medium": 1.0,     # 默认权重
    "low": 0.5,        # 低优先级任务
}

def get_task_weight(task: dict) -> float:
    priority = task.get("priority", "medium")
    return TASK_WEIGHTS.get(priority, 1.0)
```

### 5.4 负载与调度决策

| 负载范围 | Agent 状态 | 调度行为 |
|---------|-----------|---------|
| 0 | `idle` | 优先分配任务 |
| 1-49 | `busy` | 可分配任务，负载越低优先级越高 |
| 50-79 | `busy` | 可分配任务，但需考虑其他 Agent |
| 80-99 | `busy` | 谨慎分配，只派发高优先级任务 |
| 100 | `busy`（满载） | 暂停派发新任务 |

### 5.5 负载更新时机

负载在以下时机更新：

| 时机 | 更新方式 |
|------|---------|
| 心跳请求到达 | 根据 `running_task_ids` 重新计算 |
| 任务开始执行 | load += weight(task) |
| 任务完成 | load -= weight(task) |
| 任务失败/超时 | load -= weight(task) |
| Agent 暂停 | load 保持不变，等待任务停止 |

## 六、Agent 能力更新

### 6.1 能力标签更新机制

Agent 的能力标签在注册时初始化，在执行任务过程中根据行为信号动态更新。

```
标签更新触发时机：
1. 任务完成 → 标签权重增加（+0.1）
2. 任务失败 → 标签权重减少（-0.1）
3. 任务超时 → 标签权重减少（-0.05）
4. 长期未使用（30/60/90 天）→ 标签权重衰减
```

### 6.2 update_capabilities API

```python
# PUT /api/v1/agents/{agent_id}/capabilities
class UpdateCapabilitiesRequest(BaseModel):
    """手动更新 Agent 能力标签（管理员用）"""
    add_tags: CapabilityTags | None = None      # 新增标签
    remove_tags: CapabilityTags | None = None    # 移除标签
    set_weights: dict[str, float] | None = None  # 直接设置权重

class UpdateCapabilitiesResponse(BaseModel):
    ok: bool
    capability_tags: CapabilityTags
    labels: dict[str, float]  # 标签 → 权重映射

async def update_capabilities(
    agent_id: str,
    request: UpdateCapabilitiesRequest,
    db: Session
) -> UpdateCapabilitiesResponse:
    agent = await db.query(Agent).filter(Agent.id == agent_id).first()
    if not agent:
        raise HTTPException(404, "Agent not found")

    current_tags = agent.capability_tags
    current_labels = agent.labels

    # 1. 添加新标签（权重默认 0.3）
    if request.add_tags:
        for tag in _flatten_tags(request.add_tags):
            if tag not in current_labels:
                current_labels[tag] = 0.3

    # 2. 移除标签
    if request.remove_tags:
        for tag in _flatten_tags(request.remove_tags):
            current_labels.pop(tag, None)
            # 从各维度列表中移除
            for dim, tags in current_tags.items():
                if tag in tags:
                    tags.remove(tag)

    # 3. 直接设置权重
    if request.set_weights:
        for tag, weight in request.set_weights.items():
            current_labels[tag] = max(0.0, min(1.0, weight))

    agent.labels = current_labels
    agent.capability_tags = current_tags
    await db.commit()

    return UpdateCapabilitiesResponse(
        ok=True,
        capability_tags=current_tags,
        labels=current_labels
    )
```

### 6.3 心跳驱动的标签自动更新

每次心跳时，如果 Agent 上报了 `completed_task_ids_since_last_heartbeat`，Nexus 会自动更新标签：

```python
async def _update_labels_from_completion(
    agent_id: str,
    completed_task_ids: list[str],
    db: Session
):
    """
    根据任务完成情况自动更新 Agent 标签权重
    """
    for task_id in completed_task_ids:
        task = await db.query(Task).filter(Task.id == task_id).first()
        if not task:
            continue

        # 从 task 的 capability_tags 推断标签
        inferred_tags = task.capability_tags or {}

        # 增加权重
        for tag in _flatten_tags(inferred_tags):
            if tag in agent.labels:
                agent.labels[tag] = min(1.0, agent.labels[tag] + 0.1)
            else:
                # 新标签，权重较低
                agent.labels[tag] = 0.3

    await db.commit()
```

### 6.4 标签衰减机制

定时任务（每天执行）检查标签权重，执行衰减：

```python
# 每天执行一次的衰减任务
async def decay_capability_labels(db: Session):
    """
    标签权重衰减：
    - technical: 30 天未使用 → 权重 × 0.8
    - professional: 60 天未使用 → 权重 × 0.5
    - business: 90 天未使用 → 权重归零
    - management: 持续有效（不衰减）
    """
    DECAY_RULES = {
        "technical": {"days": 30, "factor": 0.8},
        "professional": {"days": 60, "factor": 0.5},
        "business": {"days": 90, "factor": 0.0},
        "management": {"days": -1, "factor": 1.0},  # 不衰减
    }

    agents = await db.query(Agent).all()
    for agent in agents:
        for tag, weight in list(agent.labels.items()):
            dimension = _get_tag_dimension(tag)
            rule = DECAY_RULES.get(dimension)
            if not rule or rule["days"] < 0:
                continue

            # 检查最后使用时间
            last_used = agent.labels_last_used.get(tag)
            if last_used and (now() - last_used).days > rule["days"]:
                new_weight = weight * rule["factor"]
                if new_weight < 0.05:
                    # 太低的标签移除
                    del agent.labels[tag]
                else:
                    agent.labels[tag] = new_weight

    await db.commit()
```

## 七、孤儿 Agent 清理

### 7.1 孤儿 Agent 定义

孤儿 Agent 是已注册但长时间与平台断开连接的 Agent：

| 条件 | 值 |
|------|-----|
| 状态为 `offline` | 连续心跳超时超过 `offline_threshold`（默认 900 秒 = 15 分钟）|
| 孤儿判定时间 | `last_heartbeat` + `offline_threshold` < 当前时间 |

### 7.2 孤儿清理策略

| 策略 | 说明 |
|------|------|
| **自动下线** | Agent 超过 `offline_threshold` 无心跳，自动标记 `offline` |
| **任务挂起** | Agent 离线时，其正在执行的任务标记为 `pending_redispatch` |
| **记录保留** | 孤儿 Agent 的记录保留 30 天（用于审计和问题排查） |
| **注册覆盖** | 孤儿重新上线时，旧记录被覆盖 |

### 7.3 孤儿 Agent 处理流程

```
健康检查定时任务触发（每 60 秒）
     │
     ▼
┌─────────────────────────────────────────────────────────────┐
│ 遍历所有非 offline 状态的 Agent                            │
│                                                              │
│ for agent in agents where status != "offline":              │
│     elapsed = now() - agent.last_heartbeat                  │
│                                                              │
│     if elapsed > offline_threshold:                         │
│         → agent.status = "offline"                          │
│         → 挂起该 Agent 的所有 running_tasks                │
│         → 记录孤儿日志                                       │
│                                                              │
│     elif elapsed > stale_threshold:                         │
│         → agent.status = "stale"  （亚健康，不派发新任务）   │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 任务重新分配                                                │
│                                                              │
│ 对于挂起的任务：                                             │
│ - 从任务队列中取出                                           │
│ - 重新匹配其他可用 Agent                                     │
│ - 派发到新 Agent                                            │
│                                                              │
│ 旧 Agent 重新上线后：                                        │
│ - 收到心跳后状态恢复为 idle                                 │
│ - 不自动接管之前的任务（已重新分配）                         │
└─────────────────────────────────────────────────────────────┘
```

### 7.4 孤儿 Agent 数据库清理

### 7.5 孤儿 Agent 上线恢复

当孤儿 Agent 重新连接时，Nexus 识别为孤儿并允许覆盖注册：

```python
# 心跳请求中发现 Agent 已 offline 超过阈值
async def heartbeat(
    agent_id: str,
    request: HeartbeatRequest,
    db: Session
):
    agent = await db.query(Agent).filter(Agent.id == agent_id).first()

    if agent and agent.status == "offline":
        elapsed = (now() - agent.last_heartbeat).seconds
        if elapsed > ORPHAN_THRESHOLD:
            # 孤儿重新上线，视为新注册（覆盖旧记录）
            agent.status = "idle"
            agent.last_heartbeat = now()
            agent.labels_last_used = {}  # 重置标签使用时间
            # 注意：旧的任务记录已被重新分配，不需要处理
```

## 附：心跳系统设计（原 20-agent-heartbeat-system-design.md）

> 以下内容从 `docs/02-架构设计/20-agent-heartbeat-system-design.md` 迁移并补充，涵盖心跳的概念、生命周期、与驾驭/标签/裁决中心的配合，以及异常处理体系。

### 附一、心跳的本质

心跳是 Nexus 系统中 **Agent 与平台之间的双向沟通机制**，不是简单的"报活"，而是 Agent 生命周期中最重要的 **控制通道**：

一次心跳同时完成三件事：
1. **报活**：Agent 告诉平台"我还活着，当前状态是什么"
2. **领活**：平台告诉 Agent "你有 X 个待执行任务"
3. **更新**：平台将 Agent 的负载、能力、连接状态同步到全局状态

心跳与健康检查的区别：

| 维度 | 心跳 | 健康检查 |
|------|------|----------|
| 触发方 | Agent 主动发起 | 平台主动扫描 |
| 频率 | 高频（30-300秒） | 低频（5-15分钟） |
| 内容 | 状态 + 领任务 | 连通性判断 |
| 目的 | 实时通信 | 离线检测 |

### 附二、心跳的完整生命周期

**单次心跳的业务流程**：

```
Agent 上报状态/负载/任务数/模型信息
        │
        ▼
Nexus：1. 记录心跳时间  2. 验证模型连通性  3. 更新 Agent 状态
       4. 分配待执行任务  5. 记录执行日志  6. 更新能力标签
        │
        ▼
Agent 拿到：新任务 + 配置变更 + 裁决结果
```

**心跳的七个处理阶段**：

| 阶段 | 名称 | 做什么 | 产出 |
|------|------|--------|------|
| 1 | **连通性验证** | 检查 Agent 的模型 API 是否可用 | 验证通过/失败 |
| 2 | **状态更新** | 更新 last_heartbeat、status、load | Agent 状态变更 |
| 3 | **负载计算** | 基于当前任务和并发限制计算新负载 | 负载评分 (0-100) |
| 4 | **任务分配** | 将 pending 任务分配给该 Agent | 任务 ID 列表 |
| 5 | **日志记录** | 记录本次心跳的完整执行链路 | 结构化日志条目 |
| 6 | **标签推断** | 根据本次行为更新能力标签权重 | 标签权重变更 |
| 7 | **响应返回** | 将待执行任务和配置变更返回给 Agent | 心跳响应 |

### 附三、心跳与驾驭中心的配合

心跳是驾驭中心的 **实时数据源**。驾驭中心只在心跳时做调度决策（推送式架构）：

**调度时机判断**：
1. 这个 Agent 在线吗？（连通性验证）
2. 这个 Agent 还能接活吗？（负载检查 < 80）
3. 有适合他的任务吗？（能力匹配）
4. 有优先级更高的 Agent 吗？（全局最优）

驾驭中心的心跳策略：

| 场景 | 策略 | 说明 |
|------|------|------|
| **正常派发** | 能力匹配 + 负载优先 | 选择最合适的 Agent |
| **紧急任务** | 跳过负载检查 | 即使 Agent 负载高也派发 |
| **Agent 离线** | 重新匹配 | 将任务重新分配给其他 Agent |
| **Agent 恢复** | 优先补偿 | 离线恢复后优先派发任务 |
| **连续失败** | 降级使用 | 连续失败超过阈值，降低匹配优先级 |

### 附四、心跳与标签系统的配合

心跳是标签更新的核心触发器。每次心跳时，如果 Agent 上报了 `completed_task_ids_since_last_heartbeat`，Nexus 自动更新标签权重：

**标签推断流程**：
- 任务类型 → business 维度
- 使用工具 → technical 维度
- 产出物类型 → professional 维度
- 系统调用行为 → management 维度

**权重更新规则**：
- 成功执行：+0.1（上限 1.0）
- 超时：-0.05
- 失败：-0.1

**衰减规则**：
- 30 天未使用（technical）：权重 × 0.8
- 60 天未使用（professional）：权重 × 0.5
- 90 天未使用（business）：权重归零
- management 维度：持续有效，不衰减

### 附五、心跳与裁决中心的配合

当任务验证连续失败 3 次时，状态变为 `disputed`，进入裁决流程：

```
任务验证失败 → 重试 3 次 → 全部失败 → 裁决中心
                                         │
                                   用户裁决
                                         │
                              ┌──────────┴──────────┐
                              ▼                     ▼
                          通过 ✅              驳回 ❌
```

心跳响应中包含裁决相关指令（rulings），Agent 据此处理任务状态。

### 附六、心跳的异常处理体系

**心跳异常分类**：
- 网络层：连接超时、DNS 解析失败、连接被拒绝
- 应用层：模型 API 不可用、Agent 进程崩溃、数据格式错误
- 业务层：Agent 负载超限、被禁用、无可用任务

**健康检查被动兜底**：
- T+5 分钟（stale）：标记为"亚健康"，不派发新任务
- T+15 分钟（offline）：标记为"离线"，已有任务挂起
- T+N 恢复：Agent 重新心跳后恢复为 idle

**连续失败计数器**：counter = 0 → 1 → 2 → 3（标记 offline）

### 附七、心跳的配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 心跳间隔 | 30-300 秒 | Agent 多久发一次心跳 |
| 在线阈值 | 60 秒 | 多少秒内有心跳算在线 |
| 忙碌阈值 | 120 秒 | 多少秒内有心跳算忙碌 |
| 离线阈值 | 300 秒 | 多少秒无心跳算离线 |
| 健康检查 stale 阈值 | 5 分钟 | 被动检测亚健康阈值 |
| 健康检查 offline 阈值 | 15 分钟 | 被动检测离线阈值 |
| 最大连续失败次数 | 3 次 | 超过此次数标记离线 |
