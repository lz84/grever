## 二、能力标签四维分类

### 2.1 四维定义

所有能力标签统一使用四维分类，存储在 JSON object 中：

```json
{
  "business": ["emergency_response", "rescue_coordination"],
  "professional": ["incident_command", "risk_assessment"],
  "technical": ["python", "fastapi", "postgresql"],
  "management": ["coordination", "task_dispatch", "grasp_read", "grasp_write"]
}
```

### 2.2 各维度详细说明

| 维度 | 英文名 | 含义 | 回答的问题 | 来源 |
|------|--------|------|-----------|------|
| **业务能力** | business | Agent 在特定业务领域的经验 | Agent 懂不懂这个行业？ | 场景分配历史、Goal/Project 业务类型 |
| **专业能力** | professional | Agent 在专业领域的技能 | Agent 能做什么专业工作？ | 产出物分析、解决方案评分 |
| **技术能力** | technical | Agent 使用的技术栈 | Agent 用什么工具/语言/框架？ | 工作区文件特征、任务执行结果 |
| **管理能力** | management | Agent 的系统级能力 | Agent 会不会用 Nexus 系统？ | 系统调用行为，自动推断 |

### 2.3 维度示例

**化工应急行业**：
```json
{
  "business": ["emergency_response", "chemical_hazard", "fire_fighting"],
  "professional": ["incident_command", "risk_assessment", "hazmat_identification"],
  "technical": ["python", "diffusion_modeling", "gis_mapping"],
  "management": ["coordination", "task_dispatch", "grasp_read", "grasp_write"]
}
```

**软件开发行业**：
```json
{
  "business": ["software_development", "web_application"],
  "professional": ["api_design", "database_optimization"],
  "technical": ["python", "fastapi", "postgresql", "docker", "typescript", "react"],
  "management": ["coordination", "task_dispatch", "grasp_read", "grasp_write"]
}
```

## 三、标签生命周期

### 3.1 状态机

```
active ──→ deprecated ──→ replaced_by（最终）
```

| 状态 | 说明 | 新场景是否可引用 |
|------|------|---------------|
| `active` | 正常可用 | ✅ 可引用 |
| `deprecated` | 已废弃，不推荐使用 | ⚠️ 已有引用可保留，新引用禁止 |
| `replaced_by` | 被替代，指向新标签 ID | ❌ 禁止引用 |

### 3.2 标签数据结构

| 字段 | 类型 | 说明 | 必填 |
|------|------|------|------|
| tag_id | TEXT (PK) | 标签唯一标识，格式：`行业前缀:能力标识符` | ✅ |
| tag_name | TEXT | 标签中文名称 | ✅ |
| dimension | TEXT | 所属维度：business/professional/technical/management | ✅ |
| level | TEXT | 级别：basic/intermediate/advanced/expert | ✅ |
| description | TEXT | 标签描述 | 否 |
| industry | TEXT | 所属行业标识 | ✅ |
| status | TEXT | 状态：active/deprecated/replaced_by | ✅ |
| replaced_by | TEXT | 替代标签 ID（当 status=replaced_by 时） | 否 |
| version_major | INTEGER | 主版本号（MAJOR 变更：破坏性变更） | 否 |
| version_minor | INTEGER | 次版本号（MINOR 变更：新增内容） | 否 |
| version_patch | INTEGER | 补丁版本号（PATCH 变更：修正） | 否 |
| tools | TEXT | 关联工具列表（JSON array） | 否 |
| examples | TEXT | 使用示例列表（JSON array） | 否 |
| prerequisites | TEXT | 前置标签 ID 列表（JSON array） | 否 |
| created_at | INTEGER | 创建时间 | ✅ |
| updated_at | INTEGER | 更新时间 | 否 |

### 3.3 版本变更规则

| 变更类型 | 影响字段 | 版本递增 |
|---------|---------|---------|
| MAJOR | 标签 ID、维度变更 | version_major++ |
| MINOR | 描述、名称、工具、示例变更 | version_minor++ |
| PATCH | tools / examples 更新 | version_patch++ |

## 附：任务分配与匹配引擎架构（原 13-task-assignment-and-matching.md）

> 以下内容从 `docs/02-架构设计/13-task-assignment-and-matching.md` 迁移，涵盖三条铁律、匹配引擎设计和能力标签在匹配中的应用。

### 附一、三条铁律

**铁律一：统一匹配引擎，调用时机是结构分解后**

| ❌ 错误做法 | ✅ 正确做法 |
|---|---|
| 每个 task 创建时自动分配 | Goal/Project 完成结构分解后，**统一触发** `assign_pending_tasks()` |
| 心跳时各自拉任务 | 一次性匹配所有下属任务，不是逐条分配 |
| 在 CRUD 文件里写分配逻辑 | 匹配引擎是**唯一入口**，禁止分散 |

**调用时机**：
```
Goal 创建
  → decompose（结构分解，生成 Projects + Tasks）
    → 统一触发 TaskAssigner.assign_pending_tasks()
      → 对每个 todo 任务：读取 capability_tags → 匹配引擎 → 分配 agent
```

**铁律二：创建 Task/Project 必须设置 depends_on + capability_tags**

创建 Task 时必填字段：

| 字段 | 用途 | 示例 |
|---|---|---|
| `depends_on` | DAG 执行顺序，决定哪个任务先跑 | `["task-abc123"]` |
| `capability_tags` | 匹配引擎的输入，四维标签 | `{"technical": ["python", "fastapi"]}` |
| `assigned_agent` | **不填**，由匹配引擎决定 | `null` |

**depends_on 设置规则**：
- 同一个 Project 内，有前后依赖关系的任务必须设置
- 无依赖的任务设为空数组 `[]`
- 跨 Project 依赖由 Project 级别的依赖关系控制

**铁律三：匹配逻辑绝对集中**

所有匹配逻辑必须在 `TaskAssigner.assign_pending_tasks()` 中完成，禁止在任何其他地方写分配代码。

### 附二、能力标签在匹配引擎中的应用

匹配引擎使用四维标签进行 Agent-Task 匹配：

| 维度 | 含义 | 匹配优先级 |
|------|------|-----------|
| `business` | 业务领域 | 高（决定专业方向） |
| `professional` | 专业技能 | 高（决定能否完成任务） |
| `technical` | 技术栈 | 中（工具/语言匹配） |
| `management` | 管理能力 | 低（协调类任务专用） |

**匹配算法**：
1. 计算 Task 各维度的 Required Tags
2. 计算 Agent 各维度的 Capability Tags
3. 求交集 + 权重 → 匹配得分
4. 按得分排序，分配给最优 Agent

### 附三、匹配引擎核心逻辑

```
assign_pending_tasks()
    │
    ▼
遍历所有 status=todo 的 Task
    │
    ├── 读取 Task.capability_tags（需要的标签）
    ├── 读取所有在线 Agent.labels（拥有的标签 + 权重）
    │
    ▼
计算匹配得分
    │
    ├── 交集命中：+weight
    ├── 缺失标签：-penalty
    └── 负载过高：-penalty
    │
    ▼
按得分排序 → 分配给最优 Agent
    │
    ▼
更新 Task.assigned_agent + Task.status=pending
```
