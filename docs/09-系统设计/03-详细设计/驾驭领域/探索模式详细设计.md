## 1. 概述

### 1.1 设计目标

Nexus 探索模式是 Reins（御）系统的目标运行模式之一，核心设计原则为 **探索式迭代优化**：通过多轮方案探索、约束调整、人工反馈，逐步收敛到满足条件的最优解。

### 1.2 核心职责

| 职责 | 说明 |
|------|------|
| 模式切换 | 支持 normal/exploration/optimization 三种模式切换 |
| 迭代管理 | 管理探索过程中的迭代轮次 |
| 约束探索 | 约束条件的多轮调整与验证 |
| 人工反馈 | 支持人类对方案的讨论与共识形成 |
| 收敛判断 | 智能判断探索是否收敛 |

## 2. 核心流程

### 2.1 探索模式完整链路

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                           探索模式完整链路                                     │
└──────────────────────────────────────────────────────────────────────────────┘

Step 1: 模式切换
  POST /api/v1/goals/{id}/mode
  mode = "exploration"
  设置 optimization_target / convergence_threshold / max_rounds
         │
         ▼
Step 2: 迭代启动
  POST /api/v1/goals/{id}/start-iteration
  创建首次迭代，初始化约束条件
         │
         ▼
Step 3: 方案生成
  Agent 根据当前约束生成候选方案
         │
         ▼
Step 4: 方案评估
  POST /api/v1/solutions/compare 计算评分
  check_convergence() 判断收敛
         │
         ▼
Step 5: AI 分析
  POST /api/v1/goals/{goal_id}/iterations/{iter_id}/analysis
  生成 AI 分析和建议
         │
         ▼
Step 6: 人工讨论
  POST /api/v1/goals/{goal_id}/iterations/{iter_id}/discuss
  人类与 AI 对话，形成共识
         │
         ▼
Step 7: 约束调整
  共识达成 → 更新约束条件
  创建下一轮迭代
         │
         ▼
Step 8: 收敛完成
  连续 3 次收敛 → 触发自动执行
```

### 2.2 探索模式状态转换

```
[normal] ──── set_mode(exploration) ────▶ [exploration]
                                            │
                                            │ set_mode(optimization)
                                            ▼
                                       [optimization]
                                            │
                                            │ 收敛完成
                                            ▼
                                       [completed]
```

## 3. 数据模型

### 3.1 Goal 探索模式字段

| 字段 | 类型 | 说明 |
|------|------|------|
| mode | VARCHAR(20) | 模式：normal/exploration/optimization |
| optimization_target | VARCHAR(200) | 优化目标描述 |
| convergence_threshold | FLOAT | 收敛阈值（默认 0.05） |
| max_rounds | INT | 最大探索轮次（默认 10） |

### 3.2 GoalIteration 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(50) | 迭代 ID |
| goal_id | VARCHAR(50) | 关联目标 ID |
| iteration_number | INT | 迭代序号（从 1 开始） |
| solution_id | VARCHAR(50) | 关联方案 ID |
| score | FLOAT | 本轮评分 |
| status | VARCHAR(20) | 状态：planned/running/completed |
| ai_analysis | TEXT | AI 分析内容 |
| ai_discussion | TEXT | 讨论记录（JSON 数组） |
| started_at | DATETIME | 开始时间 |
| completed_at | DATETIME | 完成时间 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

## 4. 状态机

### 4.1 Goal 模式状态

| 模式 | 说明 | 可转换到 |
|------|------|---------|
| `normal` | 普通执行模式 | exploration, optimization |
| `exploration` | 探索模式 | optimization, normal |
| `optimization` | 优化模式 | normal, completed |

### 4.2 Iteration 状态机

| 当前状态 | 可转换到 | 触发条件 |
|---------|---------|---------|
| `planned` | `running` | 开始执行本轮迭代 |
| `running` | `completed` | 本轮迭代完成 |
| `running` | `failed` | 本轮迭代失败 |
| `completed` | — | 终态 |

## 7. 错误处理

### 7.1 错误码定义

| 错误码 | 说明 | 处理方式 |
|--------|------|----------|
| 400 | 模式参数错误 | 必须是 normal/exploration/optimization |
| 400 | 目标已激活 | Exploration 模式目标应用 /start-iteration |
| 404 | 目标/迭代不存在 | 返回 404 错误 |
| 409 | 迭代冲突 | 等待当前迭代完成 |

### 7.2 验证规则

```python
# 模式切换验证
if req.mode not in ("normal", "exploration", "optimization"):
    raise HTTPException(
        status_code=400,
        detail="mode must be one of: normal, exploration, optimization"
    )

# 目标存在验证
row = db.execute(
    text("SELECT id FROM goals WHERE id = :id"),
    {"id": goal_id}
).fetchone()
if not row:
    raise HTTPException(status_code=404, detail="Goal not found")

# 迭代记录存在验证
iter_row = db.execute(
    text("SELECT id FROM goal_iterations WHERE id = :id AND goal_id = :gid"),
    {"id": iter_id, "gid": goal_id}
).fetchone()
if not iter_row:
    raise HTTPException(status_code=404, detail="Iteration not found")
```

## 8. 与其他模块的关系

### 8.1 与 Solutions 模块

- 探索过程生成方案（Solutions）
- 迭代记录通过 solution_id 关联方案
- 收敛判断依赖方案评分

### 8.2 与 Task 模块

- 探索模式下的 Task 执行受约束条件影响
- Task 结果反馈给迭代过程

### 8.3 与 Scheduler 模块

- 探索模式下任务调度可能受约束限制
- activate_goal 会触发调度器派发任务
