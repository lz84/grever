## 一、概述

### 1.1 文档目的

本文档定义 Nexus 系统中验证（Verification）与仲裁（Ruling）引擎的详细设计，包括验证触发机制、验证类型、状态流转、人工裁决流程和审计追踪。

### 1.2 核心概念区分

| 概念 | 验证（Verification） | 仲裁（Ruling） |
|------|---------------------|---------------|
| **执行主体** | Verifier Agent（AI，自动执行） | 人类（人工裁决） |
| **触发时机** | Task 报完成且有 acceptance_criteria | 验证循环超过 3 次仍不通过 |
| **目的** | 检查产出是否符合验收标准 | 处理 AI 无法判定的争议 |
| **结果** | 通过 / 不通过 → review_needed | 通过 / 打回 / 调整验收标准 |
| **可重复次数** | 最多 3 次循环 | 不限次数 |

**关键区别**：验证是 AI 自动的质量检查，仲裁是人类处理 AI 无法解决的争议。两者共同构成完整的三级质量保障体系。

## 二、验证引擎架构

### 2.1 Verifier Agent 定位

```
Executor Agent (coder/kouzi)          Verifier Agent (kouzi)
 │                                        │
 领任务 → 执行 → 报完成 ────────────→ 领验证 → 跑验收标准 → 通过/打回
```

- **执行者**（Executor）= 运动员，负责完成任务
- **验证者**（Verifier）= 裁判，负责按验收标准评判产出
- 两者角色分离，确保验收的独立性

### 2.2 三级 Verifier 继承链

Task / Project / Goal 三级均可设置 Verifier Agent，优先级从下往上：

| 优先级 | 来源 | 字段 |
|--------|------|------|
| 最高 | Task | `tasks.verifier_agent_id` |
| 次高 | Project | `projects.verifier_agent_id` |
| 次低 | Goal | `goals.verifier_agent_id` |
| 默认 | 系统 | `kouzi` |

当 Task 未显式设置 Verifier 时，向上查找 Project → Goal → 系统默认值。

## 三、验证触发机制

### 3.1 自动触发流程

Task 完成后自动进入验证流程：

```
Executor Agent 报完成 (POST /api/v1/tasks/{id}/complete)
    ↓
complete_task API 检测 acceptance_criteria
    ├─ 无验收标准 → status=done，completed_at 设置，正常流程结束
    └─ 有验收标准 → status=verifying，推给 Verifier Agent
        ↓
Verifier Agent 执行验证（加载对应 skill）
    ↓
验证结果
    ├─ 全部通过
    │   → 写 comment（详细验证通过意见）
    │   → status=done，completed_at 设置
    │   → 触发依赖解锁
    │
    └─ 有不通过
        → 写 comment（详细验证意见 + 不通过原因）
        → verification_cycle + 1
        ↓
        if cycle < 3:
            → status = review_needed
            → 自动派发回 Executor（附带 comment）
            → Executor 读 comment → 修复 → 重新报完成
        else:
            → status = disputed
            → 转人工裁决中心
            → 飞书通知 + 人工审核
```

### 3.2 手动触发验证

可通过 API 手动触发验证：

```
POST /api/v1/tasks/{task_id}/verify
```

适用于：人类希望重新验证已完成的 Task，或修复后重新检查。

## 四、验证类型

### 4.1 验证类型矩阵

| 类型 | type 值 | 说明 | 适用场景 |
|------|---------|------|---------|
| 编译检查 | `compile` | 执行编译命令，检查返回码 | 代码类任务 |
| API 检查 | `api` | 发送 HTTP 请求，验证状态码和响应结构 | 接口类任务 |
| 页面检查 | `page` | 访问页面 URL，验证渲染结果 | 前端类任务 |
| LLM 辅助审查 | `llm_review` | 用 LLM 做主观质量评估 | 设计/文档类任务 |
| 自定义脚本 | `custom` | 运行自定义脚本验证 | 特殊场景 |

### 4.2 acceptance_criteria Schema

```json
{
  "criteria": [
    {
      "type": "compile|api|page|file|llm_review|custom",
      "desc": "人类可读的验收描述",
      "command": "python -m py_compile xxx.py",
      "work_dir": "D:/work/research/agents-nexus",
      "endpoint": "http://localhost:8090/api/v1/xxx",
      "method": "GET",
      "expect_status": 200,
      "expect_keys": ["key1", "key2"],
      "file_path": "path/to/file.md",
      "min_bytes": 1024,
      "min_lines": 10,
      "keywords": ["关键词1", "关键词2"],
      "format": "markdown|json|yaml",
      "criteria": ["标准1", "标准2"],
      "min_score": 3,
      "script": "scripts/verify_xxx.py",
      "args": ["--flag"]
    }
  ]
}
```

### 4.3 验证策略执行顺序

```
1. 编译检查（快速失败，cost 最低）
2. 文件验证（本地文件系统，cost 低）
3. API 验证（网络请求，cost 中）
4. 自定义脚本（可能耗时，cost 中）
5. LLM 辅助验证（调 LLM，cost 最高）
```

## 五、状态流转

### 5.1 Task 状态机

```
todo ──→ in_progress ──→ verifying ──→ review_needed ────────────────┐
  │              │           │           │                            │
  │              │           │           │ (cycle < 3)                │
  │              │           │           ↓                            │
  │              │           │     review_needed → in_progress（重新派发）│
  │              │           │           │                            │
  │              │           │           │ (cycle >= 3)                │
  │              │           │           ↓                            │
  │              │           │     disputed ──→ [human_ruling] ──→ done/in_progress
  │              │           │                                      │
  │              │           │ (通过)                                │
  │              │           ↓                                      │
  │              │         done                                      │
  │              │                                                  │
  │              └── on_hold                                         │
  └── abandoned                                                        
```

验证结果通过 `task_comments` 表存储，结构化格式：

## 六、仲裁中心设计

### 6.1 仲裁触发条件

当 `verification_cycle >= 3` 且验证仍不通过时，Task 进入 `disputed` 状态，转入人工裁决流程。

### 6.2 人工裁决 API

```
POST /api/v1/tasks/{task_id}/ruling
Body: {
  "ruling": "裁决意见",
  "action": "done | in_progress | verifying"
}
```

**action 说明**：
- `done`：人类判定通过，直接标记 done
- `in_progress`：人类判定打回，任务重新派发给 Executor
- `verifying`：人类调整验收标准，重新触发验证

### 6.3 裁决提交流程

```
人类进入裁决中心 /rulings
    ↓
查看 disputed 任务详情（含所有验证轮次的 comment 历史）
    ↓
选择裁决操作（通过 / 打回 / 调整验收标准）
    ↓
提交裁决（写 human_ruling comment）
    ↓
Task 按 action 更新状态
    ↓
Executor/Verifier 按人类意见执行
```

### 6.4 裁决中心 API

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/v1/human-review/stats` | GET | 统计待处理数量 |
| `/api/v1/human-review/pending` | GET | 裁决中心列表 |
| `/api/v1/human-review/batch-ruling` | POST | 批量裁决 |
| `/api/v1/tasks/{task_id}/ruling` | POST | 单个任务裁决 |
| `/api/v1/tasks/{task_id}/verifications` | GET | 验证历史 |

### 6.5 human_ruling Comment 格式

```json
{
  "task_id": "task-xxx",
  "author": "用户",
  "author_role": "human",
  "type": "human_ruling",
  "content": "裁决: 编译错误可以忽略，这是已知问题。API 和页面检查已通过，标记为 done。",
  "metadata": {
    "ruling_action": "done",
    "overrides_verification": true
  },
  "created_at": "2026-05-05T11:00:00Z"
}
```

## 八、ResultVerifier 类设计

### 8.1 类签名

```python
class ResultVerifier:
    DEFAULT_VERIFIER = "kouzi"
    MAX_VERIFICATION_CYCLES = 3

    def resolve_effective_verifier(self, task_id: str) -> str:
        """三级继承链解析：Task → Project → Goal → DEFAULT"""

    def trigger_verification(self, task_id: str, result: str, success: bool) -> Dict:
        """触发验证流程"""

    def _run_verifier_checks(self, task_id: str, result: str) -> Tuple[bool, str]:
        """运行验收标准，返回 (passed, detail)"""

    def _write_verification_comment(
        self, task_id: str, verifier: str, cycle: int, passed: Optional[bool], detail: str
    ) -> str:
        """写入验证 comment 到 task_comments 表"""
```

### 8.2 核心逻辑

```python
def trigger_verification(self, task_id: str, result: str, success: bool) -> Dict:
    task = self.db.get_task(task_id)
    verifier = self.resolve_effective_verifier(task_id)
    passed, detail = self._run_verifier_checks(task_id, result)
    cycle = getattr(task, 'verification_cycle', 0) or 0

    if passed:
        self._write_verification_comment(task_id, verifier, cycle + 1, True, detail)
        task.status = "done"
        task.completed_at = datetime.now()
        task.result_summary = result
        self.db.commit()
        self._unlock_dependent_tasks(task_id)
        return {"passed": True, "action": "done"}
    else:
        cycle += 1
        if cycle >= self.MAX_VERIFICATION_CYCLES:
            # 超过上限 → disputed
            self._write_verification_comment(task_id, verifier, cycle, False, detail)
            self._write_verification_comment(
                task_id, verifier, cycle, None,
                f"⚠️ {cycle}次验证未通过，转人工裁决"
            )
            task.status = "disputed"
            task.verification_cycle = cycle
            task.result_summary = result
            task.error_message = detail
            self.db.commit()
            self._notify_human(task_id, cycle, detail)
            return {"passed": False, "action": "disputed", "verification_cycle": cycle}
        else:
            # cycle < 3 → review_needed，等 heartbeat 重新派发
            self._write_verification_comment(task_id, verifier, cycle, False, detail)
            task.status = "review_needed"
            task.verification_cycle = cycle
            task.result_summary = result
            task.error_message = detail
            self.db.commit()
            return {"passed": False, "action": "review_needed", "verification_cycle": cycle}
```

## 九、已知名义与限制

| 条目 | 说明 |
|------|------|
| review_needed 派发依赖 heartbeat | Heartbeat 端点需包含 `status IN ('todo', 'pending', 'review_needed')` |
| verification_cycle 初始值 | Task 首次报完成时 cycle=0，后续递增 |
| 硬删除限制 | disputed 任务不允许硬删除，只能通过裁决解决 |
| 飞书通知 | disputed 产生时推送飞书消息（辅助闹钟，不替代 Nexus 内处理） |
