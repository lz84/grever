## 1. 概述

**Vigil（鉴）** 是 Nexus 平台的安全卫士，负责信任评估体系、安全审计和日志管理、外部访问控制和权限管理、内部行为监控和异常检测。

**核心能力**:
- 双重审计（外审 + 内审）
- 实时异常检测
- **⭐ 信任评估体系**（Agent/用户/系统三维评分）
- 权限管理和认证
- 合规性保障

### 1.1 职责划分

**Vigil vs Grasp 的职责边界**:

| 功能 | Vigil（鉴） | Grasp（悟） |
|------|------------|------------|
| **信任评估体系** | ✅ 核心职责 | ❌ 不属于 |
| **认知投毒检测** | ❌ 已迁移 | ✅ 核心职责 |
| **知识质量验证** | ❌ 不属于 | ✅ 核心职责 |
| 外部访问控制 | ✅ 核心职责 | ❌ |
| 用户行为审计 | ✅ 核心职责 | ❌ |
| 系统级异常检测 | ✅ 核心职责 | ❌ |
| 合规性保障 | ✅ 核心职责 | ❌ |

**关键区分**:
- **信任评估（Vigil）**:评估 Agent/用户/系统的可信度，决定权限等级
- **认知安全（Grasp）**:防止错误认知污染知识库，评估知识质量

### 1.2 接口标准对标

**对标标准**:
- **Syslog RFC 5424**:日志格式
- **OPA (Open Policy Agent)**:策略管理
- **SIEM 系统**:安全信息和事件管理
- **OpenAPI 3.0**:RESTful 接口规范

## 3. 外审 vs 内审

### 3.1 外审（对外安全）

**定位**:保护 Nexus 平台免受外部威胁和未授权访问

#### 3.1.1 RBAC 权限管理

**角色定义**:

| 角色 | 权限范围 | 适用场景 |
|------|---------|---------|
| `super_admin` | 所有权限 | 系统管理员 |
| `admin` | 管理权限（不含系统配置） | 团队管理员 |
| `developer` | 开发相关权限 | 开发人员 |
| `viewer` | 只读权限 | 观察者 |
| `api_user` | API 访问权限 | 第三方系统 |

**权限矩阵**:

| 权限类型 | super_admin | admin | developer | viewer | api_user |
|---------|-------------|-------|-----------|--------|----------|
| 查看 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 创建 | ✓ | ✓ | ✓ | - | ✓ |
| 修改 | ✓ | ✓ | ✓ | - | - |
| 删除 | ✓ | ✓ | - | - | - |
| 管理 | ✓ | - | - | - | - |
| 审计 | ✓ | ✓ | - | - | - |

**实现说明**:

访问控制器通过角色权限映射实现 RBAC 访问控制。系统加载预设的角色权限配置，查询用户所属角色，合并角色的权限集合，判断用户是否拥有所需权限。管理员访问判断通过检查用户角色是否与管理员角色集合有交集。

#### 3.1.2 外部访问控制

**访问控制策略**:

- **认证方式**:支持 JWT、API Key、OAuth2 等多种认证方式
- **JWT 配置**:使用 RS256 算法，过期时间 3600 秒
- **API Key 配置**:通过 X-API-Key 头传递，每 90 天轮换
- **限流策略**:默认每分钟 60 次请求，每小时 1000 次请求

### 3.2 内审（对内安全）

**定位**:监控和分析 Nexus 平台内部行为，确保系统安全性和合规性

**核心能力**:
- 行为审计
- 异常检测
- **⭐ 信任评估体系**（Agent/用户/系统）

#### 3.2.1 行为审计

**审计范围**:

| 审计对象 | 审计内容 | 保留期限 |
|---------|---------|---------|
| 用户操作 | 登录、API 调用、资源创建/修改/删除 | 3 年 |
| 系统事件 | 服务启动、配置变更、系统告警 | 2 年 |
| Agent 执行 | 任务执行、技能调用、资源使用 | 1 年 |
| 安全事件 | 认证失败、异常访问、违规操作 | 永久 |

#### 3.2.2 异常检测

**检测模型**:

异常行为检测器通过规则检查和统计检查两种方式检测异常：
- **规则检查**:基于预设规则检测明显违规行为
- **统计检查**:基于统计模型检测偏离基线的异常行为

#### 3.2.3 信任评估体系（核心职责）

**定位**:信任评估是 Vigil 的核心能力，用于动态评估 Agent、用户和系统的可信度，并基于评估结果动态调整权限。

**评估对象**:

| 对象类型 | 评估内容 | 应用场景 |
|---------|---------|---------|
| Agent | 任务完成能力、可靠性、安全合规 | 分配敏感任务、权限等级 |
| 用户 | 操作行为、使用习惯、历史违规 | 资源访问限制、审批流程 |
| 系统 | 服务稳定性、异常频率、资源使用 | 服务等级、流量控制 |

##### 3.2.3.0 模型架构与设计-实现关系

信任评估采用**三维目标模型**，分两阶段实现：

```
┌────────────────────────────────────────────────────────────┐
│                     信任评估（Trust）                       │
│                                                            │
│  ┌──────────────────────┐    ┌──────────────────────────┐  │
│  │     可用性 (70%)      │    │      安全性 (30%)         │  │
│  │                      │    │                          │  │
│  │ ┌──────────────────┐ │    │  越权 / 脱敏 / 审计...   │  │
│  │ │ 能力 (40%)       │ │    │  ┌────────────────────┐  │  │
│  │ │ - 任务完成率 50%  │ │    │  │   安全维度 (30%)    │  │  │
│  │ │ - 质量分   30%   │ │    │  │   待 Phase 2 接入   │  │  │
│  │ │ - 复杂度适应 20% │ │    │  └────────────────────┘  │  │
│  │ └──────────────────┘ │    │                          │  │
│  │ ┌──────────────────┐ │    │                          │  │
│  │ │ 可靠性 (30%)     │ │    │                          │  │
│  │ │ - 稳定性   40%   │ │    │                          │  │
│  │ │ - 异常率   30%   │ │    │                          │  │
│  │ │ - 响应时间 20%   │ │    │                          │  │
│  │ │ - 恢复能力 10%   │ │    │                          │  │
│  │ └──────────────────┘ │    │                          │  │
│  │                      │    │                          │  │
│  │  ← Phase 1 已实现 →  │    │   ← Phase 2 待实现 →     │  │
│  │   (5因子工程化模型)  │    │                          │  │
│  └──────────────────────┘    └──────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
```

**Phase 1 — 可用性评估（当前代码实现）**：

代码中的 `TrustEvaluator` 使用 5 因子扁平加权模型，对应三维模型中的"能力"+"可靠性"维度：

| 工程因子 | 代码权重 | 映射到设计维度 | 说明 |
|---------|---------|--------------|------|
| `success_rate` | 35% | 能力 → 任务完成率 | 带时间衰减的成功率（7/30/90天三档） |
| `quality` | 15% | 能力 → 质量分 | 任务质量评分均值 |
| `timeliness` | 20% | 可靠性 → 响应时间 | <1s满分，>30s零分，线性插值 |
| `consistency` | 20% | 可靠性 → 稳定性 | 最近20条成功/失败交替频率 |
| `longevity` | 10% | *(辅助因子)* | 冷启动辅助：log(任务数)/3 × 60% + 活跃天数/90 × 40% |

> **设计说明**：`longevity` 是冷启动辅助因子，用于新 Agent 缺乏足够历史数据时给基础分。在完整三维模型中不作为核心信任指标。安全评分 < 0.5 时一票否决（禁止访问），不受可用性分影响。

**Phase 2 — 安全维度接入（待实现）**：

当 `vigil/access/`（访问控制）和 `vigil/common/audit.py`（审计日志）模块就绪后，接入安全维度 4 个指标：

| 安全指标 | 设计权重 | 数据源 | 状态 |
|---------|---------|--------|------|
| 越权次数 | 40% | 访问控制审计日志 | ❌ 待实现 |
| 脱敏合规 | 30% | 日志脱敏引擎 | ❌ 待实现 |
| 安全事件 | 20% | 安全事件记录 | ❌ 待实现 |
| 审计通过率 | 10% | 行为审计结果 | ❌ 待实现 |

**Phase 3 — 模型统一**：

将 `_aggregate_score` 从 5 因子扁平加权改为两层结构：`可用性×0.7 + 安全性×0.3`，保持代码与三维设计模型完全一致。

**指标**:

**评分等级**:

| 等级 | 分数范围 | 说明 |
|------|---------|------|
| 安全级 | ≥ 0.95 | 最高安全等级 |
| 可信级 | ≥ 0.85，< 0.95 | 可信 |

| 禁止级 | < 0.5 | 禁止访问（一票否决） |

##### 3.2.3.4 综合信任评分

**Phase 1 计算公式**（可用性评估）:

```
trust_score = success_rate×0.35 + consistency×0.20 + timeliness×0.20 + quality×0.15 + longevity×0.10
              - 连续失败惩罚（每次 5%，最多 30%）
```

**目标计算公式**（Phase 3 完全体）:

```
trust_score = 能力(40%) × 能力评分 + 可靠性(30%) × 可靠性评分 + 安全(30%) × 安全评分
```

其中安全评分 < 0.5 时一票否决，直接判定为"禁止访问"。

**信任等级判定逻辑**（Phase 2 起生效）:
- 如果安全评分 < 0.5 → **"禁止访问"**（一票否决）
- 综合 ≥ 0.9 且 安全 ≥ 0.95 → **"完全信任"**
- 综合 ≥ 0.75 且 安全 ≥ 0.85 → **"高度可信"**
- 综合 ≥ 0.6 且 安全 ≥ 0.7 → **"有限信任"**
- 综合 ≥ 0.4 且 安全 ≥ 0.5 → **"受限信任"**
- 其他 → **"低信任度"**

**信任等级与权限映射**:

| 信任等级 | 能力要求 | 安全要求 | 权限范围 | 任务分配 |
|---------|---------|---------|---------|---------|
| **完全信任** | L4+ | 安全级 | 全部权限 | 所有任务 |
| **高度可信** | L3+ | 可信级 | 大部分权限 | 敏感任务 |
| **有限信任** | L2+ | 观察级 | 部分权限 | 标准任务 |
| **受限信任** | L1+ | 限制级 | 受限权限 | 简单任务 |
| **低信任度** | - | 禁止级 | 禁止访问 | 禁止任务 |

**信任评估 API**:

- **获取 Agent 信任评分**: `GET /api/v1/vigil/trust/agents/{agent_id}`
- **更新信任评分（定时触发）**: `POST /api/v1/vigil/trust/agents/{agent_id}/reassess`
- **获取信任历史记录**: `GET /api/v1/vigil/trust/agents/{agent_id}/history`
- **信任降级告警**: `GET /api/v1/vigil/trust/alerts/downgrade`

**信任评估周期**:

| 评估类型 | 周期 | 触发条件 |
|---------|------|---------|
| 实时评估 | 每次任务后 | 任务完成、异常事件 |
| 定时评估 | 每小时 | 定时任务 |
| 深度评估 | 每天 | 综合所有维度 |
| 紧急评估 | 触发时 | 安全事件、越权尝试 |

#### 3.2.4 认知投毒检测

**❌ 已迁移到 Grasp（悟）**

从 v1.3 版本开始，认知投毒检测功能已完整迁移到 Grasp（悟）Agent。

**迁移原因**:
1. **职责分离**:认知安全应该是 Grasp 的核心职责
2. **完整性**:Grasp 作为认知安全守护者，需要内建防护机制
3. **效率**:减少跨 Agent 通信开销
4. **一致性**:知识创建、验证、审核一体化

**Grasp 的认知安全机制**:

```
Grasp（悟）= 认知底座 + 认知安全
├── 认知存储（What/How/Why/Lessons/Meta）
├── 认知安全
│   ├── 认知投毒检测（核心职责）
│   ├── 知识质量验证
│   └── 信任评估体系
└── 不是组件，是独立产品
```

**Vigil 的安全职责（保留部分）**:
- 外部访问控制（外审）
- 用户行为审计
- 系统级异常检测
- 合规性保障

**协作关系**:

Grasp 负责知识层面的安全防护，Vigil 负责系统层面的安全防护，两者协同工作，提供完整的安全保障。

## 5. 告警管理

### 5.1 告警规则

**告警规则示例**:

- **登录失败次数超过阈值**:当 5 分钟内登录失败次数大于 5 次时，触发警告级别告警，通知安全团队和用户
- **检测到权限提升行为**:当检测到权限变更且严重程度大于高时，触发关键级别告警，通知安全团队和管理员

## 7. 与五兄弟的协作

### 7.1 Vigil ↔ Grasp（职责边界）

**协作关系**:

Vigil 负责信任评估体系（Agent/用户可信度），Grasp 负责认知安全（投毒检测、知识质量），两者协同工作，提供完整的安全保障。

### 7.2 Vigil ↔ Reins（执行监督）

**协作流程**:

Reins 执行工作流 → Vigil 监控执行过程 → 记录审计日志

### 7.3 Vigil → Reach（权限控制）

**协作流程**:

Reach 准备调用工具 → Vigil 检查权限 + 信任评估 → 允许/拒绝

### 7.4 Vigil ← Evo（异常检测）

**协作流程**:

Evo 提供行为基线 → Vigil 检测异常 → 告警通知
