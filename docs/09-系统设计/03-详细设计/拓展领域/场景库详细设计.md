## 一、概述

### 1.1 文档目的

本文档是场景库系统的详细设计文档，聚焦于场景模板与场景实例的区别与关系、场景实例化流程、三层上下文体系、条件分支设计以及场景执行链路。是架构设计文档 [21-scenario-library-system-design.md](../../02-架构设计/21-scenario-library-system-design.md) 的实现层补充。

### 1.2 核心概念

| 概念 | 说明 |
|------|------|
| **场景模板** | 静态的、可复用的解决方案蓝图，事前定义的所有可能执行路径的全集（fullset） |
| **场景实例** | 根据具体 Goal 从场景模板实例化出来的实际执行计划，是模板的子集 |
| **上下文注入** | 将 Goal/Project/Task 三层上下文信息注入到场景实例的过程 |
| **条件分支** | 场景中预设的决策节点，根据条件表达式或人工输入决定下一步执行路径 |

## 二、场景模板与场景实例的区别和关系

### 2.1 数据模型区别

场景模板（`scenarios` 表）和场景实例（`goal_scenario_instances` 表）是两个独立的概念：

| 维度 | 场景模板（Scenario） | 场景实例（GoalScenarioInstance） |
|------|---------------------|-------------------------------|
| **存储位置** | `scenarios` 表 | `goal_scenario_instances` 表 |
| **性质** | 静态知识资产，可复用 | 动态执行记录，与 Goal 绑定 |
| **生命周期** | 独立版本管理，可发布/归档 | 随 Goal 消亡（Goal 完成/终止） |
| **内容** | 全集（fullset）：所有可能的 Project + Task + 条件 | 实例：从全集中裁剪出的实际 Project + Task 集合 |
| **可修改性** | 场景作者可编辑，更新版本号 | 引擎按预设条件增删，不可直接编辑 |
| **查询方** | 驾驭中心（匹配场景） | 驾驭中心（调度执行） |

### 2.2 数据结构对比

**场景模板的 fullset 结构**（存储在 `scenarios.fullset` JSON 字段中）：

```json
{
  "goal_tags": {
    "business": ["emergency_response"],
    "professional": ["incident_command"],
    "technical": [],
    "management": ["coordination", "task_dispatch"]
  },
  "projects": [
    {
      "name": "应急响应",
      "type": "mandatory",
      "capability_tags": { "business": ["emergency_response"], "management": ["coordination"] },
      "tasks": [
        { "name": "启动应急预案", "type": "mandatory", "capability_tags": { "management": ["coordination"] } }
      ]
    },
    {
      "name": "人员疏散",
      "type": "conditional",
      "condition": {
        "type": "human_decision",
        "prompt": "根据风向，是否需要疏散周边人员？",
        "options": ["立即疏散", "暂缓监测", "不需要"],
        "default": "暂缓监测",
        "timeout_action": "use_default",
        "timeout_minutes": 30,
        "branches": {
          "立即疏散": { "next_projects": ["疏散执行"] },
          "暂缓监测": { "next_projects": ["持续监测"] },
          "不需要": null
        }
      }
    }
  ]
}
```

**场景实例的结构**（存储在 `goal_scenario_instances` 表中）：

```json
{
  "id": "inst-xxx",
  "goal_id": "goal-xxx",
  "scenario_id": "scenario-xxx",
  "scenario_version": "v1.2",
  "status": "active",
  "instantiated_projects": [
    { "project_id": "proj-001", "source": "mandatory", "status": "created" },
    { "project_id": "proj-002", "source": "conditional:wind=north", "status": "pending" }
  ],
  "context_snapshot": { /* 实例化时的上下文快照 */ }
}
```

### 2.3 两者的关系

```
场景模板（scenarios）
     │
     │ 实例化
     ▼
场景实例（goal_scenario_instances）
     │
     │ 提供蓝图
     ▼
Goal 的 Projects + Tasks（reins_projects, reins_tasks）

关系约束：
1. 场景实例.project_id ⊆ 场景模板.fullset.projects（所有实例化出的项目必须存在于模板全集中）
2. 一个场景模板可实例化出多个场景实例（每个 Goal 一个）
3. 场景模板更新版本后，已有的场景实例不受影响（按实例化时的版本继续执行）
```

### 2.4 模板版本与实例版本

场景模板具有版本管理（`scenarios.versions` 字段），每个场景实例在实例化时锁定版本号：

```sql
-- 查询实例使用哪个版本的模板
SELECT gsi.id, gsi.scenario_version, s.fullset->>'$.name' as scenario_name
FROM goal_scenario_instances gsi
JOIN scenarios s ON s.id = gsi.scenario_id AND s.version = gsi.scenario_version;
```

## 三、场景实例化流程

### 3.1 实例化触发时机

场景实例化发生在用户创建一个 Goal 并选择了具体场景之后：

```
用户操作：创建 Goal → 选择场景模板 → 系统实例化 → 生成初始 Project + Task 列表
```

### 3.2 实例化流程步骤

```
┌─────────────────────────────────────────────────────────────┐
│ 步骤 1：解析场景模板 fullset                                │
│                                                              │
│ 读取 scenarios 表，取出 fullset JSON：                       │
│ - goal_tags：Goal 级别的能力需求                             │
│ - projects：所有可能的 Project（含 mandatory + conditional） │
│ - 每个 Project 下的 tasks                                   │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 2：收集当前上下文（注入）                                │
│                                                              │
│ 从以下来源收集信息：                                         │
│ - Goal 的 title、description                                 │
│ - 用户在创建 Goal 时填写的表单字段（context_fields）         │
│ - Project 当前已有的上下文（如果是从已有 Project 升级而来）  │
│                                                              │
│ 产出：context_snapshot（实例化时的上下文快照）               │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 3：执行初始裁剪                                        │
│                                                              │
│ 遍历 fullset.projects：                                     │
│                                                              │
│ if project.type == "mandatory":                            │
│     → 创建 Project，加入场景实例                             │
│                                                              │
│ if project.type == "conditional":                           │
│     → 评估 condition.expression                             │
│     → 如果条件已满足（在 context_snapshot 中可判定）：       │
│         → 创建 Project，标记 source="conditional:xxx"         │
│     → 如果条件未满足：                                       │
│         → 不创建，但记录条件到 pending_conditions             │
│                                                              │
│ 产出：instantiated_projects 列表（已创建 + 待创建）          │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 4：为每个 mandatory Project 创建 Task                   │
│                                                              │
│ 遍历已创建的 Projects：                                     │
│ - 从 fullset.tasks 中取出该 Project 下的所有 mandatory tasks │
│ - 创建 reins_tasks 记录                                     │
│ - conditional tasks 暂不创建（等待条件触发）                 │
│                                                              │
│ 产出：reins_tasks 初始列表                                   │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 步骤 5：保存场景实例                                        │
│                                                              │
│ 写入 goal_scenario_instances 表：                           │
│ - goal_id、scenario_id、scenario_version                   │
│ - status = "active"                                        │
│ - instantiated_projects：已创建 + 待创建项目列表             │
│ - context_snapshot：上下文快照                             │
│ - pending_conditions：待触发条件列表                        │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 实例化 API

```python
# POST /api/v1/goals/{goal_id}/instantiate-scenario
async def instantiate_scenario(
    goal_id: str,
    scenario_id: str,
    context: dict[str, Any]  # 用户提供的额外上下文
) -> GoalScenarioInstance:
    """
    将场景模板实例化为 Goal 的执行计划
    """
    # 1. 加载场景模板
    scenario = await scenarios_crud.get_scenario(scenario_id)
    fullset = scenario.fullset

    # 2. 合并上下文（Goal 描述 + 用户输入 + 已有上下文）
    merged_context = await _merge_context(goal_id, context)

    # 3. 裁剪初始 Project
    projects_to_create, pending = _prune_projects(fullset["projects"], merged_context)

    # 4. 创建 Projects 和 Tasks
    created_projects = await _create_projects(goal_id, projects_to_create)

    # 5. 保存实例
    instance = GoalScenarioInstance(
        goal_id=goal_id,
        scenario_id=scenario_id,
        scenario_version=scenario.version,
        status="active",
        instantiated_projects=[{"project_id": p.id, "source": p.source, "status": "created"}
                               for p in created_projects],
        pending_conditions=pending,
        context_snapshot=merged_context
    )
    await db.commit()
    return instance
```

### 3.4 条件评估函数

```python
def _evaluate_condition(condition: dict, context: dict) -> tuple[bool, str | None]:
    """
    评估条件分支是否满足
    返回: (是否满足, 满足时的分支标识)
    """
    ctype = condition.get("type")

    if ctype == "none":
        # 无条件，直接满足
        return True, None

    if ctype == "auto_eval":
        # 自动评估：执行表达式
        expr = condition.get("expression", "")
        try:
            # 使用安全评估器（不允许代码执行）
            result = _safe_eval(expr, context)
            return bool(result), None
        except Exception:
            return False, None

    if ctype in ("human_decision", "human_input"):
        # 人工决策/输入类：需要人类响应，初始状态为未满足
        return False, None

    return False, None
```

## 四、三层上下文体系与注入机制

### 4.1 上下文层级定义

Nexus 的上下文分为三个层级，按范围从大到小：

| 层级 | 范围 | 生命周期 | 说明 |
|------|------|---------|------|
| **Goal 上下文** | 整个 Goal | Goal 存在期间 | Goal 的 title/description、创建者、关联场景、用户提供的初始信息 |
| **Project 上下文** | 单个 Project | Project 存在期间 | Project 的名称、capability_tags、关联的 Project 列表、上一步产出 |
| **Task 上下文** | 单个 Task | Task 执行期间 | Task 的 description、agent 分配、验证规则、上游 Task 的产出 |

### 4.2 上下文注入到场景实例的过程

```
┌─────────────────────────────────────────────────────────────┐
│ Goal 上下文注入                                             │
│                                                              │
│ 在创建 Goal 时，收集：                                       │
│ - title: "某化工厂危化品泄漏"                               │
│ - description: "储罐区发生泄漏，有人报告"                    │
│ - created_by: user-xxx                                      │
│ - scenario_id: chemical-leak-001                            │
│                                                              │
│ → 写入 goals 表                                             │
│ → 传递给场景实例化器作为全局上下文                           │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ Project 上下文注入                                           │
│                                                              │
│ 实例化时，为每个创建的 Project 注入：                         │
│ - goal_context：引用 Goal 上下文（goal_id）                  │
│ - project_name：来自场景模板 fullset                        │
│ - capability_tags：来自场景模板                             │
│ - upstream_outputs：前置 Project 的产出物（如果有）           │
│                                                              │
│ → 写入 reins_projects 表（context 字段 JSON）               │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ Task 上下文注入                                              │
│                                                              │
│ 创建 Task 时，注入：                                         │
│ - project_context：引用 Project 上下文                       │
│ - task_name：来自场景模板 tasks 列表                        │
│ - assigned_capabilities：所需能力标签                        │
│ - verification_rules：验证规则（来自模板或人工配置）         │
│ - upstream_task_outputs：上游 Task 的产出                   │
│                                                              │
│ → 写入 reins_tasks 表（context 字段 JSON）                  │
└─────────────────────────────────────────────────────────────┘
```

### 4.4 上下文传递规则

上下文从上层向下层传递，子层级可以读取父上下文的引用，但不可修改：

```
Goal 上下文
     │
     ├── goal_id ✅ 可读
     ├── goal_title ✅ 可读
     └── scenario_id ✅ 可读
          │
          ▼
     Project 上下文（可读 Goal 上下文的引用）
     │
     ├── project_name ✅ 可读
     ├── goal_context_ref ✅ 可读（goal_id）
     └── upstream_outputs ✅ 可读（前置 Project 的产出）
          │
          ▼
     Task 上下文（可读 Project 上下文的引用）
     │
     ├── task_name ✅ 可读
     ├── project_context_ref ✅ 可读（project_id）
     └── upstream_outputs ✅ 可读（前置 Task 的产出）
```

## 五、条件分支设计

### 5.1 四种条件类型

场景模板中 `condition.type` 字段支持四种类型：

| 类型 | 值 | 说明 | 实例化时行为 | 执行时行为 |
|------|-----|------|------------|----------|
| **无条件** | `none` | 无条件执行 | 立即创建 | 立即执行 |
| **自动评估** | `auto_eval` | 表达式自动判定 | 评估 expression，为真则创建 | 无需等待 |
| **人工决策** | `human_decision` | 人类选择分支 | 暂不创建，等待人工 | 人工选择后按分支执行 |
| **人工输入** | `human_input` | 人类提供信息 | 暂不创建，等待人工 | 人工输入后继续执行 |

### 5.2 各类型数据结构

**type: none（无条件）**

```json
{
  "name": "应急响应启动",
  "type": "mandatory",
  "condition": { "type": "none" }
}
```

**type: auto_eval（自动评估）**

```json
{
  "name": "泄漏源控制",
  "type": "conditional",
  "condition": {
    "type": "auto_eval",
    "expression": "leak_source_confirmed == true",
    "description": "泄漏源已确认时触发"
  }
}
```

**type: human_decision（人工决策）**

```json
{
  "name": "疏散决策",
  "type": "conditional",
  "condition": {
    "type": "human_decision",
    "prompt": "根据当前风向，是否需要疏散周边人员？",

      "不需要": { "next_projects": [], "skip_projects": ["疏散相关"] }

  }
```

**type: human_input（人工输入）**

```json
{

  "type": "conditional",
  "condition": {
    "type": "human_input",
    "input_type": "number",

    "default": 0,
    "timeout_action": "use_default",
    "timeout_minutes": 60,
    "var_name": "estimated_leak_volume",
    "branches": {
      "action": "inject_context"
    }
  }
}
```

### 5.3 条件分支执行引擎

```python
# 条件分支评估核心逻辑
class ScenarioConditionEngine:
    """
    场景条件分支执行引擎
    负责评估条件类型、触发 Project/Task 创建、注入上下文
    """

    async def evaluate_and_trigger(self, instance_id: str, db: Session):
        """
        评估场景实例的所有 pending_conditions
        触发满足条件的 Project/Task 创建
        """
        instance = await self._load_instance(instance_id)

        # 获取当前全局上下文（Goal + 已完成 Project 的产出）
        global_context = await self._build_global_context(instance.goal_id)

        # 遍历所有 pending_conditions
        for cond in instance.pending_conditions:
            satisfied, branch_key = self._evaluate_condition(
                cond.condition, global_context
            )

            if satisfied:
                # 满足条件，触发 Project 创建
                await self._trigger_projects(
                    instance, cond, branch_key, global_context, db
                )
                # 从 pending 中移除
                instance.pending_conditions.remove(cond)

        await db.commit()

    def _evaluate_condition(self, condition: dict, context: dict) -> tuple[bool, str | None]:
        ctype = condition["type"]

        if ctype == "none":
            return True, None

        if ctype == "auto_eval":
            expr = condition["expression"]
            result = _safe_eval(expr, context)
            return bool(result), None

        if ctype in ("human_decision", "human_input"):
            # 这两类在人类响应前不满足
            return False, None

        return False, None
```

### 5.4 custom-create API 对条件分支的支持

`custom-create` API 用于创建自定义（不在模板中预定义）的 Project 或 Task。为支持条件分支扩展，该 API 在创建时需支持关联条件：

```python
# POST /api/v1/projects/custom-create
async def custom_create_project(
    goal_id: str,
    name: str,
    capability_tags: dict,
    trigger_condition: dict | None,  # 新增：触发条件
    context: dict
) -> Project:
    """
    创建自定义 Project，并可选关联触发条件
    """
    project = Project(
        goal_id=goal_id,
        name=name,
        capability_tags=capability_tags,
        context=context,
        status="created",
        source="custom"
    )

    if trigger_condition:
        # 将触发条件注册到场景实例的 pending_conditions
        instance = await _get_active_instance(goal_id)
        instance.pending_conditions.append(PendingCondition(
            condition=trigger_condition,
            target_project_id=project.id,
            target_type="project"
        ))
        project.status = "pending_trigger"  # 等待条件触发

    await db.commit()
    return project
```

## 六、场景执行链路

### 6.1 完整执行链路

场景执行链路从场景实例化开始，到 Goal 完成或终止结束：

```
┌─────────────────────────────────────────────────────────────┐
│ 阶段 1：场景实例化（Goal 创建 + 选择场景）                     │
│                                                              │
│ 用户创建 Goal，选择场景模板                                    │
│ → 实例化器解析 fullset                                         │
│ → 创建初始 mandatory Projects + Tasks                        │
│ → conditional Projects 加入 pending_conditions                │
│ → 场景实例状态：active                                        │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段 2：Task 调度与派发（驾驭中心）                          │
│                                                              │
│ 调度器按依赖顺序派发 Task：                                   │
│ - Task.status = "todo"                                       │
│ - 匹配 Agent（能力标签 + 负载）                               │
│ - 派发到最佳 Agent                                           │
│ - Task.status = "in_progress"                                │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段 3：Task 执行（Agent 端）                                │
│                                                              │
│ Agent 领取 Task：                                             │
│ - 读取 Task.context（包含 capability_tags、verification_rules）│
│ - 执行任务逻辑                                               │
│ - 上报心跳（报告进度、产出）                                  │
│ - 完成后提交结果                                              │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段 4：Task 完成与验证                                       │
│                                                              │
│ - Task.status = "done"                                       │
│ - 验证规则执行（编译检查/API检查/页面检查）                   │
│ - 验证通过 → 上游 Project 累计 progress                      │
│ - 验证失败 → 重试（最多 3 次）→ 进入 disputed 或 failed      │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段 5：Project 完成与条件触发检查                            │
│                                                              │
│ Project 内所有 mandatory Tasks 完成：                         │
│ → Project.status = "completed"                               │
│ → 记录产出到 global_context                                  │
│ → 条件引擎检查 pending_conditions                            │
│ → 满足 auto_eval 条件 → 触发 conditional Projects 创建        │
│ → 满足 human_decision → 暂停，等待人工决策                    │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ 阶段 6：Goal 完成                                            │
│                                                              │
│ 所有 mandatory Projects 完成：                                │
│ → Goal.status = "completed"                                 │
│ → 场景实例状态：completed                                    │
│ → 执行反馈回流（统计写入场景库）                              │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 场景 → Task → 执行 → 完成的数据流

```
场景模板（scenarios.fullset）
     │
     │ 实例化
     ▼
场景实例（goal_scenario_instances.instantiated_projects）
     │
     │ 为每个 Project 创建
     ▼
Projects（reins_projects）
     │
     │ 为每个 Project 创建 Tasks
     ▼
Tasks（reins_tasks）
     │
     │ 调度器派发
     ▼
Agent 领取（Task.assigned_agent）
     │
     │ Agent 执行 + 心跳上报
     ▼
Task 执行结果（Task.result）
     │
     │ 验证
     ▼
Task.status 更新（done / failed / disputed）
     │
     │ 验证通过，Project 进度累计
     ▼
Project.status 更新（active → completed）
     │
     │ 所有 mandatory Projects 完成
     ▼
Goal.status 更新（in_progress → completed）
```

### 6.3 条件触发后的动态 Project 创建

当条件满足（auto_eval 为真或 human_decision 人工选择）时，引擎从场景全集中取出对应的 Project：

```python
async def _trigger_projects(
    instance: GoalScenarioInstance,
    cond: PendingCondition,
    branch_key: str | None,
    context: dict,
    db: Session
):
    """
    条件满足时，从场景全集中取出对应的 Project 创建到 Goal
    """
    scenario = await scenarios_crud.get_scenario(instance.scenario_id)
    fullset = scenario.fullset

    # 获取分支定义
    branches = cond.condition.get("branches", {})
    branch = branches.get(branch_key) if branch_key else {}

    # 要创建的项目列表
    next_projects = branch.get("next_projects", [])
    skip_projects = branch.get("skip_projects", [])

    # 创建 next_projects 中的项目
    for proj_name in next_projects:
        proj_def = next(
            (p for p in fullset["projects"] if p["name"] == proj_name),
            None
        )
        if proj_def:
            await _create_project_from_def(
                instance.goal_id, proj_def, f"conditional:{branch_key}", db
            )

    # 标记 skip_projects 为跳过
    for proj_name in skip_projects:
        # 查找并标记
        pass

    # 如果是 human_input 类型，还需要注入变量到上下文
    if cond.condition.get("type") == "human_input":
        var_name = cond.condition.get("var_name")
        var_value = context.get(var_name)
        # 将变量注入到全局上下文，供后续 Task 使用
        await _inject_to_context(instance.goal_id, var_name, var_value, db)
```

## 七、错误处理

### 7.1 实例化失败

| 错误场景 | 处理策略 | 错误码 |
|---------|---------|--------|
| 场景模板不存在 | 返回 404 | `SCENARIO_NOT_FOUND` |
| 场景模板 fullset 格式错误 | 返回 500，记录日志 | `SCENARIO_INVALID_FULLSET` |
| Goal 上下文解析失败 | 返回 400，提示用户 | `INVALID_CONTEXT` |
| 实例化时 DB 写入失败 | 回滚事务，返回 500 | `INSTANCE_CREATE_FAILED` |

### 7.2 条件评估异常

| 错误场景 | 处理策略 |
|---------|---------|
| auto_eval 表达式语法错误 | 记录错误日志，条件视为不满足 |
| auto_eval 引用了不存在的上下文变量 | 记录警告，条件视为不满足 |
| 死循环检测（条件引用自身） | 抛出异常，阻止实例化 |

### 7.3 边界情况

| 场景 | 处理方式 |
|------|---------|
| 场景模板 fullset.projects 为空 | 允许实例化，创建空 Goal |
| 所有 Projects 都是 conditional 且初始都不满足 | Goal 创建成功但无 Task，等待条件触发 |
| 条件触发时对应的 Project 已在 Goal 中存在 | 跳过，不重复创建 |
| 场景实例化后模板被删除/归档 | 不影响已有实例，继续执行 |

## 附：场景库架构设计（原 21-scenario-library-system-design.md）

> 以下内容从 `docs/02-架构设计/21-scenario-library-system-design.md` 迁移，涵盖场景库的本质、与 Goal 的区别、三层上下文体系、条件分支设计和执行链路。

### 附一、场景的本质

场景（Scenario）是 Nexus 系统中**可复用的标准化解决方案模板**，不是任务列表，不是工作流，而是针对某一类问题的**完整应对框架**。

**核心特征：事前具备定义的能力**——在问题发生之前，就有能力把完整的解法定义出来。

场景回答的问题是：**"遇到这类问题，标准解法是什么？"**

**来源不依赖知识库**：
- 可以是专家凭经验纯人工定义
- 可以是从历史执行记录中提炼
- 可以是从认知库中推导
- 不管哪种来源，产出都是一个完整的场景蓝图

### 附二、场景 ≠ 目标（Goal）

| 维度 | 场景（Scenario） | 目标（Goal） |
|------|-----------------|-------------|
| **性质** | 方法论、框架 | 实际执行实例 |
| **类比** | PMBOK 全书 | 按 PMBOK 启动的一个具体项目 |
| **内容** | 全流程 + 所有分支 + 所有决策点 | 裁剪后的实际执行路径 |
| **状态** | 静态的，可复用的知识资产 | 动态的，有生命周期的执行实例 |
| **关系** | **全集** | **子集**（引用场景全集的一部分） |
| **时间维度** | 事前定义（问题发生之前） | 实例化（裁剪）→ 执行时动态变更 |

### 附三、三段时间维度的关系

| 时间维度 | 说明 | 产出 |
|---------|------|------|
| **事前定义** | 场景模板设计阶段，定义完整能力全集 | 场景蓝图（Scenario Blueprint） |
| **实例化时裁剪** | 根据已知信息，从全集中裁剪出适合本 Goal 的子集 | 场景实例（Scenario Instance） |
| **执行时变更** | 执行过程中，根据新信息动态调整执行路径 | 运行时变更（Runtime Adaptation） |

### 附四、条件分支设计

场景支持多分支并行执行，通过条件（Condition）控制分支激活：

```
条件节点：IF <condition> THEN <branch>
                        ELSE <branch>
```

**条件类型**：
- `input_required`：需要人类输入
- `capability_match`：Agent 能力匹配
- `context_match`：上下文匹配
- `time_based`：时间条件

**条件评估时机**：
- 场景实例化时（静态裁剪）
- Project 执行前（动态激活）
- 任务完成后（状态触发）

### 附五、场景实例化流程

```
Goal 创建
    │
    ▼
用户选择场景模板（Scenario）
    │
    ▼
Nexus 根据 Goal 的行业/领域/已知条件裁剪场景
    │
    ▼
生成场景实例（Scenario Instance）
    │
    ▼
驾驭中心读取 Projects/Tasks 进行调度
    │
    ▼
执行过程中条件触发，动态激活分支
```

### 附六、场景库的系统定位

场景库在 Nexus 全域中的位置：

| 组件 | 与场景库的关系 |
|------|--------------|
| **驾驭中心** | 场景实例化后，驾驭中心调度 Projects/Tasks |
| **Reins** | 负责 Goal 的结构分解，输出 Projects/Tasks |
| **Grasp** | 提供领域知识，支持场景裁剪和条件匹配 |
| **Vigil** | 审核场景模板的安全性、合规性 |
| **Evo** | 从历史执行中学习，优化场景模板 |
| **Reach** | 对外暴露场景库 API，支持第三方场景导入 |
