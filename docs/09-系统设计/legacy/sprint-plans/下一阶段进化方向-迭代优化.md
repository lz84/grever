# 下一阶段进化方向：从任务编排到迭代优化

> 日期：2026-05-09  
> 来源：克里米亚大桥抢修场景分析  
> 基线：`v1.0-waterfall`（Sprint 66 完成，瀑布式流程闭环）  
> 定位：Nexus 下一阶段的进化路线图

---

## 一、核心定位重申

**Nexus 不干活，只驾驭。**

- 不替代智能体做仿真、算路径、管资源
- 不负责底层领域算法和工程计算
- **核心价值**：统一纳管智能体、拆解任务、编排协同、方案比较、迭代优化、全局决策

---

## 二、探索模式设计（新增）

### 2.1 概念定义

**探索模式**（Optimization Mode）：Nexus 不再只是"派发→执行→完成"，而是"派发→执行→记方案→比较→判断→调整→再派"的循环过程，通过迭代逼近最优解。

### 2.2 层级关系

| 层级 | 是否有探索模式 | 职责 |
|------|--------------|------|
| **目标** | ✅ 有 | 定调：这个事要不要试出最优解 |
| **工程** | ✅ 有（继承自目标，可覆盖） | 决定：我底下的任务要不要迭代循环 |
| **任务** | ❌ 没有 | 干活：执行 → 交结果 → 完事 |

### 2.3 探索模式的触发

创建目标时用户选择：

- **常规模式**（默认）：做完就行，不迭代
- **探索模式**：持续迭代到收敛，需要定义：
  - 优化目标：最短工期 / 最低成本 / 综合最优
  - 收敛阈值：改进幅度 < 5% 时收敛
  - 最大轮次：默认 10 轮

### 2.4 工程继承逻辑

工程创建时自动继承所属目标的 mode，但可以单独覆盖：

```
目标："大桥抢修" → 探索模式
    ├── 工程一：战损勘察 → 常规模式（手动覆盖，扫一遍就行）
    ├── 工程二：方案仿真推演 → 探索模式（继承，要试出最优）⭐
    ├── 工程三：资源调度 → 常规模式（手动覆盖，算一次就行）
    └── 工程四：施工管控 → 常规模式（手动覆盖，按方案执行）
```

### 2.5 方案是目标级

**方案不是工程级，也不是任务级，是目标级。**

一个方案是目标下所有工程+任务在一次完整迭代中的结果快照：

```
目标："恢复大桥通行"（探索模式）
├── 方案 #1：临时钢桁架方案
│   ├── 工程一结果：扫描完成
│   ├── 工程二结果：工期3天，安全0.7 → 不达标
│   └── 工程三结果：构件调度方案
├── 方案 #2：局部换梁方案
│   ├── 工程一结果：补充扫描
│   ├── 工程二结果：工期7天，安全1.5 → 达标 ✅
│   └── 工程三结果：调整后的构件调度
└── 方案 #3：全段重建方案
    ├── 工程一结果：全面评估
    ├── 工程二结果：工期21天 → 战时不可用
    └── 工程三结果：大规模调度方案
```

同一工程在不同方案下会被重新执行、结果不同。方案记录"这一轮尝试，整个目标下的所有工程任务跑出了什么结果"。

## 三、人类干预机制（新增）

### 3.1 现有触发机制（4 种）

| 触发方式 | 状态 | 谁发起 | 需要什么 |
|---------|------|--------|---------|
| Agent 主动请求 | `waiting_human` | 智能体 | 人类提供数据/确认 |
| 验证失败重试 | `review_needed` | Verifier | 自动重派，不需要人 |
| 连续失败升级 | `disputed` | Verifier（3次失败） | **人类裁决** |
| 用户主动评论 | `review_needed` | 人类 | 用户觉得需要改 |

### 3.2 现有机制的缺陷

探索模式下，人类干预更频繁，但现有机制缺少：

**"阶段性强制人类输入"的节点**。

大桥场景里，某些关键节点必须人类拍板才能继续：
- "方案一不达标，是否继续试方案二？"
- "3 个方案都跑完了，人类选哪个？"
- "约束调整后，人类确认是否合理？"

现在系统里没有这种"必须人类点头才能往下走"的机制。

### 3.3 需要新增：强制人类确认节点

在探索模式下，关键轮次自动暂停，等人类确认：

```
第 1 轮完成 → 方案A 不达标
    ↓
Nexus 自动暂停，状态 → waiting_human
    ↓
人类介入：
    ├── "继续" → Nexus 调整约束，派发第 2 轮
    ├── "换方向" → Nexus 重新定义优化目标
    └── "停止" → 宣布收敛
```

**触发时机**：
- 每一轮完成后（可选：仅前 3 轮）
- 收敛判断即将触发时（让人类决定是否接受）
- 约束调整幅度超过阈值时

## 四、页面设计

### 4.1 页面清单

| # | 页面 | 类型 | 路由 | 说明 |
|---|------|------|------|------|
| 1 | **ProjectDetail**（改造） | 现有页面 | `/projects/{id}` | 新增迭代控制区域 + 约束历史 Tab |
| 2 | **SolutionList**（新增） | 新页面 | `/goals/{id}/solutions` | 目标级方案列表页 |
| 3 | **SolutionCenter**（新增） | 新页面 | `/solutions` | 方案对比页 |
| 4 | **SolutionDetail Modal**（新增） | 弹窗组件 | — | 方案详情弹窗（多处复用） |

### 4.2 业务流程（3 条主线）

#### 主线 1：方案记录流程

```
用户在探索模式下创建目标 → 定义优化目标 + 收敛标准
    ↓
LLM 分解目标 → 生成工程（继承探索模式） → 生成任务
    ↓
任务派发 → 智能体执行 → 任务 done
    ↓
后端检测：目标探索模式 → 自动提取方案参数 → 写入方案
    ↓
用户进入 SolutionList → 看到方案列表
    ↓
点击方案名 → 弹出 SolutionDetail Modal → 查看完整参数
```

#### 主线 2：迭代决策流程

```
一轮任务全部完成 → 方案入库
    ↓
Nexus 自动触发比较引擎 → 当前方案 vs 历史方案
    ↓
ProjectDetail 顶部更新迭代状态："第3轮/最大10轮，进行中"
    ↓
Nexus 判断收敛条件：
    ├─ 改进 < 5% → 准备宣布收敛 → **强制人类确认**
    ├─ 达到最大轮次 → 暂停 → 等待人类决策
    └─ 仍有优化空间 → 保持"进行中" → 自动继续
    ↓
人类看到比较结果 → 点击"继续迭代" 或 "宣布收敛"
    ↓
继续迭代 → Nexus 自动调整约束 → 重新派发任务 → 下一轮
宣布收敛 → 标记当前方案为最优 → 可沉淀为场景模板
```

#### 主线 3：方案对比流程

```
用户进入 SolutionCenter 页面（/solutions）
    ↓
选择目标 → 加载该目标下所有方案
    ↓
页面展示：
    ├─ 对比表格（横向：方案A/B/C，纵向：工期/成本/安全/风险）
    └─ 收敛趋势图（折线图：各指标随轮次变化）
    ↓
点击方案 → 弹出 SolutionDetail Modal → 查看关联工程任务
    ↓
点击最优方案 → 一键沉淀为场景模板
```

### 4.3 各页面详细业务逻辑

#### ProjectDetail（改造）

**新增区域：**

| 区域 | 位置 | 内容 | 数据来源 |
|------|------|------|---------|
| 迭代状态条 | 页面顶部 | 当前轮次、状态（进行中/收敛/暂停）、最新比较摘要 | `GET /api/v1/solutions/compare?goal_id=xxx` |
| 迭代控制面板 | 状态条下方 | 优化目标选择、最大轮次输入、改进阈值输入、操作按钮 | `POST /api/v1/goals/{id}/iterate` |
| 约束历史 Tab | Tab 栏新增 | 每轮约束条件的时间线 | `GET /api/v1/goals/{id}/constraints` |

**交互逻辑：**
- 任务 done → 自动刷新迭代状态条
- 点击"继续迭代" → 调用 iterate API → 后端调整约束 + 重新派发 → 状态更新
- 点击"宣布收敛" → 标记当前方案最优 → 状态变为 converged
- 点击方案名 → 弹出 SolutionDetail Modal

#### SolutionList（新页面，目标级）

**页面结构：**

```
/goals/{id}/solutions
├── 标题：目标名 + "方案列表"
├── 操作栏：[手动创建方案] [返回目标详情]
└── 表格
    ├── 轮次 (#1, #2, #3...)
    ├── 方案名
    ├── 状态标签（达标/不达标/最优/否决）
    ├── 核心参数（前3个维度缩写显示）
    ├── 评分（0-100）
    ├── 创建时间
    └── 操作（[查看详情] [标记最优] [否决] [删除]）
```

**业务逻辑：**
- 数据来源：`GET /api/v1/solutions?goal_id=xxx`
- 方案自动捕获：探索模式下任务完成后后端自动创建
- 方案包含目标下所有工程的执行结果快照

#### SolutionCenter（新页面）

**页面结构：**

```
/solutions
├── 筛选栏：[目标选择器 ▼] [显示维度复选框]
├── 对比表格区
│   ├── 表头：维度 \ 方案A / 方案B / 方案C / ...
│   ├── 行：工期、安全系数、成本、风险评分...
│   ├── 最优值：绿色背景
│   ├── 最优方案列：金色边框
│   └── 底部：[沉淀为场景] 按钮（点击最优方案）
├── 趋势图区
│   ├── 折线图（横轴：轮次，纵轴：指标值）
│   └── 可切换的指标复选框
└── 方案详情抽屉（点击任意方案）
    ├── 完整参数
    ├── 关联工程任务列表
    ├── 约束条件
    └── 操作：[标记最优] [否决] [删除]
```

**业务逻辑：**
- 数据来源：`GET /api/v1/solutions/compare/multi?goal_id=xxx`（表格）+ `GET /api/v1/solutions/trend?goal_id=xxx`（图表）
- 对比逻辑：后端返回多维度数据，前端自动计算最优值并高亮
- 趋势逻辑：后端按轮次聚合各维度数值，前端用折线图渲染
- 沉淀逻辑：点击"沉淀为场景" → 调用现有场景创建 API → 将最优方案的参数作为场景模板

## 五、数据模型（新增）

### 5.1 solutions 表（方案库）

```sql
CREATE TABLE solutions (
    id TEXT PRIMARY KEY,                    -- UUID
    goal_id TEXT REFERENCES goals(id),      -- 方案属于目标（目标级）
    round INTEGER DEFAULT 1,                -- 迭代轮次
    name TEXT,                              -- 方案名称
    status TEXT,                            -- optimal / compliant / non_compliant / rejected
    parameters TEXT,                        -- JSON: {"工期": "7天", "安全系数": 1.5, ...}
    dimensions TEXT,                        -- JSON: ["工期", "安全系数", "成本", "风险"]
    score REAL,                             -- 综合评分（0-100）
    is_optimal BOOLEAN DEFAULT 0,           -- 是否最优方案
    project_ids TEXT,                       -- JSON: [本轮涉及的工程ID]
    task_ids TEXT,                          -- JSON: [本轮涉及的任务ID]
    constraints TEXT,                       -- JSON: 本轮约束条件
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);

CREATE INDEX idx_solutions_goal ON solutions(goal_id);
CREATE INDEX idx_solutions_round ON solutions(goal_id, round);
```

### 5.2 iteration_constraints 表（约束历史）

```sql
CREATE TABLE iteration_constraints (
    id TEXT PRIMARY KEY,
    goal_id TEXT REFERENCES goals(id),
    round INTEGER,
    constraints TEXT,                       -- JSON: {"工期": {"max": "6天", "reason": "..."}}
    reason TEXT,                            -- 调整原因
    created_by TEXT,                        -- 'system' / 'human'
    created_at TIMESTAMP
);

CREATE INDEX idx_constraints_goal ON iteration_constraints(goal_id);
```

### 5.3 goals/projects 表新增字段

```sql
-- goals 表新增
ALTER TABLE goals ADD COLUMN mode TEXT DEFAULT 'normal';
ALTER TABLE goals ADD COLUMN optimization_target TEXT;          -- 最短工期 / 最低成本 / 综合最优
ALTER TABLE goals ADD COLUMN convergence_threshold REAL DEFAULT 0.05;  -- 改进幅度阈值
ALTER TABLE goals ADD COLUMN max_rounds INTEGER DEFAULT 10;

-- projects 表新增
ALTER TABLE projects ADD COLUMN mode TEXT DEFAULT 'normal';     -- 继承自 goal，可覆盖
```

```
┌──────────────────────────────────────────────────────────────┐
│                     目标创建阶段                               │

└──────────────────────┬───────────────────────────────────────┘
                       ↓
┌──────────────────────────────────────────────────────────────┐
│                     任务分解阶段                               │
│  LLM 分解 → 生成工程（继承探索模式） → 生成任务                │
│  工程可以单独切换为常规模式                                    │
└──────────────────────┬───────────────────────────────────────┘
                       ↓

│                     第 1 轮执行                                │

└──────────────────────┬───────────────────────────────────────┘
                       ↓

│                     收敛判断阶段                               │
```

## 七、与现有能力的关系

| 新能力 | 依赖现有 | 增强现有 |
|--------|---------|---------|
| 方案记忆 | 任务状态机 | 扩展结果记录维度（不仅 done/failed，还要方案参数） |
| 迭代决策回路 | 任务派发 | 增加"派→收→比→再派"的循环逻辑 |
| 方案对比视图 | 认知库 | 在经验基础上增加方案对比维度 |
| 强制人类确认 | 人工仲裁（disputed） | 从"被动等争议"到"主动暂停等确认" |

---

## 八、不做的事

以下能力明确**不在**Nexus 范畴内，由外部智能体自行实现：

- ❌ 仿真推演算法（工程仿真内核）
- ❌ 路径规划算法（运筹优化求解器）
- ❌ 物资/装备库存管理
- ❌ 施工排班调度
- ❌ 结构安全计算

Nexus 只负责：**定义目标 → 拆解任务 → 派发执行 → 收回结果 → 记忆方案 → 比较迭代 → 强制人类确认 → 决策最优**
