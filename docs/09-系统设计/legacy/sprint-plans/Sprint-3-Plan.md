# Sprint 3 计划：多智能体协同演示

## 目标

**核心目标**: 实现一个端到端可演示的多智能体协同工作流，向客户展示 Nexus 平台的智能体编排、任务分解、实时协同和可视化能力。

**核心原则**: 
- 演示 = 真实产品功能的一个用例，不是专门做的 demo 页面
- 先有功能界面，再让功能"正好"覆盖救灾场景
- 客户看到的是真实产品，救灾只是恰好跑通的一个场景

**交付物**: 
- Grasp 功能界面（预案管理、知识检索、认知评估）
- Reins 功能界面（Agent 管理、目标管理、任务编排、执行监控）
- Workflow 工作流引擎
- 抢险救灾场景完整演示链路

## 流程二：目标创建与执行（前台操作，演示时做）

```
1. 描述目标
   └─ 用户输入：
      - 目标类型：地震救援 / 洪水救灾 / 战场补给
      - 目标级别：Ⅰ级 / Ⅱ级 / Ⅲ级响应
      - 灾情描述：7.2级地震，震中位于X区，影响3个乡镇
      - 其他备注：X区人口密集，有化工厂
    ↓
2. 目标解析
   └─ 系统自动识别：
      - 灾情类型：地震
      - 严重程度：7.2级 → Ⅰ级响应
      - 关键风险：人口密集区 + 化工厂
    ↓
3. 预案匹配
   └─ Grasp 检索推荐：
      - 预案A：城市地震搜救（匹配度 95%）
      - 预案B：危化品泄漏处置（匹配度 82%）
      - 预案C：医疗救护（匹配度 78%）
    ↓
4. 综合研判
   └─ Grasp 生成初步方案：
      - 合并 A+B 的核心步骤
      - 识别 4 个救援阶段、15 个任务
      - 标记资源缺口和风险提示
    ↓
5. ★ 人工修正（对话式即时修正）★
   └─ 用户通过对话框与系统交互：
      "把化工厂处置调到最高优先级"
      → 系统：已调整，化工厂泄漏处置升为 P0
      "C区已经疏散了，去掉C区搜救"
      → 系统：已删除 C 区搜救任务（3 个子任务）
      "增加一个环境监测任务"
      → 系统：已添加环境监测任务，分配给物资 Agent
    ↓
6. 激活执行
   └─ 一键激活 → 自动生成任务 DAG → Reins 开始执行
    ↓
7. 执行监控
   └─ 实时状态 + 冲突检测 + 动态调整
    ↓
8. 执行报告
   └─ 完成评估 + 反馈到 Grasp（闭环优化预案）
```

## Demo 场景：抢险救灾多智能体协作

### 场景描述

某地发生 7.2 级地震，启动应急响应。Nexus 平台调度多智能体团队完成救援任务。

### Agent 角色

| Agent | 职责 | 能力标签 |
|-------|------|---------|
| 搜救 Agent | 发现被困人员、标记位置 | 生命探测、废墟评估 |
| 医疗 Agent | 伤情评估、医疗物资调配 | 伤情分诊、医疗资源管理 |
| 物资 Agent | 物资库存管理、调配运输 | 库存优化、物流调度 |
| 指挥 Agent | 全局协调、冲突仲裁 | 资源优化、优先级排序 |

### 任务分解结构

```
抢险救灾（Goal）
├── 震区评估（Project）
│   ├── 卫星图像分析（Task）→ 搜救 Agent
│   ├── 地面报告收集（Task）→ 搜救 Agent
│   └── 伤亡预估（Task）→ 医疗 Agent
│
├── 人员搜救（Project）
│   ├── A 区搜救（Task）
│   │   ├── 生命探测（Subtask）→ 搜救 Agent
│   │   └── 废墟清理（Subtask）→ 搜救 Agent
│   └── B 区搜救（Task）
│       ├── 生命探测（Subtask）
│       └── 人员转移（Subtask）
│
├── 医疗救护（Project）
│   ├── 野战医院建立（Task）→ 医疗 Agent
│   ├── 伤员后送（Task）→ 指挥 Agent
│   └── 医疗物资补给（Task）→ 物资 Agent
│
└── 物资调配（Project）
    ├── 饮用水配送（Task）→ 物资 Agent
    ├── 食品配送（Task）→ 物资 Agent
    └── 帐篷搭建（Task）→ 物资 Agent
```

### 协同冲突场景（展示核心价值——参考军事领域流程）

救灾场景的冲突解决机制与军事后勤同源，以下三类冲突在救灾中更直观、更有情感冲击力：

**场景 1：资源竞争（军事映射：A/B 阵地争弹药 → A/B 区争搜救）**
- A 区和 B 区同时请求搜救，但搜救 Agent 只有一个
- 指挥 Agent 根据伤亡严重程度（A 区 3 人 / B 区 12 人）重新分配优先级
- 展示效果：冲突告警 → 指挥 Agent 仲裁 → 优先级调整 → 任务重新分配

**场景 2：依赖阻塞（军事映射：油料车被征用 → 唯一运输车被调去送急救药品）**
- 医疗物资配送依赖运输车辆，但唯一运输车被临时调去送急救药品
- 物资任务自动阻塞 → 系统寻找替代方案（调用备用车辆 / 调整配送顺序）
- 展示效果：依赖关系图高亮阻塞链路 → 自动推荐替代方案 → 人工确认或自动执行

**场景 3：动态响应（军事映射：前线新增伤员 → 发现新被困人员集中点）**
- 执行中突发：C 区发现大量被困人员（原计划中不存在）
- 系统自动生成 C 区搜救任务 → 全局资源重新调度 → 低优先级任务暂停
- 展示效果：新任务动态插入 → 全局调度动画 → 受影响的已有任务状态变化

### 冲突解决流程（统一机制）

```
冲突检测 → 冲突上报 → 指挥 Agent 仲裁 → 资源重分配 → 状态同步 → 冲突消除
    ↑                                                        │
    └──────────────── 持续监控 ←────────────────────────────────┘
```

1. **检测**：系统持续监控资源分配，发现同一资源被多个任务请求
2. **上报**：冲突事件推送到 UI（红色告警），记录到 ExecutionTracker
3. **仲裁**：指挥 Agent 根据优先级（伤亡人数、紧急程度、依赖关系）做出决策
4. **重分配**：更新任务状态，通知相关 Agent
5. **同步**：所有 Agent 收到最新状态，UI 实时更新
6. **消除**：冲突标记清除，记录仲裁日志
