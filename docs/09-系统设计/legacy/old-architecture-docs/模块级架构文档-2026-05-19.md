# Nexus 系统模块级架构文档

> 更新：2026-05-19
> 范围：驾驭模块（Goal/Project/Task/Scenario/Workflow）+ 智能体模块

---

## 一、系统定位

Nexus 是一套**多智能体任务调度与验证平台**。

核心能力：
1. 将战略目标分解为可执行的任务
2. 将任务派发给合适的智能体执行
3. 验证执行结果，形成闭环
4. 支持探索模式（发散）和优化模式（收敛）

## 三、驾驭模块（任务执行体系）

### 3.1 Goals（战略目标）

**职责**：Goal 是最高层的执行单元，代表一个完整的业务目标。

**生命周期**：
```
draft → active → in_progress → completed
                  ↓
              paused / disputed
```

**三种运行模式**：
- `normal`：常规模式，按依赖顺序执行
- `exploration`：探索模式，发散执行，允许多路径并行
- `optimization`：优化模式，收敛执行，自动淘汰低效方案

**核心字段**：
- `mode`：运行模式
- `progress`：完成进度（0-100）
- `matched_scenario_id`：关联的场景模板
- `workspace_path`：代码仓地址（用于代码执行类 Goal）
- `run_status`：idle / waiting_next_round / running / converged

---

### 3.2 Projects（子项目）

**职责**：Goal 的下一级组织单元，一个 Goal 可以包含多个 Project。

**用途**：将 Goal 拆分为大的阶段性成果，每个 Project 下管理一批相关的 Task。

**当前状态**：Project 在当前代码中有实体定义，但与 Goal/Task 的边界较模糊，是历史遗留的中间层概念。

---

### 3.3 Tasks（任务）

**职责**：Task 是 Nexus 的**唯一执行单元**。所有智能体的执行对象都是 Task。

**生命周期**：
```
todo → in_progress → done
                  ↘ failed / timeout / paused / disputed
```

**关键设计**：
- `assigned_agent`：派发给哪个智能体
- `depends_on`：依赖关系（DAG 边的来源）
- `next_step`：DAG 渲染用的后续节点（与 depends_on 冗余共存）
- `acceptance_criteria`：验收标准（JSON 格式）
- `result_summary`：执行结果摘要
- `verification_status`：pending / reviewing / approved / rejected

**派发流程**：
1. Scheduler 扫描 `todo` 状态且无未完成依赖的 Task
2. 匹配最优智能体（基于 capabilities 和当前负载）
3. 派发给智能体执行
4. 智能体完成后提交结果
5. 验证流程判定通过/失败

---

### 3.4 Scenarios（场景模板）

**职责**：Scenarios 是**可复用的任务模板库**。从场景市场获取，定义了特定类型问题的标准解决流程。

**与 Goal 的关系**：
- 创建 Goal 时可指定 `matched_scenario_id`
- Goal 会继承 Scenario 的任务结构
- 场景市场（Scenario Marketplace）提供模板的浏览和订阅

**核心字段**：
- `capability_tags`：该场景需要的智能体能力标签
- `task_templates`：预定义的任务模板

---

### 3.5 Workflows（工作流）

**职责**：Workflow 定义了 Goal 的**执行策略**——包括用什么智能体、什么顺序、什么验证方式。

**当前状态**：Workflow 有数据实体，但与 Goal 的 `mode` 字段存在功能重叠（都是描述执行策略），系统中有重复实现。

---

## 四、智能体模块

### 4.1 Agent（智能体）

**职责**：Agent 是实际执行任务的工作节点。Nexus 支持多智能体并行。

**注册机制**：
- 智能体通过 OpenClaw 注册到 Nexus
- 注册时提供：名称、地址、能力标签、触发模式、最大并发数
- 智能体通过心跳维持在线状态

**当前系统中的 5 个智能体**：

| 代号 | 名称 | 角色 | 模型 |
|------|------|------|------|
| 扣子 | 扣子 | 开发专员 | qwen3-30b |
| 蚊子 | 蚊子 | 内容专员 | minimax |
| 麻子 | 麻子 | 技术专家 | minimax |
| 谷子 | 谷子 | 股票交易 | minimax |
| 刚子 | 刚子 | 指挥调度 | minimax |

**能力标签体系**：
- 四维标签：business / professional / technical / management
- 用于 Task 派发时的最优匹配

---

### 4.2 负载管理（Load Management）

**职责**：跟踪每个智能体的当前工作负载，避免过载。

**负载计算公式**：
```
load = min(100, pending_tasks / max_concurrent_tasks * 100)
```

**动态计算**：负载不再依赖智能体心跳上报的缓存值，而是实时查询 tasks 表统计该智能体当前 `todo/in_progress/pending/paused` 状态的任务数。

**这是本次重构的核心改进**：统一了列表页和详情页的负载显示，全部使用动态计算结果。

---

### 4.3 智能体调度（Scheduler）

**职责**：Scheduler 是调度循环，每 tick 检查待执行任务并派发。

**调度策略**：
1. 扫描 `todo` 状态 Task
2. 过滤掉有未完成依赖的 Task
3. 按优先级和负载选择最优智能体
4. 派发并更新 Task 状态为 `in_progress`

**超时恢复**：若 Task 超过 `max_duration` 未完成，Scheduler 自动标记为 `timeout` 并尝试重新派发。

---

## 五、验证体系

### 5.1 Verification（验证流程）

**职责**：Task 完成后必须经过验证才能标记为 `done`。

**验证模式**：
- `automatic`：自动验证（基于 acceptance_criteria 的脚本检查）
- `human_review`：人工审核（需要人类介入判断）
- `agent_verification`：智能体验证（派发给验证智能体）

**争议处理**：
- 若验证失败，Task 进入 `disputed` 状态
- 进入 Dispute 仲裁流程
- 仲裁结果决定最终状态

---

### 5.2 Human Review（人工审核）

**职责**：当智能体无法自动判定结果时，提交给人类决策。

**流程**：
1. Task 完成后请求人工审核
2. 人类查看结果并做出决定（approve / reject / request_retry）
3. 根据人类决策流转

---

## 六、前端模块

### 6.1 Dashboard（工作台）

**职责**：系统概览页，展示全局状态。

**显示内容**：
- 智能体在线状态和负载
- 进行中的 Goal 进度
- 任务统计（总数/进行中/完成/失败）

### 6.2 Goals 页面

**职责**：Goal 的增删改查和状态管理。

**核心功能**：
- 创建 Goal（可指定 Scenario 模板）
- Goal 详情页（显示分解出的 Task 列表）
- Goal 模式切换（normal / exploration / optimization）
- 分解预览（GoalDecomposePreview）
- 迭代控制（IterationControlPanel）

### 6.3 Agent Management（智能体管理）

**职责**：管理所有注册的智能体。

**核心功能**：
- 智能体列表（负载、状态、当前任务数）
- 智能体详情（能力标签、模型、注册信息）
- 设置智能体状态（online / offline / busy）
- 重新注册、强制下线

### 6.4 Scenarios 页面

**职责**：浏览和管理场景模板。

**核心功能**：
- 场景市场浏览
- 订阅/取消订阅场景
- 从场景创建 Goal

### 6.5 Workflows 页面

**职责**：管理工作流定义。

**核心功能**：
- 查看工作流 DAG 拓扑
- 工作流执行历史

### 6.6 Settings（设置）

**职责**：系统级配置。

**内容**：
- 调度器参数
- 验证策略配置
- 能力标签体系管理

## 七、数据模型关系

```
Goal (战略目标)
  ├── 1:N → Project (子项目)
  │           └── 1:N → Task (任务)
  │                        ├── assigned_agent → Agent
  │                        ├── depends_on → Task (DAG边)
  │                        ├── acceptance_criteria (验收标准)
  │                        └── verification_status
  │
  ├── 1:1 → Scenario (场景模板，可选)
  │           └── capability_tags (需要的能力)
  │
  └── 1:1 → Workflow (工作流，可选)

Agent (智能体)
  ├── 1:N → Task (分配给该智能体的任务)
  └── capability_tags (四维能力标签)

Task (任务)
  └── 1:N → HumanReviewRequest (人工审核请求)

Scenario (场景模板)
  └── 1:N → TaskTemplate (任务模板)
```

# Nexus 业务架构文档

## 一句话定位

**你给目标，Nexus 拆、派、跑、验——一个多智能体协同执行平台。**

## 全景图

```
                    驾驭体系（大脑）
          ┌──────────────────────────────────┐
          │  目标管理  →  任务编排            │
          │       ↘           ↓               │
          │     场景实例化  →  场景定义        │

          │       ↘           ↓               │
          │      任务验证  →  人工审核          │
          └────────────────┬─────────────────┘
                           │ 派发任务/收结果
                    ┌──────▼──────┐
                    │  智能体体系  │
                    │  （手脚）    │
                    └─────────────┘
```

两大板块：**驾驭体系**（怎么跑） + **智能体体系**（谁去跑）。

## 一、智能体体系（谁去跑）

### 1. 智能体注册

**问题**：Nexus 得知道自己有哪些"手"可用。

**能力**：
- 智能体上线后向 Nexus 报到——报身份、报能力、报最大并发能力
- 注册后进入"在册员工名单"，可以被派活
- 注销后不再接收新任务

**上下游**：
- 上游：智能体主动报到
- 下游：所有模块（基础数据源）

### 2. 智能体接入

**问题**：活怎么派下去？结果怎么收回来？

**能力**：
- **推送模式**：Nexus 主动推任务给智能体，实时通知（SSE）
- **拉取模式**：智能体定时问 Nexus 有没有新活（Polling）
- 任务派发 → 智能体领取 → 执行 → 提交结果，全流程状态可追踪

**上下游**：
- 上游：智能体驱动派发
- 下游：智能体执行、心跳上报

### 3. 心跳监控

**问题**：智能体活着还是挂了？挂了得知道，不能派活给死人。

**能力**：
- 在线智能体定时向 Nexus 报平安
- 超时未心跳 → 自动标记离线
- 离线时自动回收它手上的活，重新派给别人
- 连续多次没心跳才真正标记下线（防网络抖动误判）

**上下游**：
- 上游：智能体定时上报
- 下游：能力匹配（只选活着的）、智能体驱动（离线回收任务）

### 4. 负载管理

**问题**：每个智能体有上限，不能塞太多活，也不能让它闲着。

**能力**：
- 实时统计每个智能体手上有多少待处理任务
- 计算负载百分比 = 待处理任务数 / 最大并发数 × 100%
- 动态实时计算（不依赖缓存），直接从任务表统计
- 负载高 → 不派新活；负载低 → 优先派给它

**上下游**：
- 上游：任务表数据、智能体注册信息（最大并发数）
- 下游：能力匹配（选最闲的）、前端（负载看板展示）

### 5. 状态管理

**问题**：智能体可以人为干预——强制下线、重启、设为忙碌。

**能力**：
- 查看/设置智能体状态（在线/离线/忙碌/空闲）
- 强制下线：回收所有在手上任务
- 重启：清空状态重新注册
- 清理僵尸任务：处理卡住不推进的任务

**上下游**：
- 上游：管理员手动操作、心跳超时自动触发
- 下游：调度引擎（离线智能体不参与调度）

## 二、驾驭体系（怎么跑）

### 6. 目标管理

**问题**：用户要达成什么业务目的？Nexus 先知道方向。

**能力**：
- 创建目标（Goal），描述要达成的业务目的
- 设定优先级、执行模式
- 跟踪目标进度：完成了百分之多少
- 状态流转：草稿 → 激活 → 执行中 → 完成/失败

**三种执行模式**：

| 模式 | 业务含义 | 典型场景 |
|------|---------|---------|
| 常规模式 | 按部就班执行 | 任务明确，直接派活 |
| 探索模式 | 多路径发散尝试 | 方案对比，找最优解 |
| 优化模式 | 多轮迭代收敛 | 持续改进，逐步逼近最优 |

**上下游**：
- 上游：用户创建、场景实例化
- 下游：任务编排（拆任务）、任务验证（回灌进度）

### 7. 任务编排

**问题**：目标太大怎么办？拆成小任务，排好顺序和依赖。

**能力**：
- 目标分解：将一个 Goal 拆解为多个可执行 Task
- 依赖管理：任务 A 等任务 B 做完才能开始（DAG 调度）
- 自动分解：LLM 自动拆解目标
- 进度汇总：Task 完成情况自动回灌到 Goal 进度条

**上下游**：
- 上游：目标管理（提供目标）
- 下游：智能体驱动（领取可执行任务）、能力匹配（匹配所需能力）

### 8. 场景定义

**问题**：同类问题反复出现，每次从头定义目标太麻烦。

**能力**：
- 定义场景模板：比如"火灾应急救援"、"代码重构"等标准解法
- 场景包含：目标描述模板、任务结构、需要的能力类型、执行策略
- 场景市场：浏览、订阅、管理所有可用场景
- 场景打能力标签：这个场景需要哪些能力的智能体

**上下游**：
- 上游：管理员创建/编辑
- 下游：场景实例化（读取模板内容）

### 9. 场景实例化

**问题**：从模板到实战——把场景模板变成真实可执行的目标。

**能力**：
- 选场景 → 自动创建对应 Goal
- Goal 自动继承场景的任务结构和执行策略
- 关联场景 ID，后续可追溯
- 执行完成后回灌反馈，优化场景模板（越用越准）

**上下游**：
- 上游：用户选择场景
- 下游：目标管理（创建真实 Goal）、场景定义（读模板）、场景定义（回灌反馈）

### 10. 能力匹配系统

**问题**：有 5 个智能体都能干活，这个任务该派给谁？

**能力**：
- 任务声明需要的能力标签（业务/专业/技术/管理 四维标签）
- 智能体注册时声明自己的能力标签
- 匹配算法：任务需求 vs 智能体能力 → 算匹配度
- 权重衰减模型：智能体在某领域成功越多，该领域权重越高
- 综合选优：匹配度 + 当前负载 + 历史成功率

**上下游**：
- 上游：智能体注册信息、任务编排（任务能力需求）
- 下游：智能体驱动（派给最合适的）、任务验证（选验证智能体）

### 11. 智能体驱动

**问题**：任务拆好了，怎么让智能体真正跑起来？

**能力**：
- 扫描所有待执行任务（todo 状态、依赖已满足）
- 按优先级排序，决定执行顺序
- 派发任务到最优智能体
- 监控执行：超时回收、失败重试
- 定时 tick 循环，持续驱动任务流转
- DAG 驱动：按依赖关系解锁后续任务

**上下游**：
- 上游：任务编排（提供可执行任务）
- 下游：智能体接入（派发任务）、任务验证（执行完触发验证）、能力匹配、负载管理

### 12. 任务验证

**问题**：智能体说"做完了"，真的做完了吗？质量合格吗？

**能力**：
- 任务完成后自动触发验证
- 三种验证方式：
  - **自动验证**：基于验收标准脚本检查（编译通过？测试通过？）
  - **智能体验证**：派给验证智能体做主观判断（代码质量？架构合理性？）
  - **人工审核**：人类亲自确认
- 验证通过 → 任务标记 done
- 验证失败 → 进入争议仲裁或打回重做

**上下游**：
- 上游：智能体驱动（执行完成触发）
- 下游：目标管理（更新进度）、人工审核（需要人介入时）、能力匹配（选验证智能体）

### 13. 人工审核

**问题**：有些判断机器做不了，必须人说了算。

**能力**：
- 验证不通过时提交人工审核
- 任务超时未响应，自动提醒人类
- 人类决策：批准 / 拒绝 / 要求重试
- 超时自动处理：超过指定时间无人处理，按预设策略自动执行

**上下游**：
- 上游：任务验证（提交审核请求）
- 下游：智能体驱动（审核通过后流转状态）

## 三、模块调用关系

### 核心调用链路

```
用户创建目标（目标管理）
  → 任务编排（拆任务，建依赖）
  → 调度引擎循环：
      ├─ 能力匹配（找最合适的智能体）
      │    ├─ 心跳监控（还活着吗？）
      │    └─ 负载管理（有空吗？）
      ├─ 智能体驱动（派发任务）
      │    └─ 智能体接入（推给智能体执行）
      ├─ 任务验证（执行完检查质量）
      │    ├─ 能力匹配（选验证智能体）
      │    └─ 人工审核（机器判断不了时交给人）
      └─ 目标管理（回灌进度）
```

### 支撑关系矩阵

| 被支撑模块 | 依赖哪些模块 | 说明 |
|-----------|-------------|------|
| 智能体驱动 | 任务编排、能力匹配、负载管理、智能体接入 | 核心调度引擎，是中枢 |
| 能力匹配 | 智能体注册、负载管理、心跳监控 | 决策层，给调度引擎出主意 |
| 任务验证 | 能力匹配、人工审核 | 质量把关 |
| 场景实例化 | 场景定义、目标管理 | 模板→实战的桥梁 |
| 任务编排 | 目标管理 | 目标拆解器 |
| 人工审核 | 任务验证 | 人类兜底 |

### 基础支撑层

| 模块 | 支撑谁 | 说明 |
|------|--------|------|
| 智能体注册 | 所有模块 | 基础数据——"有哪些员工" |
| 心跳监控 | 能力匹配、智能体驱动 | "员工还活着吗" |
| 负载管理 | 能力匹配、前端展示 | "员工忙不忙" |
| 状态管理 | 调度引擎、管理员 | "员工什么状态" |
