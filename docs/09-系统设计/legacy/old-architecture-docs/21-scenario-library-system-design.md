## 一、场景库的本质

### 1.1 场景是什么

场景（Scenario）是 Nexus 系统中**可复用的标准化解决方案模板**。它不是任务列表，不是工作流，而是针对某一类问题的**完整应对框架**。

**核心特征：事前具备定义的能力**——在问题发生之前，就有能力把完整的解法定义出来。

**类比**：

| 场景库中的场景 | 现实中的类比 |
|---------------|-------------|
| 化工厂危化品泄漏应急预案 | PMBOK 全书 |
| 城市内涝应急响应流程 | 医院急救指南 |
| 代码重构标准流程 | 烹饪菜谱 |

场景回答的问题是：**"遇到这类问题，标准解法是什么？"**

**来源不依赖知识库**：
- 可以是专家凭经验**纯人工定义**
- 可以是从历史执行记录中**提炼**
- 可以是从认知库中**推导**
- 不管哪种来源，产出都是一个**完整的场景蓝图**

### 1.2 场景 ≠ 目标（Goal）

这是场景库最核心的概念区分：

| 维度 | 场景（Scenario） | 目标（Goal） |
|------|-----------------|-------------|
| **性质** | 方法论、框架 | 实际执行实例 |
| **类比** | PMBOK 全书 | 按 PMBOK 启动的一个具体项目 |
| **内容** | 全流程 + 所有分支 + 所有决策点 | 裁剪后的实际执行路径 |
| **状态** | 静态的，可复用的知识资产 | 动态的，有生命周期的执行实例 |
| **关系** | **全集** | **子集**（引用场景全集的一部分） |
| **时间维度** | 事前定义（问题发生之前） | 实例化（根据已知信息裁剪）→ 执行时动态变更 |

```
三个时间维度的关系：

  ┌─────────────────────────────────────────────────────────────┐
  │ 事前定义                                                    │
  │  场景 = 全集（PMBOK）                                      │
  │  定义所有可能的 Project + Task + 分支 + 条件                │
  │  静态不变，除非有人主动编辑场景定义                          │
  └─────────────────────────────────────────────────────────────┘
                           │ 实例化
                           ▼
  ┌─────────────────────────────────────────────────────────────┐
  │ 实例化时裁剪                                                │
  │  根据当时已知的信息（现场情况/用户输入）                     │
  │  从全集中选出"已知必须做"的部分作为初始 Goal               │
  │  不满足条件的 conditional 项目/Task 暂不创建                  │
  │  → 得到初始 Project + Task 列表（Goal 的子集）              │
  └─────────────────────────────────────────────────────────────┘
                           │ 执行
                           ▼
  ┌─────────────────────────────────────────────────────────────┐
  │ 执行时变更                                                │
  │  执行过程中，新情况出现 → 条件满足 → 引擎从全集中取出      │
  │  相应的 Project/Task 创建到 Goal                           │
  │  执行过程中，情况变化 → 某项目不再需要 → 跳过/暂停         │
  │  → Goal 的 Project + Task 集合随执行动态演变               │
  └─────────────────────────────────────────────────────────────┘

关键：变更的依据是场景全集，变更的动作由场景条件触发，不是随意修改场景。
```

### 1.3 场景 ≠ 工作流（Workflow）

| 维度 | 场景 | 工作流 |
|------|------|--------|
| 抽象层级 | 高（业务视角的完整解法） | 低（步骤顺序的执行描述） |
| 包含内容 | 目标模板 + 工程模板 + 任务模板 + 条件分支 + 能力要求 | 步骤 + 依赖关系 |
| 条件分支 | 完整定义（含人类决策/自动评估） | 基本依赖关系 |
| 能力描述 | 四维能力标签 | Agent 类型 |

**工作流是场景的底层执行引擎，场景是工作流的业务语义包装。**

### 1.4 核心定位

场景库在 Nexus 中的定位：

```
┌─────────────────────────────────────────────────┐
│              Nexus 平台全貌                       │
│                                                   │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐   │
│  │  场景库   │───▶│ 驾驭中心  │───▶│ 任务派发  │   │
│  │ (蓝图库)  │    │ (调度器)  │    │ (执行器)  │   │
│  └──────────┘    └──────────┘    └──────────┘   │
│       ▲              │              │            │
│       │              ▼              ▼            │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐   │
│  │ 标签系统  │◀──│ 能力匹配  │◀──│ 目标管理  │   │
│  │          │    │          │    │          │   │
│  └──────────┘    └──────────┘    └──────────┘   │
│                                                   │

```

## 二、场景的时间维度与生命周期

### 2.1 三段时间维度的关系

场景在时间轴上经历三个阶段，核心是**事前定义 → 实例化裁剪 → 执行时变更**：

```
  ┌─────────────────────────────────────────────────────────────┐
  │ 【阶段一】事前定义                                        │
  │                                                          │
  │  场景 = 全集（PMBOK）                                   │
  │                                                          │
  │  编写场景蓝图（所有可能的 Project + Task + 分支 + 条件）  │
  │  审核 → 测试 → 上线 → 成为可用场景模板                   │
  │                                                          │
  │  产出：场景定义（静态，可复用）                          │
  └─────────────────────────────────────────────────────────────┘
                               │ 实例化
                               ▼
  ┌─────────────────────────────────────────────────────────────┐
  │ 【阶段二】实例化时裁剪                                       │
  │                                                              │
  │  用户遇到问题 → 创建 Goal → 选择场景                        │
  │  根据当时已知的信息（现场情况/已知条件）                   │
  │  从全集中选出"已知必须做"的部分                           │
  │  条件未满足的 conditional 项目暂不创建                      │
  │                                                              │
  │  产出：初始 Goal（Project + Task 列表，Goal 的子集）        │
  └─────────────────────────────────────────────────────────────┘
                               │ 执行
                               ▼
  ┌─────────────────────────────────────────────────────────────┐
  │ 【阶段三】执行时变更                                         │
  │                                                              │
  │  执行过程中，新情况出现 → 条件满足                         │
  │  → 引擎从全集中取出对应 Project/Task → 创建到 Goal        │
  │                                                              │
  │  执行过程中，情况变化 → 某项目不再需要                     │
  │  → 跳过/暂停该 Project                                    │
  │                                                              │
  │  产出：Goal 的 Project + Task 集合随执行动态演变            │
  │                                                              │
  │  ※ 变更的依据是场景全集，变更的动作由场景条件触发           │
  │    不是随意修改场景，而是按场景预设的路径执行                 │
  └─────────────────────────────────────────────────────────────┘
                               │
                               │ 执行完成
                               ▼
  ┌─────────────────────────────────────────────────────────────┐
  │ 【反馈回流】                                                │
  │                                                              │
  │  执行结果（成功率/耗时/冲突）→ 统计回写入场景库           │
  │  高频执行路径 → 建议固化到场景模板                         │
  │  场景本身随反馈持续优化 → 新版本                           │
  └─────────────────────────────────────────────────────────────┘
```

### 2.2 七个核心阶段（时间维度视角）

| 阶段 | 名称 | 处于哪个时间维度 | 做什么 | 产出 |
|------|------|----------------|--------|------|
| 1 | **定义** | 事前 | 编写场景蓝图（所有 Project/Task/条件/能力） | 场景定义（全集） |
| 2 | **发布** | 事前 | 审核、测试、上线 | 可用场景模板 |
| 3 | **匹配** | 事前→实例化 | 根据目标描述推荐场景 | 场景推荐列表 |
| 4 | **实例化** | 实例化时 | 根据已知信息从全集中裁剪初始 Goal | 初始 Project + Task 列表 |
| 5 | **指导执行** | 执行时 | 根据新信息触发条件 → 从全集取 Project/Task | 动态任务增减 |
| 6 | **反馈** | 执行完成 | 执行结果回流到场景库 | 统计数据 |
| 7 | **优化** | 事后 | 根据反馈优化场景定义 | 新版本场景 |

## 三、场景的来源：事前定义的三种方式

### 3.1 核心观点：事前定义不依赖知识库

场景的"事前定义"是指**有能力在问题发生之前就把完整解法定义出来**。这不要求必须有知识库——全凭专家经验人工定义同样可以。

```
场景来源的本质：

  场景的来源 = 事前定义的能力
                │
                ├── 专家人工定义（全凭经验，不需要知识库）
                ├── 知识库推导（从历史沉淀中提炼）
                └── 模板复制改造（从已有场景派生）
```

### 3.2 事前定义的三种方式

#### 方式一：纯人工定义（不依赖知识库）

最直接的方式：专家凭经验直接编写场景蓝图。

```
专家 → 编写场景蓝图 → 审核 → 上线
```

**典型场景**：
- 新兴领域：还没有足够历史数据，无法从知识库提炼，但专家知道标准流程
- 定制场景：用户明确知道自己的业务，不需要系统帮他推导
- 简单场景：流程简单，直接填表就能定义

**特点**：
- 不需要知识库
- 不需要执行记录
- 直接从专家头脑中提取
- 质量取决于专家经验

#### 方式二：从知识库推导

从认知库（Grasp）中积累的历史知识提炼场景。

```
执行完成 → 认知积累（Grasp）→ 同领域认知聚合 → LLM 总结场景模板 → 用户确认 → 存入场景库
```

**触发时机**：
- 认知库积累了大量同领域 facts/patterns/lessons
- 多个认知条目具有相同领域标签
- 系统建议"是否保存为场景？"

**关键**：
- 知识库是辅助，不是必需
- LLM 负责总结和结构化
- 最终由专家确认，保证质量

#### 方式三：从执行中捕捉

从正在执行的目标/项目中提炼出可复用的场景。

```
执行中的 Goal/Project → 用户点击"保存为场景" → 打包 DAG + Tasks → 编辑完善 → 存入场景库
```

**触发时机**：
- 某个执行路径被验证有效
- 用户希望将其固化为可复用模板

**特点**：
- 来源是真实执行记录，不是专家头脑
- 可能包含专家没有意识到的细节
- 质量取决于执行记录完整性

### 3.3 事前定义的产出：完整场景蓝图

无论用哪种方式定义，事前定义的产出都是一个**完整的场景蓝图**：

```
事前定义的产出：

{
  "name": "化工厂危化品泄漏应急预案",
  "fullset": {
    "goal_tags": { ... },       ← 目标级能力需求
    "projects": [                ← 所有可能的 Project
      { "name": "应急响应",     ← mandatory
        "tasks": [...] },
      { "name": "疏散",         ← conditional
        "condition": {...},
        "tasks": [...] },
      ...
    ]
  }
}

场景蓝图 = 事前定义的完整"零件库"
```

### 3.4 事后提炼（执行完成 → 回流优化）

除了事前定义，还有**事后提炼**——执行完成后的反馈回流：

```
执行完成 → 统计数据写入场景库
             → 高频路径 → 建议固化
             → 问题路径 → 建议优化
             → 反馈 → 优化场景定义 → 新版本
```

**注意**：事后提炼不是场景的来源，而是场景的**优化闭环**。

### 3.5 来源方式汇总

| 阶段 | 方式 | 是否需要知识库 | 典型场景 |
|------|------|--------------|---------|
| **事前定义** | 纯人工定义 | ❌ 不需要 | 新兴领域、定制场景 |
| **事前定义** | 知识库推导 | ✅ 需要 | 有历史沉淀的领域 |
| **事前定义** | 执行捕捉 | ❌ 不需要（用执行记录） | 正在运行的系统 |
| **事后提炼** | 反馈回流 | ❌ 不需要 | 已有场景的优化 |

**关键澄清**：
- 事前定义不需要知识库，纯人工也可以
- 知识库推导是"锦上添花"，不是唯一路径
- 执行捕捉是"从实践中学习"，也不需要知识库
- 知识库的价值在于：某些领域的历史知识太复杂，只有专家经验的场景不够用，需要从大量执行记录中提炼

### 3.4 来源类型汇总

### 3.6 来源类型汇总

| source 值 | 含义 | 是否需要知识库 | 所属时间维度 |
|-----------|------|--------------|------------|
| `manual` | 专家纯人工定义 | ❌ 不需要 | 事前定义 |
| `ai_generated` | LLM 基于目标标题自动生成 | ❌ 不需要 | 事前定义 |
| `cognitive_derived` | 从认知库推导 | ✅ 需要 | 事前定义 |
| `execution_derived` | 从执行记录中提炼 | ❌ 不需要（用执行记录） | 事前定义 |
| `execution_flowback` | 执行完成后自动回灌 | ❌ 不需要 | 事后提炼 |
| `template` | 从模板复制改造 | ❌ 不需要 | 事前定义 |
| `evolved` | 执行反馈驱动自动演化 | ❌ 不需要 | 事后优化 |

**关键说明**：
- `execution_derived`（执行推导）本质上是**事前定义的一种方式**——从真实执行记录中提炼，不是"实时捕捉"。执行记录只是信息来源，不是实时触发的意思。
- 知识库（认知库）是**辅助工具**，不是必需条件
- `manual` 和 `execution_derived` 都不依赖知识库，全凭专家经验或执行实践

## 四、场景的定义结构

### 4.1 三层结构：基本信息 + 项目流程 + 任务模板

场景通过三层结构完整定义：

```
┌─────────────────────────────────────────────────┐
│           场景定义的三层结构                       │
│                                                   │
│  ┌─────────────────────────────────────────┐   │
│  │ 第一层：基本信息                           │   │
│  │                                          │   │
│  │  - 名称（name）                          │   │
│  │  - 分类（category）：earthquake/fire/...  │   │
│  │  - 描述（description）                    │   │
│  │  - 触发条件（triggers）                   │   │
│  │  - 来源类型（source）                     │   │
│  │  - 认知关联（cognition_refs）             │   │
│  └─────────────────────────────────────────┘   │
│                                                   │
│  ┌─────────────────────────────────────────┐   │
│  │ 第二层：项目流程（Project Workflow）        │   │
│  │                                          │   │
│  │  Phase 1: 应急响应                       │   │
│  │    depends_on: []                        │   │
│  │  Phase 2: 现场处置                       │   │
│  │    depends_on: [Phase 1]                 │   │
│  │  Phase 3: 善后恢复                       │   │
│  │    depends_on: [Phase 2]                 │   │
│  └─────────────────────────────────────────┘   │
│                                                   │
│  ┌─────────────────────────────────────────┐   │
│  │ 第三层：任务模板（Task Templates）          │   │
│  │                                          │   │
│  │  Phase 1 下的任务：                       │   │
│  │    - 水位监测（agent: 蚊子, priority: high）│   │
│  │    - 预警发布（agent: 蚊子, priority: critical）│   │
│  │  Phase 2 下的任务：                       │   │
│  │    - 人员疏散（agent: 谷子, priority: critical）│   │
│  │    ...                                   │   │
│  └─────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

### 4.2 场景的核心数据结构（fullset）

### 4.3 字段汇总

| 层级 | 字段 | 类型 | 说明 |
|------|------|------|------|

## 六、场景与标签系统的配合

### 6.1 标签系统回顾

能力标签体系用四个维度描述能力：

| 维度 | 含义 | 示例 |
|------|------|------|
| **业务能力** (business) | 懂不懂这个行业 | emergency_response, chemical_hazard |
| **专业能力** (professional) | 能做什么专业工作 | incident_command, risk_assessment |
| **技术能力** (technical) | 用什么工具/语言 | python, fastapi, postgresql |
| **管理能力** (management) | 会不会用系统 | coordination, task_dispatch, grasp_read |

### 6.2 场景是标签的需求方

```
场景的 fullset 中定义 capability_tags：

{
  "goal_tags": { ... },  ← Goal 级别的能力需求
  "projects": [
    {
      "name": "应急响应",
      "capability_tags": { ← 这个项目需要什么能力
        "business": ["emergency_response"],
        "professional": ["incident_command"],
        "technical": ["python"],
        "management": ["coordination", "task_dispatch"]
      },
      "tasks": [
        {
          "name": "启动应急预案",
          "capability_tags": { ← 这个任务需要什么能力
            "business": ["emergency_response"],
            "management": ["coordination", "task_dispatch"]
          }
        }
      ]
    }
  ]
}

匹配过程：
场景的 capability_tags ──▶ 标签系统 ──▶ Agent
"需要协调能力"              ──▶ 找到刚子
"需要应急经验"              ──▶ 找到谷子
"需要 Python 技术"          ──▶ 找到扣子
```

### 6.3 场景标签的双向作用

```
正向：场景定义需求 ──▶ 匹配 Agent
  "这个任务需要 coordination 能力"
       ──▶ 找到有 coordination 标签的 Agent

反向：Agent 执行结果 ──▶ 丰富场景
  "Agent 用 Python 完成了应急任务"
       ──▶ 场景的 technical 标签增加 python
       ──▶ 场景的 success_rate 更新
       ──▶ 场景的 feedback 回流

闭环：
场景定义需求 → 匹配 Agent → 执行任务
     → 产生结果 → 丰富场景标签 → 优化下次匹配
```

### 6.4 标签衰减模型

每个标签附带一个**置信度权重** (0.0 ~ 1.0)：

```
初始权重：
  - 手动配置：1.0（最高置信度）
  - 首次观察到：0.3（低置信度）

更新规则：
  - 观察到新证据：weight = min(1.0, weight + 0.1)
  - 未观察到（过期）：weight = max(0.0, weight - 0.05)

过期阈值：
  - technical: 30 天未用 → 权重降至 0
  - professional: 60 天未用 → 权重降至 0
  - business: 90 天未用 → 权重降至 0
  - management: 持续有效（不会过期）

为什么不同维度不同过期时间：
  - 技术栈更新快（30 天没用可能已经换了）
  - 专业能力中等（60 天）
  - 业务领域相对稳定（90 天）
  - 管理能力是系统功能，不会"过期"
```

## 七、场景的时间维度：事前 / 实例化 / 执行

这是场景库最核心的设计原则：**场景在事前定义，实例化时裁剪，执行时变更**。三个时间维度缺一不可。

### 7.1 事前定义：场景 = 全集（PMBOK）

场景在问题发生之前就已经具备定义的能力。它包含：

```
一个场景 = 所有可能要做的事 + 所有可能的分支 + 所有决策点

来源可以是：
- 专家纯人工定义（不依赖知识库）
- 从认知库提炼（知识辅助）
- 从执行记录捕捉（实践提炼）
- 从模板复制改造（复用已有）

类似：
- PMBOK 包含项目管理所有可能的方法论
- 医院急救指南包含所有可能的急救场景
- 烹饪菜谱包含所有可能的食材搭配和步骤

场景的全集（fullset）包含：
- 目标级能力需求（goal_tags）
- 所有可能的 Project（mandatory + conditional）
- 每个 Project 下的所有 Task
- 每个 Task 的能力要求（capability_tags）
- 每个 Project/Task 的触发条件（condition）
- HITL 定义（人类介入节点）
```

**全集的规模**：一个场景可能包含 10-50 个 Project 和 50-200 个 Task，因为要涵盖"所有可能的分支"。但实际执行时，一个具体 Goal 只会用到其中的 10-20%。

### 7.2 实例化时裁剪：根据已知信息选取子集

实例化发生在用户创建一个具体 Goal 的时候。此时只知道部分信息（现场情况、用户描述），需要从全集中选出"已知必须做"的部分：

```
场景全集（PMBOK）
    │
    │  实例化时已知信息：
    │  - 化工厂泄漏 ✅
    │  - 泄漏源未确认 ❌
    │  - 风向未知 ❌
    │  - 火情未知 ❌
    │
    ▼
初始裁剪规则：
  - mandatory 项目/Task → 全部创建
  - conditional：条件已满足 → 创建
  - conditional：条件未满足 → 暂不创建

初始 Goal：
  ✅ Project: 应急响应
     - Task: 启动应急预案
     - Task: 通知指挥中心
  ❌ Project: 泄漏源控制（泄漏源未确认，条件未满足）
  ❌ Project: 疏散（风向未知，条件未满足）
  ❌ Project: 善后恢复（还未到那一步）

初始 Goal = 全集裁剪后的子集（约 10-20% 的场景内容）
```

### 7.3 执行时变更：按场景预设路径动态增减

这是最关键的设计：**执行中的变更不是随意修改场景，而是由场景预设的条件触发，从全集中取出对应的 Project/Task**。

```
时间线 ────────────────────────────────────────────▶

T=0: 创建 Goal → 初始裁剪
     → 只创建"应急响应"项目和对应任务

T=5min: 现场报告"风向是北风"
     → 引擎读取场景全集
     → 条件 wind_direction == 'north' 满足
     → 创建"人员疏散南侧"项目 + 对应任务
     → 这是执行中的"增"

T=15min: 现场报告"泄漏源已确认"
     → 引擎读取场景全集
     → 条件 leak_source_confirmed 满足
     → 创建"泄漏源控制"项目 + 对应任务

T=30min: 现场报告"泄漏量很小"
     → 引擎根据场景中的可裁剪指南
     → 跳过/暂停"大规模疏散"项目
     → 这是执行中的"减"

执行结束时，Goal 可能经历了：
  - 初始裁剪：5 个 Project
  - 执行中动态增加：2 个 Project（条件满足触发）
  - 执行中动态跳过：1 个 Project（条件不再满足）
  - 最终执行：6 个 Project（含初始 + 增加 - 跳过）
```

**执行时变更的规则**：
- **"增"**：条件满足 → 从全集中取出对应 Project/Task → 创建到 Goal
- **"减"**：条件不再满足（如现场情况变化）→ 跳过/暂停对应 Project
- **变更的依据**：场景全集（不是随意修改场景蓝图）
- **变更的触发**：场景中预设的条件（condition）

### 7.4 场景全集是执行时变更的"零件库"

```
场景全集（事前定义）：

{
  "projects": [
    { "name": "应急响应",    "type": "mandatory"    },  ← 初始就创建
    { "name": "泄漏源控制",  "type": "conditional",
      "condition": { "expr": "leak_source_confirmed" } },  ← T=15min 满足时创建
    { "name": "疏散-北侧",  "type": "conditional",
      "condition": { "expr": "wind_direction == 'north'" } }, ← T=5min 满足时创建
    { "name": "疏散-南侧",  "type": "conditional",
      "condition": { "expr": "wind_direction == 'south'" } }, ← 始终未满足，不创建
    { "name": "善后恢复",  "type": "conditional",
      "condition": { "expr": "all_tasks_completed" } }  ← 最后满足
  ]
}

执行过程中：
  - 初始：创建"应急响应"
  - T=5min：创建"疏散-北侧"（风向=北）
  - T=15min：创建"泄漏源控制"（确认泄漏源）
  - T=30min：跳过"疏散-南侧"（风向≠南）
  - 最后：创建"善后恢复"（所有任务完成）

场景全集 = 事前定义的"零件库"，执行时按需取用
```

### 7.5 事前 / 实例化 / 执行三阶段的关系

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   事前定义              实例化裁剪           执行时变更     │
│                                                             │
│   场景                 Goal                  Goal          │
│   =                   =                      =             │
│   全集                 已知子集              动态子集       │
│   (PMBOK)             (初始计划)            (实际执行)     │
│                                                             │
│   不变                 一次性裁剪             条件触发      │
│   除非主动编辑         根据已知信息           动态增减      │
│                                                             │
│   引擎不可修改          引擎只读全集           引擎按条件取用 │
│                                                             │
└─────────────────────────────────────────────────────────────┘

三者之间的约束：

1. Goal 的 Project/Task 集合 ⊆ 场景全集
   （Goal 里的所有内容，都必须在场景的全集里）

2. 实例化后的 Goal ⊆ 执行中的 Goal
   （执行只能比初始裁剪多，不能凭空创造场景外的内容）

3. 场景全集中的 conditional 项目/Task
   可能在执行中被创建，也可能在执行中被跳过
```

## 八、Human-in-the-Loop（HITL）

### 8.1 核心理念

**Task 是唯一的执行单元**。Goal 和 Project 提供组织能力，但不直接执行。HITL 的暂停和恢复最终都落到 Task 级别。

HITL 不是"问一个问题"，而是一个**完整的生命周期**：
```
场景定义（写蓝图） → 目标执行（暂停） → 人类响应 → 引擎恢复
```

### 8.2 三种 HITL 类型

| 类型 | 场景定义格式 | 目标执行时行为 | 适用场景 |
|------|-------------|---------------|---------|
| **信息收集** `human_input` | 填空式，需要人类提供具体信息 | 暂停，显示输入框，收集后存入 context → 继续执行 | "当前风向？"、"泄漏量多少？" |
| **决策选择** `human_decision` | 选择题，给选项 | 暂停，显示选项，人类选择 → 按 branches 走对应分支 | "是否疏散？A/B/C" |
| **争议仲裁** `human_arbitration` | 两个 Agent 方案冲突 | 暂停，展示两个方案对比，人类裁决 | "方案 A vs 方案 B，你选哪个？" |

### 8.3 场景中的 HITL 定义

场景作者在蓝图中定义 HITL 条件：

```json
{
  "condition": {
    "type": "human_decision",
    "prompt": "根据当前火情，是否需要疏散周边人员？",
    "options": ["立即疏散", "暂缓监测", "不需要"],
    "default": "暂缓监测",
    "timeout_action": "use_default",
    "timeout_minutes": 30,
    "branches": {
      "立即疏散": { "next_projects": ["人员疏散"] },
      "暂缓监测": { "next_projects": ["持续监测"] },
      "不需要": null
    }
  }
}
```

### 8.4 HITL 的完整生命周期

```
Step 1: 引擎遇到 HITL 条件（非 auto_eval 类型）
  └─ 创建 HumanInputRequest，status = pending
  └─ 暂停当前分支的所有 Task 调度
  └─ Goal/Project 标记为 awaiting_human_input

Step 2: 通知
  └─ 通知 Goal 的 verifier_agent（如刚子）
  └─ 前端显示"需要人类决策"弹窗
  └─ 推送通知到飞书/微信（可选）

Step 3: 人类响应
  └─ 人类点击弹窗，看到问题 + 选项/输入框
  └─ 人类选择/填写并提交
  └─ 更新 HumanInputRequest：status = answered

Step 4: 恢复执行
  └─ 引擎读取答案
  └─ 查 branches → 确定下一步
  └─ 创建对应的 Project/Task（或跳过）
  └─ 恢复调度循环

Step 5: 超时处理
  └─ 超过 timeout_minutes 无人响应
  └─ 根据 timeout_action 执行：
     - use_default: 使用 default 值继续
     - skip_project: 跳过整个 Project
     - skip_task: 跳过当前 Task
     - escalate: 升级通知（更高级别的 verifier）
```

### 8.5 前端 UI

根据 `input_type` 渲染不同界面：

| input_type | UI 组件 |
|-----------|--------|
| `human_input` (text) | 文本输入框 + 提交按钮 |
| `human_input` (number) | 数字输入框 |
| `human_input` (multiline) | 多行文本框 |
| `human_decision` | 单选按钮组 / 下拉选择 |
| `human_arbitration` | 左右对比面板（方案 A vs 方案 B）+ 选择按钮 |

## 九、场景与日志系统的配合

### 9.1 日志系统的定位

日志系统是场景的**执行记录仪**和**优化依据**：

| 功能 | 说明 |
|------|------|
| **执行追踪** | 记录每次场景实例化的完整执行链路 |
| **统计分析** | 计算场景的成功率、平均耗时、冲突次数 |
| **版本对比** | 记录场景每次变更的执行效果差异 |
| **优化依据** | 基于执行日志优化场景定义 |

### 9.2 场景执行的全链路追踪

```
场景实例化 ──▶ 全链路追踪

traceId: a1b2c3d4-e5f6-7890-abcd-ef1234567890

[用户创建 Goal]
     │ 选择场景：化工厂危化品泄漏应急预案
     ▼
[场景实例化]
     │ traceId 生成，关联 scenario_id
     ▼
[任务1: 启动应急预案]
     │ traceId 携带，记录 Agent、耗时、结果
     ▼
[任务2: 通知指挥中心]
     │ traceId 携带，记录 Agent、耗时、结果
     ▼
[条件分支：是否需要疏散]
     │ traceId 携带，记录人类决策和分支选择
     ▼
[任务3: 疏散南侧村庄]（分支"北风"触发）
     │ traceId 携带，记录 Agent、耗时、结果
     ▼
[执行完成]
     ▼
[反馈回流]
     │ 统计数据写入场景库
     │ 成功率、耗时、冲突次数更新

任意节点都可以通过 traceId + scenario_id 查到
该场景的完整执行历史
```

### 9.3 场景统计的数据流

```
每次场景实例化产生的数据：

1. 执行日志 (execution_logs)
   - 每个任务的成功/失败/超时
   - 耗时、Agent、结果摘要

2. 统计数据 (scenario metrics)
   - 总执行次数 / 成功次数 / 失败次数
   - 成功率
   - 平均耗时 / 最短耗时 / 最长耗时
   - 平均冲突次数
   - 平均步骤完成率

3. 版本记录 (versions)
   - 每次场景变更的版本号
   - 版本号对应的执行效果
   - 版本间的差异对比

4. 认知关联 (scenario_cognitions)
   - 场景来源于哪些认知条目
   - 认知条目的相关度评分

这些数据用于：
- 场景推荐排序（成功率高的排前面）
- 场景版本优化（新版本比旧版本好吗）
- 场景裁剪建议（哪些步骤经常被跳过）
- Agent 匹配优化（哪些 Agent 执行效果好）
```

## 十、场景的版本管理

### 10.1 为什么需要版本

场景不是一次性写完就不变的。它随着执行反馈不断进化：

```
场景进化路径：

v1.0  ── 首次创建
  │
  │  执行 10 次，成功率 85%
  │  反馈：疏散步骤经常被跳过，不需要
  │
v1.1  ── 移除疏散步骤，简化流程
  │
  │  执行 5 次，成功率 95%
  │  反馈：缺少泄漏源确认步骤
  │
v1.2  ── 新增泄漏源确认步骤
  │
  │  执行 3 次，成功率 100%
  │  → 当前版本
```

### 10.2 版本与执行效果的关联

| 版本 | 执行次数 | 成功率 | 平均耗时 | 变化原因 |
|------|----------|--------|----------|----------|
| v1.0 | 10 | 85% | 45 分钟 | 首次创建 |
| v1.1 | 5 | 95% | 30 分钟 | 移除冗余步骤 |
| v1.2 | 3 | 100% | 35 分钟 | 新增必要步骤 |

### 10.3 版本对比

用户可以对比两个版本的差异：

```
v1.0 → v1.1 变更：
  - 移除步骤："疏散通知"
  - 修改步骤："现场处置" → 增加"确认泄漏源"
  - 成功率：85% → 95% ↑
  - 平均耗时：45min → 30min ↓

结论：v1.1 更优，建议升级
```

## 十一、场景的匹配与推荐

### 11.1 场景匹配引擎

当用户创建一个新目标时，系统自动推荐最合适的场景：

```
┌─────────────────────────────────────────────────┐
│           场景匹配引擎                             │
│                                                   │
│  输入：                                           │
│  - 目标标题："某化工厂危化品泄漏了"                │
│  - 目标描述："储罐区发生泄漏，有人报告"            │
│                                                   │
│  匹配过程：                                       │
│                                                   │
│  1. 关键词提取                                   │
│     → [化工厂, 危化品, 泄漏, 储罐]                │
│                                                   │
│  2. 能力标签匹配                                  │
│     → business: chemical_hazard, emergency         │
│     → professional: incident_command              │
│                                                   │
│  3. 场景库扫描                                    │
│     → 计算每个场景的匹配度                         │
│                                                   │
│  4. 排序推荐                                      │
│     ✅ 化工厂危化品泄漏应急预案 (92%)              │
│     ⚠️  工业安全事故通用响应 (65%)                │
│     ❌ 城市内涝应急响应 (12%)                    │
│                                                   │
│  输出：推荐场景列表 + 匹配度 + 匹配理由            │
└─────────────────────────────────────────────────┘
```

### 11.2 匹配度计算

```
匹配度 = Σ(场景需求标签 ∩ 目标关键词的权重) / 场景需求标签总数

示例：

目标需要：{"化工厂", "危化品", "泄漏", "应急"}
场景A有：  {"化工厂", "危化品", "泄漏", "应急", "疏散", "监测"}

交集 = {"化工厂", "危化品", "泄漏", "应急"} (4 个全中)
得分 = 4 / 4 = 100% → 完美匹配

场景B有：  {"化工厂", "应急", "火灾", "疏散"}
交集 = {"化工厂", "应急"} (2/4)
得分 = 2 / 4 = 50% → 部分匹配
```

## 十三、设计原则

### 13.1 核心原则

| 原则 | 说明 |
|------|------|
| **场景 = 全集** | 场景是所有可能要做的事，目标是根据现场信息裁剪出的子集 |
| **事前定义** | 场景在问题发生之前就定义好，是完整的"零件库" |
| **实例化裁剪** | 根据已知信息从全集中选取初始子集（mandatory + 条件已满足） |
| **执行时变更** | 根据新情况从全集中取出对应 Project/Task，不是随意修改场景 |
| **动态裁剪** | 不是一次性实例化就结束，而是执行中持续按条件从全集中调整 |
| **条件驱动** | 场景中的 condition 是变更的触发器，不是线性执行 |
| **能力标签** | 场景用四维标签描述需求，与 Agent 的标签系统对接 |
| **反馈回流** | 执行结果回流到场景库，场景越用越准 |
| **HITL 内建** | 场景定义时就决定哪些节点需要人类介入 |
| **版本管理** | 场景随反馈进化，版本间可对比可回滚 |
| **双通道来源** | 认知库提炼 + 执行中捕捉，两条通道互为补充 |

### 13.2 系统边界

| 属于场景库 | 不属于场景库 |
|-----------|-------------|
| 场景定义和存储 | Agent 执行过程 |
| 场景匹配和推荐 | 任务派发和调度 |
| 场景实例化和裁剪 | 任务验证和裁决 |
| 场景统计和版本 | 日志存储和分析 |
| HITL 条件定义 | HITL 通知和 UI |
| 场景来源管理 | 认知库本身 |

### 13.3 与外部系统的交互

```
场景库的外部依赖：

┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  认知库       │     │ 驾驭中心     │     │ 消息系统     │
│  (Grasp)     │     │ (Scheduler)  │     │ (飞书等)     │
│              │     │              │     │              │
│ 认知推导场景  │     │ 场景匹配     │     │ HITL 通知    │
│ 执行推导场景  │     │ 实例化调度   │     │ 场景更新     │
│ 知识沉淀     │     │ 动态调整     │     │ 场景推荐     │
└──────┬───────┘     └──────┬───────┘     └──────┬───────┘
       │                    │                    │
       └────────────────────┼────────────────────┘
                            │
```

## 附录：概念速查表

| 概念 | 一句话解释 |
|------|-----------|
| **场景** | 可复用的标准化解决方案模板，回答"遇到这类问题怎么做" |
| **全集** | 场景包含的所有可能任务/分支/决策点（类似 PMBOK 全书） |
| **事前定义** | 场景在问题发生之前就定义好，是完整的"零件库" |
| **实例化** | 从场景全集中根据已知信息裁剪出初始 Goal 的过程 |
| **实例化裁剪** | 根据当时已知的信息（现场情况/用户输入）从全集中选取子集 |
| **执行时变更** | 执行中按预设条件从全集中取出对应 Project/Task，不是随意修改 |
| **子集** | 根据现场信息从全集中裁剪出的实际执行路径 |
| **动态裁剪** | 执行中根据新信息从全集中持续调整（增/减）Project 和 Task |
| **条件驱动** | 场景中的 condition 是变更的触发器，满足条件则取出对应内容 |
| **HITL** | Human-in-the-Loop，场景中需要人类介入的节点 |
| **信息收集** | HITL 类型之一，需要人类提供具体信息 |
| **决策选择** | HITL 类型之一，需要人类做选择题 |
| **争议仲裁** | HITL 类型之一，两个方案冲突需要人类裁决 |
| **场景匹配** | 根据目标描述从场景库中推荐最合适的场景 |
| **能力标签** | 四维标签（business/professional/technical/management） |
| **场景反馈** | 执行结果回流到场景库，用于优化场景定义 |
| **场景版本** | 场景随反馈进化的版本号，支持对比和回滚 |
| **场景来源** | 手动创建/认知推导/执行推导/模板复制/自动演化 |
| **fullset** | 场景的全集标签，包含 goal_tags/projects/tasks 三层能力要求 |
| **驾驭中心** | 调度决策引擎，读取场景蓝图决定任务派发 |
| **标签系统** | 能力匹配体系，场景是需求方，Agent 是供给方 |
| **认知库** | 知识沉淀系统，是场景来源之一 |
| **双通道** | 事后提炼（认知库）+ 实时捕捉（执行中），两条来源通道 |
| **条件满足** | 执行中满足场景预设的 condition → 触发器取出对应 Project/Task |
