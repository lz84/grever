## 一、心跳的本质

### 1.1 心跳是什么

心跳是 Nexus 系统中 **Agent 与平台之间的双向沟通机制**。它不是"报活"，而是 Agent 生命周期中最重要的 **控制通道**：

```
Agent ──心跳──▶ Nexus
    ◀──任务派发──
    ◀──配置变更──
    ◀──状态指令──
```

一次心跳完成三件事：
1. **报活**：Agent 告诉平台"我还活着，当前状态是什么"
2. **领活**：平台告诉 Agent "你有 X 个待执行任务"
3. **更新**：平台将 Agent 的负载、能力、连接状态同步到全局状态

### 1.2 心跳 ≠ 健康检查

| 维度 | 心跳 | 健康检查 |
|------|------|----------|
| 触发方 | Agent 主动发起 | 平台主动扫描 |
| 频率 | 高频（30-300秒） | 低频（5-15分钟） |
| 内容 | 状态 + 领任务 | 连通性判断 |
| 目的 | 实时通信 | 离线检测 |

心跳是 **Agent 自证**，健康检查是 **平台巡检**。两者配合，构成完整的 Agent 状态感知体系。

### 1.3 核心定位

心跳系统在 Nexus 中的定位：

```
┌─────────────────────────────────────────────────┐
│              Nexus 智能体管理层                    │
│                                                   │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐   │
│  │ 能力标签  │◄───│  心跳系统  │───▶│ 日志系统  │   │
│  │  体系    │    │ (核心枢纽) │    │          │   │
│  └──────────┘    └──────────┘    └──────────┘   │
│       ▲              │              ▲            │
│       │              ▼              │            │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐   │
│  │ 智能匹配  │◄───│ 驾驭中心  │◄───│ 裁决中心  │   │
│  │          │    │          │    │          │   │
│  └──────────┘    └──────────┘    └──────────┘   │
└─────────────────────────────────────────────────┘
```

**心跳是核心枢纽**：
- 向上：驱动驾驭中心的调度决策
- 向左：为能力标签体系提供实时行为信号
- 向右：向日志系统输出结构化执行记录
- 向下：直接管理 Agent 的生命周期

## 二、心跳的完整生命周期

### 2.1 单次心跳的业务流程

```
┌─────────┐     心跳请求      ┌─────────┐
│  Agent   │ ───────────────▶ │  Nexus   │
│          │                  │  Platform│
│ 上报:    │                  │          │
│ - 当前状态 │                  │ 1. 记录心跳时间              │
│ - 负载信息 │                  │ 2. 验证模型连通性            │
│ - 正在执行 │                  │ 3. 更新 Agent 状态           │
│   的任务数 │                  │ 4. 分配待执行任务            │
│ - 模型信息 │                  │ 5. 记录执行日志              │
│          │                  │ 6. 更新能力标签              │
│          │◀──────────────── │ 7. 返回 pending 任务列表     │
│          │   心跳响应        │          │
│ 拿到:    │                  │          │
│ - 新任务  │                  │          │
│ - 配置    │                  │          │
│ - 指令    │                  │          │
└─────────┘                  └─────────┘
```

### 2.2 心跳的七个处理阶段

| 阶段 | 名称 | 做什么 | 产出 |
|------|------|--------|------|
| 1 | **连通性验证** | 检查 Agent 的模型 API 是否可用 | 验证通过/失败 |
| 2 | **状态更新** | 更新 last_heartbeat、status、load | Agent 状态变更 |
| 3 | **负载计算** | 基于当前任务和并发限制计算新负载 | 负载评分 (0-100) |
| 4 | **任务分配** | 将 pending 任务分配给该 Agent | 任务 ID 列表 |
| 5 | **日志记录** | 记录本次心跳的完整执行链路 | 结构化日志条目 |
| 6 | **标签推断** | 根据本次行为更新能力标签权重 | 标签权重变更 |
| 7 | **响应返回** | 将待执行任务和配置变更返回给 Agent | 心跳响应 |

### 2.3 心跳周期

```
时间线 ─────────────────────────────────────────────▶

Agent 注册
    │
    ▼
┌─────┐     ┌─────┐     ┌─────┐     ┌─────┐
│心跳1 │────▶│心跳2 │────▶│心跳3 │────▶│心跳4 │  ← 正常周期
│领任务│     │领任务│     │领任务│     │空闲  │
└─────┘     └─────┘     └─────┘     └─────┘
                              │
                         超时未响应
                              │
                              ▼
                         ┌─────────┐     ┌─────────┐
                         │健康检查  │────▶│标记离线  │  ← 异常处理
                         │(被动)   │     │          │
                         └─────────┘     └─────────┘
```

## 三、心跳与驾驭中心的配合

### 3.1 驾驭中心是什么

驾驭中心是 Nexus 的 **调度决策引擎**，负责回答三个问题：

1. **谁该干什么** —— 任务分配决策
2. **谁干得最好** —— 智能匹配决策
3. **出了问题怎么办** —— 异常处理决策

### 3.2 心跳为驾驭中心提供什么

心跳是驾驭中心的 **实时数据源**：

```
                    ┌──────────────────────────────┐
                    │         驾驭中心              │
                    │                              │
                    │  ┌────────┐  ┌──────────┐   │
                    │  │调度策略 │  │匹配规则  │   │
                    │  │引擎    │  │引擎      │   │
                    │  └───┬────┘  └────┬─────┘   │
                    │      │            │          │
                    └──────┼────────────┼──────────┘
                           │ 决策依据    │
                           ▼            │
┌──────────────────────────────────┐    │
│         心跳系统                 │    │
│                                  │    │
│ 每次心跳输出：                   │    │
│  ┌──────────────────────────┐   │    │
│  │ Agent 实时状态            │   │    │
│  │ - online/busy/offline    │───┼────┘
│  │ - 当前负载 (0-100)        │
│  │ - 正在执行的任务数        │─── 驾驭中心用它判断"这个 Agent 还能不能接活"
│  │ - 模型连通性             │
│  │ - 连续离线次数           │─── 驾驭中心用它判断"这个 Agent 靠不靠谱"
│  └──────────────────────────┘   │
│                                  │
│ 每次心跳触发：                   │
│  ┌──────────────────────────┐   │
│  │ 任务派发机会             │   │
│  │ - 检查 pending 任务队列  │─── 驾驭中心在心跳响应时决定"给不给他派活"
│  │ - 能力匹配              │
│  │ - 负载检查              │─── 驾驭中心在心跳响应时决定"派多少活"
│  │ - 优先级排序            │
│  └──────────────────────────┘   │
└──────────────────────────────────┘
```

### 3.3 驾驭中心的调度时机

驾驭中心 **只在心跳时** 做调度决策（推送式架构）：

```
触发条件：Agent 发来心跳
         ↓
驾驭中心判断：
  1. 这个 Agent 在线吗？（连通性验证）
  2. 这个 Agent 还能接活吗？（负载检查 < 80）
  3. 有适合他的任务吗？（能力匹配）
  4. 有优先级更高的 Agent 吗？（全局最优）
         ↓
决策输出：
  ✅ 派发任务 → 心跳响应中包含 pending 任务列表
  ❌ 不派发 → 心跳响应为空，Agent 继续等待
```

### 3.4 驾驭中心的心跳策略

| 场景 | 策略 | 说明 |
|------|------|------|
| **正常派发** | 能力匹配 + 负载优先 | 选择最合适的 Agent |
| **紧急任务** | 跳过负载检查 | 即使 Agent 负载高也派发 |
| **Agent 离线** | 重新匹配 | 将任务重新分配给其他 Agent |
| **Agent 恢复** | 优先补偿 | 离线恢复后优先派发任务 |
| **连续失败** | 降级使用 | 连续失败超过阈值，降低匹配优先级 |

## 四、心跳与标签系统的配合

### 4.1 标签系统回顾

能力标签体系是 Nexus 的 **Agent 能力画像**，用四个维度描述 Agent：

| 维度 | 含义 | 示例 |
|------|------|------|
| **业务能力** (business) | 懂不懂这个行业 | emergency_response, software_development |
| **专业能力** (professional) | 能做什么专业工作 | report_writing, api_design |
| **技术能力** (technical) | 用什么工具/语言 | python, fastapi, postgresql |
| **管理能力** (management) | 会不会用系统 | coordination, task_dispatch, grasp_read |

### 4.2 心跳是标签更新的核心触发器

```
┌─────────────────────────────────────────────────┐
│           能力标签自动更新机制                     │
│                                                   │
│  Agent 执行任务                                    │
│      │                                            │
│      ▼                                            │
│  心跳时上报执行结果                                │
│      │                                            │
│      ▼                                            │
│  ┌─────────────────────┐                         │
│  │  标签推断引擎        │                         │
│  │                     │                         │
│  │  观察信号采集：      │                         │
│  │  - 任务类型 → 能力维度│                         │
│  │  - 产出物类型 → 专业  │                         │
│  │  - 使用技术 → 技术栈  │                         │
│  │  - 系统行为 → 管理    │                         │
│  │                     │                         │
│  │  权重更新：          │                         │
│  │  - 成功执行 +权重     │                         │
│  │  - 长期不用 -权重     │                         │
│  │  - 30/60/90天衰减    │                         │
│  └─────────────────────┘                         │
│      │                                            │
│      ▼                                            │
│  capability_tags 更新                              │
│      │                                            │
│      ▼                                            │
│  影响下次智能匹配                                   │
└─────────────────────────────────────────────────┘
```

### 4.3 心跳中的标签推断流程

```
单次心跳 ──▶ 标签推断

┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 任务执行结果  │───▶│ 信号提取     │───▶│ 标签匹配     │
│              │    │              │    │              │
│ - 完成了什么  │    │ - 提取任务    │    │ - 任务类型 →  │
│ - 怎么完成的  │    │   类型特征    │    │   business   │
│ - 用了什么    │    │ - 提取技术    │    │ - 使用工具 →  │
│ - 结果如何    │    │   栈特征      │    │   technical   │
│              │    │ - 提取产出物  │    │ - 产出物 →    │
│              │    │   特征        │    │   professional│
│              │    │ - 提取系统    │    │ - 系统调用 →  │
│              │    │   行为特征    │    │   management  │
└──────────────┘    └──────────────┘    └──────────────┘
                                                 │
                                                 ▼
                                          ┌──────────────┐
                                          │ 权重更新     │
                                          │              │
                                          │ 成功 → +0.1  │
                                          │ 超时 → -0.05 │
                                          │ 失败 → -0.1  │
                                          │ 上限 1.0     │
                                          │ 衰减 30/60/90│
                                          └──────────────┘
```

### 4.4 标签衰减模型

```
Agent 标签权重随时间衰减：

权重
1.0 ┤                                    ╱
    │                                ╱
0.8 ┤                            ╱
    │                        ╱
0.6 ┤                    ╱
    │                ╱
0.4 ┤            ╱
    │        ╱
0.3 ┤────╱          ← 初始权重（新标签）
    │  ╱
0.2 ┤╱
    │
0.0 ┼────────┬────────┬────────┬────────▶ 时间
    0       30天     60天     90天

规则：
- 首次获得标签：权重 0.3
- 每次成功执行同类型任务：+0.1（上限 1.0）
- 30 天未使用：权重 × 0.8
- 60 天未使用：权重 × 0.5
- 90 天未使用：权重归零（标签过期）
```

### 4.5 标签与匹配的闭环

```
┌─────────────────────────────────────────────────┐
│           标签驱动的调度闭环                       │
│                                                   │
│    心跳 ──▶ 标签更新 ──▶ 匹配优化 ──▶ 任务派发     │
│     │                                          │  │
│     │  Agent 执行任务                           │  │
│     │      │                                   │  │
│     │      ▼                                   │  │
│     │  产生行为信号 ──▶ 心跳上报                │  │
│     │                                          │  │
│     └────────── 闭环 ──────────────────────────┘  │
│                                                   │
│  结果：Agent 越干越准，标签越用越精                 │
│  "用进废退" —— 不用的标签会衰减，常用的标签会增强   │
└─────────────────────────────────────────────────┘
```

## 五、心跳与日志系统的配合

### 5.1 日志系统的定位

日志系统是 Nexus 的 **全链路记录器**，为心跳提供：

1. **可追溯性**：每次心跳都可回溯
2. **可分析性**：结构化数据支持事后分析
3. **可观测性**：实时监控 Agent 健康状态

### 5.2 心跳产生的日志类型

```
单次心跳产生的日志：

┌─────────────────────────────────────────────────┐

│     ── 记录心跳本身的信息                         │
│     - 时间、Agent ID、状态                        │
│     - 负载信息                                    │
│     - 耗时                                       │
│                                                   │
│  2. 执行日志 (execution_logs)                     │
│     ── 记录心跳中执行的操作                        │

│     - 错误信息                                    │
│                                                   │
│  3. 链路日志 (trace/spans)                        │
│     ── 记录跨 Agent 的调用链路                     │
```

## 六、心跳与裁决中心的配合

### 6.1 裁决中心是什么

裁决中心是 Nexus 的 **异常处理入口**，处理三类需要人工介入的情况：

| 类型 | 触发条件 | 处理方式 |
|------|----------|----------|
| **待裁决** (disputed) | 任务验证连续 3 次失败 | 人工判断任务是否通过 |
| **待审批** (waiting_human) | 任务完成需要确认 | 人工审批通过 |
| **待协助** (human_input) | 任务执行需要额外输入 | 人工提供输入 |

### 6.2 心跳如何触发裁决

```
正常流程：
Agent 心跳 ──▶ 领任务 ──▶ 执行 ──▶ 完成 ──▶ 验证通过 ✅

异常流程：
Agent 心跳 ──▶ 领任务 ──▶ 执行 ──▶ 完成 ──▶ 验证失败
                                              │
                                         重试 3 次
                                              │
                                         全部失败
                                              │
                                              ▼
                                      ┌──────────────┐
                                      │ 裁决中心      │
                                      │              │
                                      │ 状态变为     │
                                      │ "disputed"   │
                                      │              │
                                      │ 通知用户     │
                                      │ 等待裁决     │
                                      └──────────────┘
                                              │
                                         用户裁决
                                              │
                                    ┌─────────┴─────────┐
                                    ▼                   ▼
                                通过 ✅              驳回 ❌
                                    │                   │
                                    ▼                   ▼
                              标记完成              重新派发
```

### 6.3 心跳中的裁决信号

```
心跳响应中包含裁决相关指令：

┌─────────────────────────────────────────────────┐
│ 心跳响应                                         │
│                                                   │
│ {                                                 │
│   "pending_tasks": [...],                        │
│   "rulings": [                                    │
│     {                                             │
│       "task_id": "xxx",                          │
│       "type": "disputed",                        │
│       "ruling": "pass",  ← 用户裁决结果           │
│       "comment": "代码质量可以接受"               │
│     }                                            │
│   ],                                             │
│   "config_updates": {...}                        │
│ }                                                │
│                                                   │
│ Agent 拿到裁决结果后：                             │
│ - 如果 pass → 标记任务完成                        │
│ - 如果 reject → 领取新任务继续执行                │
└─────────────────────────────────────────────────┘
```

## 七、心跳的异常处理体系

### 7.1 心跳异常分类

```
心跳异常树：

心跳失败
├── 网络层
│   ├── 连接超时（Agent 端网络问题）
│   ├── DNS 解析失败
│   └── 连接被拒绝（平台端口变更）
│
├── 应用层
│   ├── 模型 API 不可用（vLLM 宕机）
│   ├── Agent 进程崩溃
│   └── 心跳数据格式错误
│
└── 业务层
    ├── Agent 负载超限（拒绝派发）
    ├── Agent 被禁用（管理员操作）
    └── 无可用任务（队列为空）
```

### 7.2 健康检查的被动兜底

当 Agent 长时间不来心跳时，健康检查被动介入：

```
时间线 ────────────────────────────────────────────▶

上次心跳: T=0
          │
          ▼
     T+5分钟 (stale)
          │  健康检查发现 5 分钟没心跳
          │  标记为 "stale"（亚健康）
          │  不派发新任务，但保留已有任务
          │
          ▼
     T+15分钟 (offline)
          │  健康检查发现 15 分钟没心跳
          │  标记为 "offline"（离线）
          │  已有任务挂起，等待重新分配
          │
          ▼
     T+N (恢复)
          │  Agent 重新心跳
          │  恢复为 "online"
          │  重新领取任务
```

### 7.3 连续失败处理

```
连续离线计数器：

心跳成功 ──▶ counter = 0
心跳失败 ──▶ counter += 1

counter = 0    ✅ 正常
counter = 1    ⚠️  第一次失败
counter = 2    ⚠️  第二次失败
counter = 3    🔴 标记为 offline，停止派发

恢复心跳 ──▶ counter 归零
```

## 八、心跳的配置管理

### 8.1 心跳参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| **心跳间隔** | 30-300 秒 | Agent 多久发一次心跳 |
| **在线阈值** | 60 秒 | 多少秒内有心跳算在线 |
| **忙碌阈值** | 120 秒 | 多少秒内有心跳算忙碌 |
| **离线阈值** | 300 秒 | 多少秒无心跳算离线 |
| **健康检查 stale 阈值** | 5 分钟 | 被动检测亚健康阈值 |
| **健康检查 offline 阈值** | 15 分钟 | 被动检测离线阈值 |
| **最大连续失败次数** | 3 次 | 超过此次数标记离线 |

### 8.2 不同触发模式的心跳策略

| 触发模式 | 心跳间隔 | 说明 |
|----------|----------|------|
| **轮询** (polling) | 30 秒 | Agent 主动定期来心跳 |
| **SSE** (server-sent) | 按需 | 平台推送，Agent 响应 |
| **混合** | 轮询 + SSE | 轮询保底，SSE 加速 |

## 十、设计原则

### 10.1 核心原则

| 原则 | 说明 |
|------|------|
| **心跳是主动轮询** | Agent 定期来，不是平台去查 |
| **心跳是双向通道** | 不只是报活，更是领活 |
| **心跳是调度时机** | 只在心跳时做调度决策 |
| **心跳是标签来源** | 每次心跳都是标签更新的机会 |
| **心跳是日志源头** | 每次心跳都产生结构化日志 |
| **心跳是健康信号** | 连续心跳失败 = 不健康 |

### 10.2 系统边界

| 属于心跳系统 | 不属于心跳系统 |
|-------------|---------------|
| Agent 状态感知 | Agent 内部执行逻辑 |
| 任务派发决策 | 任务执行过程 |
| 标签推断触发 | 标签匹配算法 |
| 心跳日志记录 | 日志存储和分析 |
| 连通性验证 | 模型 API 实现 |

### 10.3 与外部系统的交互

```
心跳系统的外部依赖：

┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  模型服务     │     │  知识系统     │     │  消息系统     │
│  (vLLM/API)  │     │  (Grasp)     │     │  (飞书等)     │
│              │     │              │     │              │
│ 连通性验证    │     │ 知识查询      │     │ 异常通知      │
│ 模型状态      │     │ 经验匹配      │     │ 裁决推送      │
└──────┬───────┘     └──────┬───────┘     └──────┬───────┘
       │                    │                    │
       └────────────────────┼────────────────────┘
                            │
                            ▼
                    ┌──────────────┐
                    │  心跳系统     │
                    └──────────────┘
```

| 概念 | 一句话解释 |
|------|-----------|
| **心跳** | Agent 与平台之间的双向沟通机制 |
| **报活** | Agent 告诉平台"我还活着" |
| **领活** | Agent 从平台领取待执行任务 |

| **健康检查** | 平台主动扫描 Agent 是否在线 |

| **offline** | 离线状态，15分钟无心跳 |
| **驾驭中心** | 调度决策引擎，回答"谁该干什么" |
| **标签系统** | Agent 能力画像，四维标签 + 权重 |
| **日志系统** | 全链路记录器，traceId 贯穿 |
| **裁决中心** | 异常处理入口，处理人工介入 |
| **traceId** | 全链路追踪 ID，贯穿整个请求生命周期 |
| **标签衰减** | 标签权重随时间自然衰减，用进废退 |
