## 1. 概述

任务分解是 Reins（御）驾驭能力的核心——将高层目标拆解为可执行的子任务，并识别依赖关系，实现最优并行。

### 1.1 核心目标

| 目标 | 说明 |
|------|------|
| **可执行** | 每个子任务 1-4 小时，Agent 可独立完成 |
| **可协作** | 明确依赖关系，支持多 Agent 并行 |
| **可追踪** | 父子任务关系清晰，进度可监控 |
| **可认知** | 分解过程可调用 Grasp 获取领域知识 |

### 1.2 分解原则

1. **粒度优先** — 子任务必须在 1-4 小时之内
2. **原子性** — 每个子任务描述一个独立操作
3. **可验证** — 每个子任务有明确的完成标志
4. **依赖清晰** — 子任务间依赖关系显式表达
5. **能力匹配** — 子任务所需能力与 Agent 能力匹配

## 2. 分解策略

### 2.1 策略类型

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **auto** | LLM 智能分解，结合 Grasp 认知 | 通用场景，默认策略 |
| **conservative** | 粗粒度分解，少拆多层 | 高层目标、探索性任务 |
| **aggressive** | 细粒度分解，拆到 1 小时 | 需要精确控制、并行度高的任务 |

### 2.2 auto 策略（默认）

**分解流程**:

```
输入: task { title, description, constraints }
         │
         ├─→ Grasp.retrieve({ query: description, type: [pattern, lesson] })
         │    └─→ 返回相关分解模式和最佳实践
         │
         ├─→ LLM 分解提示词（含 Grasp 认知上下文）
         │
         ├─→ 生成子任务候选列表
         │
         ├─→ 粒度校验（每个子任务 1-4h）
         │    ├─> 超过 4h → 继续分解
         │    └─< 1h → 合并相邻任务
         │
         ├─→ 依赖关系识别
         │
         └─→ 输出: { subtasks[], dependencies[], can_parallel[] }
```

**LLM 分解提示词模板**:

```
你是一个任务分解专家。请将以下任务分解为可执行的子任务。

## 原始任务
标题: {title}
描述: {description}
约束: {constraints}

## 分解要求
1. 每个子任务必须在 1-4 小时完成
2. 每个子任务必须可独立验证完成
3. 识别子任务间的依赖关系
4. 标注可并行执行的任务

## Grasp 认知上下文
{grasp_context}

## 输出格式
JSON格式，包含：
- subtasks: 子任务列表
- dependencies: 依赖关系
- can_parallel: 可并行任务组
```

### 2.3 conservative 策略

适用于：
- 高层目标（Epic）
- 探索性任务（不确定如何拆）
- 需要人类审阅的场景

**特点**:
- 分解深度最多 2 层
- 子任务粒度 3-8 小时
- 不强制并行优化
- 输出供人类审阅后再分解

### 2.4 aggressive 策略

适用于：
- 需要多 Agent 并行执行
- 精确进度控制
- 短期冲刺任务

**特点**:
- 分解深度可达 4 层
- 子任务粒度强制 1-2 小时
- 最大化并行度
- 自动识别串行瓶颈

## 3. 粒度标准

### 3.1 粒度定义

| 粒度 | 时长 | 说明 | 示例 |
|------|------|------|------|
| **XS** | 0.5-1h | 原子操作 | 写一个函数、修改一个配置 |
| **S** | 1-2h | 单一操作 | 实现一个 API 端点、写一个测试用例 |
| **M** | 2-3h | 小型任务 | 实现一个表单、完成一个组件 |
| **L** | 3-4h | 中型任务 | 实现一个模块、完成一个页面 |
| **XL** | >4h | 大型任务（需继续分解） | 实现一个子系统 |

### 3.2 粒度校验规则

```
IF estimated_hours < 1:
    → 合并到相邻任务（依赖最少的那个）
    
IF estimated_hours > 4:
    → 继续分解，直到所有子任务 ≤ 4h
```

### 3.3 粒度调整策略

| 情况 | 调整方式 |
|------|---------|
| 子任务 < 1h | 合并到前置或后置任务 |
| 子任务 1-4h | 保持不变 |
| 子任务 > 4h | LLM 重新分解，或拆分关键路径 |
| 合并后仍 < 1h | 考虑提升到父任务粒度 |

## 4. 依赖关系处理

### 4.1 依赖类型

| 类型 | 英文 | 说明 | 图示 |
|------|------|------|------|
| **结束到开始** | FS (Finish-to-Start) | A 完成后 B 才能开始 | A → B |
| **开始到开始** | SS (Start-to-Start) | A 开始后 B 才能开始 | A ═ B |
| **结束到结束** | FF (Finish-to-Finish) | A 结束后 B 才能结束 | A ↔ B |
| **开始到结束** | SF (Start-to-Finish) | A 开始后 B 才能结束（很少用） | A ← B |

### 4.2 依赖识别算法

```
输入: subtasks[]（无依赖信息）
输出: dependencies[]

1. FOR each subtask_i in subtasks:
       FOR each subtask_j in subtasks (j ≠ i):
           IF subtask_j.requires(subtask_i.output):
               dependencies.push({ from: i, to: j, type: FS })
           IF subtask_j.uses(subtask_i.resource):
               dependencies.push({ from: i, to: j, type: SS })

2. 识别可并行任务:
   parallel_groups = []
   remaining = subtasks
   WHILE remaining NOT empty:
       # 找出所有没有未完成依赖的任务
       ready = remaining.filter(t => all dependencies.to(t) are resolved)
       parallel_groups.push(ready)
       remaining.remove(ready)

3. 返回 { dependencies, parallel_groups }
```

### 4.3 依赖冲突处理

| 冲突类型 | 处理方式 |
|---------|---------|
| 循环依赖 | 报错，要求 LLM 重新分解 |
| 缺失依赖 | 自动补充 FS 依赖 |
| 冗余依赖 | 自动删除 |

### 4.4 关键路径识别

```
关键路径 = 从起点到终点的最长路径（按依赖关系）

1. 计算每个节点的最早开始时间 (EST)
2. 计算每个节点的最晚开始时间 (LST)
3. 找出总浮动时间为 0 的任务 → 关键路径
```

## 5. 并行化优化

### 5.1 并行度计算

```
max_parallelism = |parallel_groups[0]|  # 第一层可并行的任务数
avg_parallelism = Σ|group| / |parallel_groups|  # 平均并行度
```

### 5.2 并行优化策略

| 策略 | 说明 | 触发条件 |
|------|------|---------|
| **串改并** | 识别可并行的任务，主动拆分依赖 | 存在 FS 但实际可并行的任务 |
| **合并依赖** | 将连续的单任务串行合并，减少通信开销 | 并行度 > 4 |
| **拆分瓶颈** | 将关键路径上的任务拆分，提升并行度 | 关键路径长度 > 5 |

### 5.3 并行度与效率

```
假设总工作量 W，Agent 数量 N，并行度 P：

理论最短时间 = W / N（理想并行）
实际最短时间 ≈ W / P（受依赖限制）

效率 = (W / N) / (W / P) = P / N

当 P = N 时，效率 = 100%
当 P < N 时，存在 Agent 空闲
当 P > N 时，任务过多，需要更多 Agent
```

## 6. Grasp 集成

### 6.1 分解前认知增强

在 LLM 分解前，调用 Grasp 获取相关领域知识：

```
Grasp.retrieve({
  query: task.description,
  type: ["pattern", "lesson"],
  tags: task.required_capabilities,
  limit: 5
})
```

**返回示例**:

```json
{
  "items": [
    {
      "content": "大型任务分解时，先识别数据依赖再安排并行",
      "type": "lesson",
      "confidence": 0.9
    },
    {
      "content": "API 设计任务应先定义接口契约再实现",
      "type": "pattern", 
      "confidence": 0.85
    }
  ]
}
```

### 6.2 分解模式库

Grasp 可存储常见任务的分解模式，供后续复用：

| 模式名 | 适用任务 | 分解结构 |
|--------|---------|---------|
| **API开发** | 实现 REST API | 设计 → 接口 → 实现 → 测试 → 文档 |
| **前端页面** | 实现 UI 页面 | 设计 → 组件 → 集成 → 样式 → 测试 |
| **数据迁移** | 数据迁移任务 | 备份 → 验证 → 迁移 → 校验 |
| **功能开发** | 通用功能开发 | 需求 → 设计 → 实现 → 测试 → 部署 |

## 7. 执行示例

### 7.1 示例：实现用户认证系统

**输入**:

```
标题: 实现用户认证系统
描述: 实现完整的用户注册、登录、登出功能，包括JWT token和密码加密
约束: ["必须使用 bcrypt", "Token 有效期 24 小时"]
```

**分解结果**:

```
subtasks:
  1. "设计用户数据模型和 API 接口" (2h, P0)
  2. "实现用户注册 API（bcrypt 加密）" (3h, P0, depends_on: [1])
  3. "实现用户登录 API（JWT 生成）" (3h, P0, depends_on: [1])
  4. "实现用户登出 API（Token 失效）" (2h, P0, depends_on: [2, 3])
  5. "编写认证模块单元测试" (3h, P1, depends_on: [2, 3])
  6. "编写 API 集成测试" (3h, P1, depends_on: [4])

dependencies:
  - { from: 1, to: 2, type: FS }
  - { from: 1, to: 3, type: FS }
  - { from: 2, to: 4, type: FS }
  - { from: 3, to: 4, type: FS }
  - { from: 2, to: 5, type: FS }
  - { from: 3, to: 5, type: FS }
  - { from: 4, to: 6, type: FS }

can_parallel:
  [1]           # 起点
  [2, 3]        # 设计和实现可并行（注册和登录）
  [5]           # 单元测试
  [6]           # 集成测试
```

**甘特图**:

```
时间:  0-2h   2-5h   5-8h   8-11h  11-14h
任务1: [设计]
任务2:         [注册实现]
任务3:         [登录实现]
任务4:                  [登出实现]
任务5:                  [单元测试]
任务6:                           [集成测试]

关键路径: 1 → 2 → 4 → 6 (8h)
总工期: 14h（理论最优: 11h，效率: 79%)
```

## 8. 与 Reins-Skill 的关系

### 8.1 接口对应

| Reins-Skill 接口 | 本文档内容 |
|-----------------|----------|
| `task_decompose` | 调用本设计的分解算法 |
| `task_create` | 创建分解后的子任务 |
| `agent_invoke` | 调用 Agent 执行子任务 |

### 8.2 调用流程

```
Reins.task_decompose(task_id)
         │
         ├─→ 本文档：分解策略选择
         ├─→ 本文档：粒度校验（1-4h）
         ├─→ 本文档：依赖关系识别
         ├─→ 本文档：并行度优化
         ├─→ Reins.task_create (批量创建子任务)
         └─→ 返回分解结果
```
