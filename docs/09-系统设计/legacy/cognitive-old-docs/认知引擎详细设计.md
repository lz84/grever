## 1. 定位与职责边界

### 1.1 悟是什么

**悟（Grasp）** 是 Nexus 平台的认知核心，以 **Microsoft GraphRAG** 为技术底座，负责管理平台所有认知资产——包括事实知识、操作模式、经验教训、判断逻辑和元认知。

悟不是孤立的知识库，而是五兄弟协同平台中**唯一负责认知存储与认知安全的组件**。它为 Reins（御）的任务执行、Evo（进）的经验进化、Reach（触）的工具选择提供认知支撑。

### 1.2 悟做什么

- **认知存储**：以知识图谱 + 向量检索为核心，存储和管理六类认知条目
- **认知检索**：为 Agent 提供语义检索、意图理解和知识注入能力
- **认知安全**：检测认知投毒、验证知识质量、管控认知回流
- **认知回流审核**：对 Agent 产生的新经验进行分级审核，防止错误知识扩散
- **认知生命周期管理**：追踪知识溯源、管理知识版本、自动清理低质量知识

### 1.3 悟不做什么

| 边界 | 说明 |
|------|------|
| **不负责外部访问控制** | 由 Vigil（鉴）负责，悟专注于认知层面安全 |
| **不负责行为审计** | 由 Vigil（鉴）提供系统级审计，悟仅保留认知相关日志 |
| **不负责工作流执行** | 由 Reins（御）负责，悟仅提供认知支撑 |
| **不负责工具调用** | 由 Reach（触）负责，悟仅提供工具使用经验 |
| **不直接对外暴露网络 API** | 通过 Agent SDK 封装为 Skill 对内服务，遵循 MCP 协议 |

### 1.4 与 GraphRAG 的关系

| 维度 | Microsoft GraphRAG | Nexus 悟（Grasp） |
|------|-------------------|-------------------|
| 系统定位 | 独立的知识图谱 RAG 系统 | 五兄弟协同平台的一部分 |
| 认知类型 | 通用知识 | 六类认知（事实/模式/经验/判断/元认知） |
| 安全机制 | 基础权限控制 | 认知投毒检测 + 质量验证 + 回流审核 |
| 审核流程 | 无 | 完整的三阶段审核流程 |
| 工作流集成 | 独立系统 | 与 Reins 深度集成 |
| 工具调用 | 无 | 通过 Reach 调用外部工具 |
| 经验学习 | 无 | 与 Evo 协同进化 |
| 审计追踪 | 基础日志 | 完整认知溯源链条 |

## 2. 核心概念与模型

### 2.1 六类认知概念

悟将认知资产划分为六类，覆盖"是什么、怎么做、为什么/对不对、教训、元认知"五个维度：

| # | 认知类型 | 英文 | 说明 | 示例 |
|---|---------|------|------|------|
| 1 | **本体知识** | Fact | 客观事实、定义、实体关系（What） | "Nexus 平台包含五个核心 Agent：悟、御、进、触、鉴" |
| 2 | **方法论** | Methodology | 操作步骤、最佳实践（How） | "任务分解应先识别依赖关系，再安排并行任务" |
| 3 | **判断逻辑** | Logic | Why（原因解释）和 Whether（决策判断） | "为什么任务分解失败率高？"、"这段代码是否符合质量标准？" |
| 4 | **经验教训** | Lesson | 从执行中总结的成功/失败模式 | "调用第三方 API 时必须设置超时，否则会阻塞工作流" |
| 5 | **元认知** | Meta-Knowledge | 关于认知本身的认知（学习如何学习） | "这类问题应优先检索经验教训而非事实知识" |
| 6 | **行为模式** | Pattern | 可复用的思维/行为/解决方案模式 | "遇到未知错误时，先检查日志再检索相似问题" |

> **注**：行为模式（Pattern）与经验教训（Lesson）的区别在于——Pattern 是可复用的结构化模板，Lesson 是从具体经历中提炼的教训。前者偏向"范式"，后者偏向"经历"。

### 2.2 认知条目结构

```typescript
interface CognitionItem {
  cognition_id: string;        // 唯一标识，格式：cog-{uuid}
  type: CognitionType;          // 六类认知类型之一
  content: string;              // 认知内容文本
  tags: string[];               // 标签列表，用于分类检索
  confidence: number;           // 置信度 0-1
  quality_score: number;        // 质量评分 0-1
  source: SourceInfo;           // 来源信息
  status: CognitionStatus;      // 生命周期状态
  created_at: number;           // 创建时间戳
  updated_at: number;           // 更新时间戳
  version: number;              // 版本号
}

type CognitionType = 'fact' | 'methodology' | 'logic' | 'lesson' | 'meta' | 'pattern';

type CognitionStatus = 'published' | 'pending_review' | 'rejected' | 'archived';
```

### 2.3 溯源信息结构

每条认知条目携带完整溯源信息，形成可审计的认知链条：

```typescript
interface SourceInfo {
  agent_id: string;           // 来源 Agent ID
  task_id?: string;            // 关联任务 ID
  channel: SourceChannel;     // 来源渠道
  created_at: number;          // 创建时间戳
}

type SourceChannel = 'execution_feedback' | 'manual' | 'auto_learn' | 'api_import';

interface TraceInfo {
  cognition_id: string;
  versions: VersionRecord[];       // 版本变更记录
  reviews: ReviewRecord[];          // 审核记录
  quality_history: QualityRecord[]; // 质量评分历史
}
```

## 3. 核心流程

### 3.1 认知检索流程（retrieve）

Agent 通过 Grasp Skill 调用认知检索，高频使用场景包括：任务拆解前的最佳实践检索、工具选择时的经验检索、错误处理时的模式检索。

**检索流程**：

1. **格式验证**：检查查询参数合法性
2. **意图解析**：提取关键实体和意图类型
3. **向量检索**：在向量库中执行语义相似度搜索
4. **关键词检索**：执行关键词匹配作为补充
5. **结果融合**：合并向量和关键词结果，按相关度排序
6. **过滤排序**：按类型、标签、最低置信度过滤
7. **分页返回**：按 limit/offset 分页返回结果

```typescript
interface RetrieveQuery {
  query: string;                    // 查询文本（必填）
  type?: CognitionType[];           // 限定认知类型
  tags?: string[];                  // 限定标签（AND 匹配）
  min_confidence?: number;          // 最低置信度，默认 0.7
  min_quality?: number;             // 最低质量评分，默认 0.6
  limit?: number;                   // 返回数量上限，默认 10，最大 50
  offset?: number;                  // 分页偏移
}

interface RetrieveResult {
  items: CognitionItem[];
  total: number;
  has_more: boolean;
  query_time_ms: number;
  from_cache: boolean;               // 是否来自缓存
}
```

### 3.2 认知注入流程（inject）

Agent 执行任务后产生新经验，通过 inject 注入认知库，触发审核流程。

**注入流程**：

1. **格式验证**：检查内容格式、长度、编码
2. **安全检测**：调用 PoisonDetector 检测投毒风险
3. **质量评分**：调用 QualityValidator 计算初始评分
4. **分流审核**：根据评分决定自动通过或进入审核队列
5. **写入存储**：根据 status 写入对应存储
6. **返回结果**：返回认知 ID 和状态

```typescript
interface InjectQuery {
  type: CognitionType;             // 认知类型（必填）
  content: string;                  // 认知内容（必填）
  source: SourceInfo;               // 来源信息（必填）
  tags?: string[];                 // 标签列表
  confidence?: number;             // 置信度 0-1，默认 0.8
  metadata?: Record<string, any>;  // 附加元数据
}

interface InjectResult {
  cognition_id: string;
  status: 'injected' | 'pending_review' | 'rejected';
  quality_score: number;
  poison_risk?: number;             // 投毒风险评分（若检测到）
}
```

### 3.3 认知回流审核流程

**背景**：Agent 在执行任务过程中产生的经验反馈需要审核后才能回流到全局知识库。缺少审核机制可能导致错误经验扩散。

#### 3.3.1 三阶段审核流程

```
经验反馈产生
     │
     ▼
┌─────────────┐
│ 阶段1：自动初筛 │
│ 规则30%+一致性40%+来源30% │
     │
     ├── 置信度 > 80% ──→ 自动通过 / 进入人工抽检
     │
     ├── 置信度 50%-80% ──→ 进入阶段2自动复核
     │
     └── 置信度 < 50% ──→ 拒绝
```

**阶段1：自动初筛**

| 检查维度 | 权重 | 说明 |
|---------|------|------|
| 规则检查 | 30% | 格式合规、内容完整 |
| 一致性检查 | 40% | 与现有知识是否冲突 |
| 来源可信度 | 30% | 来源 Agent 历史成功率 |

**阶段2：自动复核**（仅置信度 50%-80% 的反馈）

- 交叉验证：检查是否存在反例
- 多模型一致性检查：使用多个模型评估内容一致性
- 共识度计算：多模型评估结果共识度高则提升置信度
- 置信度 > 90%：自动通过；< 90%：拒绝

**阶段3：人工审核**（置信度 50%-80% 或随机抽检）

审核员通过 Nexus Console 查看审核材料（反馈内容、来源信息、自动审核结果、相似知识、反例），做出批准或拒绝决策，并记录审核日志。

#### 3.3.2 审核规则配置

| 场景 | 规则 |
|------|------|
| 高可信来源自动通过 | 来源信任分 > 0.95 且内容长度 10-5000 字 |
| 敏感内容强制人工审核 | 包含敏感主题或个人信息的反馈 |
| 新 Agent 加强审核 | Agent 经验天数 < 30 天，阈值提高到 0.9 |
| 高置信度自动通过 | 自动初筛评分 > 0.95 |

### 4.1 整体架构

悟采用"认知底座 + 认知安全"的分层架构设计：

```
┌─────────────────────────────────────────────────────────────┐
│                    Grasp（悟）— 认知核心                     │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────────┐    │
│  │              认知安全层（Nexus 增强）                 │    │
│  │  ┌──────────────┐  ┌──────────────┐                 │    │
│  │  │ 认知投毒检测  │  │  质量验证   │                 │    │
│  │  │ PoisonDetect │  │QualityValid │                 │    │
│  │  └──────────────┘  └──────────────┘                 │    │
│  │  ┌──────────────┐  ┌──────────────┐                 │    │
│  │  │   信任评估   │  │  回流审核   │                 │    │
│  │  │TrustEvalutor │  │FeedbackRoute │                 │    │
│  │  └──────────────┘  └──────────────┘                 │    │
│  └─────────────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              意图理解层（GraphRAG 基础）               │    │
│  │     语义解析 · 意图识别 · 上下文管理                   │    │
│  └─────────────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              知识管理层（GraphRAG 基础）               │    │
│  │     知识图谱 · 向量检索 · 知识索引 · 社区检测           │    │
│  └─────────────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              认知应用层                               │    │
```

### 4.2 核心组件

| 组件 | 职责 | 技术实现 |
|------|------|---------|
| IntentParser（意图解析器） | 语义解析、意图识别 | NLP + LLM |
| KnowledgeGraph（知识图谱） | 知识图谱存储与查询 | Neo4j / NebulaGraph |
| VectorStore（向量存储） | 语义向量检索 | Milvus / Faiss |
| PoisonDetector（投毒检测器） | 认知投毒检测 | 内容分析 + 信誉评分 + 模式匹配 |
| QualityValidator（质量验证器） | 质量评分 | 多维度评分模型 |
| TrustEvaluator（信任评估器） | 来源信任评估 | 来源信誉评分 |
| FeedbackRouter（反馈路由器） | 认知回流路由 | 规则引擎 + 评分模型 |
| ReviewManager（审核管理器） | 人工审核流程 | 审核队列 + Nexus Console |

### 4.4 与五兄弟的协作

| 协作关系 | 说明 |
|---------|------|

## 5. 接口规范

### 5.1 接口设计原则

悟通过 Agent SDK 的 Grasp Skill 对内提供服务，不直接暴露外部网络 API。接口遵循 MCP（Model Context Protocol）协议规范。

| 接口 | 方法 | 说明 |
|------|------|------|
| `inject` | 认知注入 | 将新认知注入知识库，触发审核流程 |
| `retrieve` | 认知检索 | 从知识库检索匹配的认知条目 |
| `update` | 认知更新 | 更新已存在的认知条目 |

### 5.2 retrieve 接口

**方法签名**：`retrieve(query: RetrieveQuery) -> RetrieveResult`

```typescript
// 请求示例
{
  "query": "任务分解的最佳实践",
  "type": ["lesson", "pattern"],
  "tags": ["最佳实践"],
  "min_confidence": 0.7,
  "limit": 5
}

// 响应示例
{
  "items": [
    {
      "cognition_id": "cog-xxx",
      "type": "lesson",
      "content": "任务分解时，应先识别依赖关系再安排并行",
      "tags": ["任务分解", "最佳实践"],
      "confidence": 0.95,
      "quality_score": 0.92,
      "source": { "agent_id": "agent-001", "task_id": "task-123", "channel": "execution_feedback" },
      "created_at": 1744166400000,
      "updated_at": 1744166400000
    }
  ],
  "total": 1,
  "has_more": false,
  "query_time_ms": 42,
  "from_cache": true
}
```

### 5.3 inject 接口

**方法签名**：`inject(query: InjectQuery) -> InjectResult`

```typescript
// 请求示例
{
  "type": "lesson",
  "content": "调用第三方 API 时必须设置超时，否则会阻塞工作流",
  "source": { "agent_id": "agent-001", "task_id": "task-456", "channel": "execution_feedback" },
  "tags": ["API调用", "错误处理"],
  "confidence": 0.9
}

// 响应示例
{
  "cognition_id": "cog-yyy",
  "status": "pending_review",
  "quality_score": 0.85
}
```

### 5.4 update 接口

**方法签名**：`update(cognition_id: string, query: UpdateQuery) -> UpdateResult`

```typescript
// 请求示例
{
  "cognition_id": "cog-xxx",
  "content": "修正后的任务分解步骤",
  "confidence": 0.95
}

// 响应示例
{
  "cognition_id": "cog-xxx",
  "status": "updated",
  "quality_score": 0.91,
  "updated_at": 1744170000000
}
```

### 5.5 错误码规范

| 错误码 | 说明 | 是否重试 |
|--------|------|---------|
| `INVALID_CONTENT` | 内容格式无效 | 否 |
| `POISON_DETECTED` | 检测到投毒风险 | 否 |
| `LOW_QUALITY` | 质量评分低于阈值 | 否 |
| `NOT_FOUND` | 认知 ID 不存在 | 否 |
| `FORBIDDEN` | 无更新权限 | 否 |
| `STORAGE_ERROR` | 存储写入失败 | 是（3次指数退避） |
| `SERVICE_UNAVAILABLE` | Grasp 服务不可用 | 是（5次指数退避） |

## 6. 数据模型

### 6.1 核心实体

| 实体 | 核心属性 | 说明 |
|------|---------|------|
| CognitionItem（认知条目） | id, type, content, confidence, quality_score, status | 核心知识实体 |
| Concept（概念） | id, name, description, parent_id | 概念分类树 |
| Source（来源） | id, type, task_id, agent_id, user_id, timestamp | 溯源信息 |
| Quality（质量） | accuracy, freshness, consistency, coverage, citation_rate | 五维质量评分 |
| ReviewRecord（审核记录） | reviewer_id, decision, confidence, comment, timestamp | 审核轨迹 |
| VersionRecord（版本记录） | version, changed_at, changed_by, change_type | 版本变更 |

### 6.2 知识图谱结构

```
Concept（概念）
  └── CognitionItem（认知条目）
        ├── source → Source（来源）
        ├── quality → Quality（质量）
        ├── applies_to → TaskType（适用任务类型）
        └── reviewed_by → ReviewRecord（审核记录）
```

### 6.3 质量评分体系

| 维度 | 权重 | 计算方式 |
|------|------|---------|
| 准确性（Accuracy） | 40% | 使用成功率 × 用户反馈 |
| 时效性（Freshness） | 20% | 更新频率 × 有效期 |
| 一致性（Consistency） | 20% | 与现有知识冲突度 |
| 覆盖面（Coverage） | 10% | 应用场景数量 |
| 引用率（Citation） | 10% | 被其他知识引用次数 |

**来源信任度**：

| 等级 | 评分区间 | 说明 |
|------|---------|------|
| 高 | 0.8-1.0 | 历史成功率高，贡献质量好 |
| 中 | 0.5-0.8 | 有一定历史记录 |
| 低 | 0.2-0.5 | 历史记录较少或质量不稳定 |
| 黑名单 | < 0.2 | 多次提供低质量或投毒内容 |

### 6.4 知识生命周期状态

```
pending_review（待审核）
     │
     ├── 审核通过 ──→ published（已发布）
     │
     └── 审核拒绝 ──→ rejected（已拒绝）

published（已发布）
     │
     ├── 需要更新 ──→ needs_update（待更新）
     │
     ├── 质量低下 ──→ archived（已废弃）
     │
     └── 过期 ──→ archived（已废弃）

archived（已废弃）
     │
     └── 重新激活 ──→ published（已发布）
```

## 7. 安全与限制

### 7.1 认知投毒防御

**检测维度**：

| 维度 | 检测内容 | 方法 |
|------|---------|------|
| 内容一致性 | 是否与现有知识冲突 | 相似度比较 + 矛盾检测 |
| 来源可信度 | 来源是否有历史记录 | 信誉评分 + 白名单检查 |
| 模式异常 | 是否与已知投毒模式相似 | 模式匹配 + 异常检测 |
| 用户行为 | 用户是否有恶意历史 | 行为分析 + 黑名单检查 |

**检测流程**：内容分析 → 内容检查关口 → 来源验证 → 模式匹配 → 接受/拒绝/告警

### 7.2 访问限制

| 限制类型 | 限制值 | 说明 |
|---------|-------|------|
| 单次检索数量 | ≤ 50 条 | 防止大量数据泄露 |
| 内容长度 | 10-5000 字 | 防止过短垃圾或过长攻击 |
| 检索频率 | 1000 QPS/Agent | 防止高频滥用 |
| 注入频率 | 10 次/分钟/Agent | 防止批量投毒 |

### 7.3 边界约束

| 约束 | 说明 |
|------|------|
| **认知隔离原则** | 不同租户的认知数据严格隔离，不可跨租户检索 |
| **最小权限原则** | Agent 只能操作自己创建的认知条目，审核员可操作所有条目 |
| **可审计原则** | 所有认知操作均记录审计日志，不可篡改 |
| **生命周期约束** | 低质量知识（评分 < 0.3）自动归档，归档 365 天后删除 |
