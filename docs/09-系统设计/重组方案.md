# 后台代码重组方案

## 问题分析

当前 `src/` 和 `src/reins/` 存在 5 组重叠：

| 重叠项 | src/ (基础设施层) | reins/ (业务层) | 实际使用 |
|--------|-------------------|-----------------|----------|
| **grasp/** | 14 py, 214KB — Grasp 主实现 | 4 py, 40KB — 封装/降级层 | 两者都在用，但功能不同 |
| **database/** | 10 py, 106KB — SQLAlchemy DB 层 | 1 py, 3KB — 薄封装 | `reins.database` 被 93 个文件引用 |
| **eventbus/** | 9 py, 63KB — 完整 EventBus | 3 py, 34KB — 独立 EventBus | `reins.event_bus` 被 7 个文件引用，`eventbus` 被 15 个引用 |
| **data/** | 2 个 jsonl 文件 | 1 个 jsonl 文件 | 存放不同类型数据 |
| **共享模块** | auth/ common/ exception/ | 无 | auth 3 处、exception 0 处引用 |

**根因**：`reins/` 本应是纯业务逻辑包，但历史积累把自己的基础设施（DB、EventBus、Grasp 封装）也放在里面，和 `src/` 的通用基础设施重叠。

## 具体迁移步骤

### Step 1: 合并 database（影响最大，最先做）
- 将 `reins/database.py` 的功能合并到 `src/database/` 
- 创建 `src/shared/database/__init__.py` 导出统一入口
- 全局替换 `from reins.database` → `from shared.database`（93 个文件）

### Step 2: 合并 eventbus
- 将 `reins/event_bus.py`、`reins/events.py`、`reins/eventbus_integration.py` 合并到 `src/eventbus/`
- 创建 `src/shared/eventbus/__init__.py`
- 全局替换 `from reins.event_bus` → `from shared.eventbus`（7 个文件）
- 全局替换 `from eventbus` → `from shared.eventbus`（15 个文件）

### Step 3: 迁移共享模块到 shared/
```
src/auth/     → src/shared/auth/
src/common/   → src/shared/common/
src/exception/ → src/shared/exception/
src/data/     → src/shared/data/
```

### Step 4: 清理 reins/ 根级文件
将根级散落文件归类到对应子目录：
- `engine.py`, `workflow_engine.py`, `assignment.py`, `background_tasks.py`, `recovery.py` → `reins/core/`
- `context_injector.py` → `reins/scheduler/` 或 `reins/core/`
- `correction_engine.py` → `reins/services/` 或 `reins/evo/`
- `sanitize.py` → `reins/vigil/`
- `audit.py` → `reins/vigil/`
- `state_machine.py` → `src/shared/`
- `agent_sdk.py` → `src/shared/` 或独立 `agent_sdk/`
- `config.py`, `main.py` → 保留在 `reins/` 根级
- `test_*.py` → 移到 `src/tests/`
- `tracker.py`, `tracker_sync.py` → 已有 `tracking/` 目录，删除 shim 或合并
- `_legacy_models.py` → `reins/models/` 或逐步清理

### Step 5: 合并 data 目录
- `src/data/` + `reins/data/` → `src/shared/data/`
- 统一路径引用

### Step 6: 处理 grasp/ 重叠
- `src/grasp/`（14 py，主实现）保持不变 — 这是独立服务
- `reins/grasp/`（4 py，封装层）保留在 reins 内 — 这是业务层的降级/封装
- 两者职责不同，不需合并，但文档说明边界

## 影响评估

| 操作 | 影响文件数 | 风险 |
|------|-----------|------|
| database 合并 | ~93 imports | 中 |
| eventbus 合并 | ~22 imports | 中 |
| shared 目录迁移 | ~8 imports | 低 |
| reins 内部归拢 | 0（内部移动） | 低 |
| grasp 重叠处理 | 0（保持不变） | 无 |

---

## 建议执行顺序

```
1. 创建 src/shared/ 目录结构
2. 迁移 auth/common/exception 到 shared/（低风险）
3. 合并 database（最高风险，先做）
4. 合并 eventbus
5. 归拢 reins/ 根级文件
6. 清理 data/ 重叠
7. 全局更新 import
8. 编译验证 + 启动验证
```

---

## 预期效果

- **重叠消除**：5 组重叠全部消除
- **职责清晰**：shared = 基础设施，grasp = 外部服务，reins = 业务逻辑
- **import 统一**：所有基础设施从 `shared.*` 导入，业务从 `reins.*` 导入
- **reins 根级文件**：从 21 个减少到 ~3 个（`__init__.py`, `config.py`, `main.py`）
