## 1. 概述

本文档描述 Nexus 智能体协同驾驭平台的整体技术架构，作为概要设计层的基础参考。Nexus 是构建于 OpenClaw 之上的智能体协作平台，通过"五兄弟"Agent（Grasp、Reins、Evo、Reach、Vigil）的协同工作，实现复杂的任务处理和智能服务。

## 3. 模块划分与职责

Nexus 平台的核心模块及其职责如下：

### 3.1 ReinsCore（任务调度核心）

**职责**：NexusScheduler 的核心调度引擎，负责目标、项目、任务的完整生命周期管理。

| 功能 | 说明 |
|------|------|
| 目标管理 | 接收高层目标，分解为可执行任务，跟踪执行进度 |
| 项目管理 | 创建项目、管理成员、跟踪整体进度 |
| 任务管理 | 任务分配、状态追踪、流转控制、依赖管理 |
| Agent 注册 | Agent 启动时注册、注销、心跳保持、状态更新 |
| Agent 发现 | 按能力/状态查询 Agent，用于任务分配 |
| 争议管理 | Agent 间冲突的裁决和处理 |

### 3.2 SceneLibrary（场景库）

**职责**：存储和管理可复用的标准化解决方案模板（场景），并驱动任务的动态实例化。

| 功能 | 说明 |
|------|------|
| 场景定义 | 三层结构（基本信息 + 项目流程 + 任务模板） |
| 场景匹配 | 根据目标描述推荐最合适的场景模板 |
| 实例化裁剪 | 根据已知信息从全集中裁剪出初始 Goal |
| 执行时变更 | 按场景预设条件动态增减 Project/Task |
| HITL 定义 | 定义人类介入节点（信息收集/决策选择/争议仲裁） |

### 3.3 AgentRegistry（智能体管理）

**职责**：管理所有注册到 Nexus 的 Agent，提供能力注册、健康状态跟踪、负载管理。

| 功能 | 说明 |
|------|------|
| Agent 注册/注销 | 接收 Agent 的注册请求，维护在线状态 |
| 心跳处理 | 接收 Agent 心跳，更新负载和状态信息 |
| 能力同步 | 同步 Agent 的能力变更到调度引擎 |
| 负载评估 | 基于心跳数据计算 Agent 负载评分 |

### 3.4 GrASP（任务分解）

**职责**：将高层用户目标分解为可执行的 Task 序列，是 Reins 与 Grasp 的集成模块。

| 功能 | 说明 |
|------|------|
| 意图理解 | 调用 Grasp 分析用户目标，提取意图和领域上下文 |
| 任务分解 | 基于场景模板 + Grasp 认知生成任务列表 |
| 认知抽取 | 任务派发前，抽取相关认知上下文随任务下发 |
| 认知回流 | 任务完成后，将执行经验反馈给 Grasp |

### 3.5 EvoAgent（验证仲裁）

**职责**：对任务执行结果进行验证，对连续失败的任务进行裁决，并驱动 Agent 能力进化。

| 功能 | 说明 |
|------|------|
| 结果验证 | 对 Task 执行结果进行验证（编译检查/API 检查/页面检查） |
| 争议裁决 | 连续 3 次验证失败时，触发人工裁决流程 |
| 能力进化 | 根据执行结果更新 Agent 能力标签权重 |
| 经验提取 | 从成功执行中提取可复用的经验模式 |

### 3.6 HITL（人机协同）

**职责**：管理需要人类介入的节点，包括信息收集、决策选择和争议仲裁。

| 功能 | 说明 |
|------|------|
| 暂停调度 | 遇到 HITL 条件时暂停相关 Task/Project/Goal |
| 通知人类 | 推送通知到前端/飞书等渠道 |
| 响应处理 | 接收人类响应，恢复调度流程 |
| 超时处理 | 超过设定的超时时间后执行预设的默认行为 |

### 3.7 CapabilityTag（能力标签）

**职责**：为 Agent 和 Task 提供统一的能力描述语言，实现自动化的智能体匹配。

| 功能 | 说明 |
|------|------|
| 标签管理 | 四维标签（business/professional/technical/management） |
| 权重计算 | 基于执行结果动态调整标签权重 |
| 标签衰减 | technical 30 天、professional 60 天、business 90 天衰减 |
| 智能匹配 | 根据 Task 能力需求匹配最合适的 Agent |

## 4. 请求流程

以下描述一个典型用户请求从进入 Nexus 到完成的完整流程：

### 4.1 完整流程图

```
[用户输入目标]
       │
       ▼
┌──────────────────┐
│  意图理解        │  ← Grasp 分析用户目标，提取意图和领域上下文
│  Grasp           │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  场景匹配        │  ← SceneLibrary 根据目标描述推荐最合适的场景
│  SceneLibrary    │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  实例化裁剪      │  ← 根据已知信息从场景全集中裁剪出初始 Goal
│  SceneLibrary    │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  任务分解        │  ← GrASP 将 Goal 分解为可执行的 Task 列表
│  GrASP           │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  能力匹配        │  ← CapabilityTag 根据 Task 能力需求匹配最佳 Agent
│  CapabilityTag   │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  任务派发        │  ← ReinsCore 通过 A2A Hub 将 Task 派发给 Agent
│  ReinsCore       │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  Agent 执行      │  ← OpenClaw 上的 Agent 执行 Task（含认知上下文）
│  OpenClaw Agent  │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  结果验证        │  ← EvoAgent 对 Task 执行结果进行验证
│  EvoAgent        │
└──────┬───────────┘
       │
       ├─── 验证通过 ───┐
       │                 │
       ▼                 ▼
┌──────────────┐  ┌──────────────────┐
│ 认知回流      │  │  争议裁决        │
│ Grasp 记录经验│  │  HITL 人工介入   │
└──────────────┘  └────────┬─────────┘
                            │
                            ▼
                     ┌──────────────────┐
                     │  任务完成        │
                     │  继续调度下一个  │
                     └──────────────────┘
```

### 4.2 流程说明

| 阶段 | 模块 | 说明 |
|------|------|------|
| 1. 意图理解 | Grasp | 用户输入高层目标，Grasp 分析意图和领域上下文 |
| 2. 场景匹配 | SceneLibrary | 根据目标描述从场景库匹配最佳场景模板 |
| 3. 实例化裁剪 | SceneLibrary | 根据已知信息从场景全集中裁剪初始 Goal（Project + Task） |
| 4. 任务分解 | GrASP | 将 Goal 分解为具体的 Task 序列 |
| 5. 能力匹配 | CapabilityTag | 根据每个 Task 的 capability_tags 匹配最合适的 Agent |
| 6. 任务派发 | ReinsCore + A2A Hub | 通过 OpenClaw 心跳机制将 Task 派发给对应 Agent |
| 7. Agent 执行 | OpenClaw Agent | Agent 领取任务，附带认知上下文，执行任务 |
| 8. 结果验证 | EvoAgent | 验证执行结果，通过则完成，失败则重试或触发裁决 |
| 9. 认知回流 | GrASP → Grasp | 任务完成后将执行经验写回认知库 |
| 10. 动态变更 | SceneLibrary | 执行过程中按场景预设条件动态增减 Project/Task |

### 5.2 部署模式

| 场景 | 方案 | 说明 |
|------|------|------|
| 当前开发 | Windows 单机 | FastAPI + Vite dev server + SQLite，所有组件本地运行 |

| 数据类型 | 存储方案 | 说明 |

## 6. 与外部系统的集成

### 6.1 OpenClaw 集成

OpenClaw 是 Nexus 的 Agent 运行时底座，Nexus 通过其 API 进行管理：

| 集成点 | 方式 | 说明 |
|--------|------|------|
| 任务派发 | OpenClaw API | 通过 API 触发 Agent 执行任务 |
| 心跳机制 | WebSocket | Agent 定期心跳，Nexus 在心跳响应中派发任务 |
| 状态同步 | A2A Hub | Agent 间通过 A2A Hub 通信，不经 Nexus 中转 |
| 认知检索 | OpenClaw Skill | Agent 内嵌 Grasp Skill，本地检索认知 |

**关键原则**：Nexus 不修改 OpenClaw 源码，只调用其 API。

### 6.2 vLLM 集成

vLLM 提供本地大语言模型推理服务：

| 集成点 | 方式 | 说明 |
|--------|------|------|
| 意图理解 | HTTP API | Grasp 调用 vLLM 进行意图分析 |
| 任务分解 | HTTP API | GrASP 调用 vLLM 生成任务序列 |
| 方案生成 | HTTP API | EvoAgent 调用 vLLM 生成评估方案 |

### 6.3 Agent 集成

Nexus Agent 运行在 OpenClaw Gateway 之上，通过以下方式接入：

```
Nexus Agent（OpenClaw 实例）
       │
       │ 1. 注册请求（Agent ID + 能力标签）
       ▼
Nexus AgentRegistry
       │
       │ 2. 心跳保持（POST /agents/{id}/heartbeat）
       ▼
Nexus 调度器 ←──── 心跳响应中携带 pending_tasks
       │
       │ 3. 任务派发（通过 REST API）
       ▼
Agent 领取并执行任务
```

| Agent | 角色 | 模型 | 用途 |
|-------|------|------|------|
| 刚子 | CEO/指挥调度 | MiniMax-M2.7-highspeed | 协调调度、任务分配 |
| 谷子 | CFO/业务专家 | MiniMax-M2.7 | 股票交易、业务分析 |
| 麻子 | CTO/技术专家 | MiniMax-M2.7 | 代码开发、架构设计 |
| 蚊子 | CIO/内容专员 | MiniMax-M2.7 | 内容创作、文档撰写 |
| 扣子 | 开发专员 | Qwen3-Coder | 代码实现、Bug 修复 |

| 步骤 | 说明 |
|------|------|
| 注册 | Agent 启动时向 Nexus 注册，提交能力标签和状态信息 |
| 心跳 | Agent 定期发送心跳，Nexus 在响应中返回待执行任务 |
| 派发 | Nexus 通过 REST API 将 Task 派发给对应 Agent |
| 执行 | Agent 领取任务，执行后上报结果 |

### 6.4 飞书集成

Nexus 通过飞书提供人工介入通知和即时协作：

| 集成点 | 方式 | 说明 |
|------|------|------|
| HITL 通知 | 飞书消息 | 当 Task 需要人工介入时，推送通知到飞书 |
| 裁决通知 | 飞书消息 | disputed 任务产生时，推送裁决请求 |
| 状态同步 | 飞书机器人 | Nexus 关键事件同步到飞书群 |

## 7. 安全与审计

### 7.1 认证授权

| 机制 | 说明 |
|------|------|
| 认证 | JWT / OAuth2 / API Key |
| 授权 | RBAC（基于角色的访问控制） |
| 多租户 | 租户隔离、资源配额 |

### 7.2 审计日志

Vigil 贯穿所有层，记录完整的操作日志：

| 级别 | 保留时间 | 用途 |
|------|---------|------|
| 关键审计日志 | 3 年 | 合规、追责 |
| 操作日志 | 1 年 | 问题排查 |
| 访问日志 | 30 天 | 性能分析 |
| 调试日志 | 7 天 | 开发调试 |
