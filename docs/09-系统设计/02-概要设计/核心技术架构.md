## 一、架构愿景

### 设计原则

```
每个模块独立代码，互不耦合。
模块之间通过消息总线以 Command 模式通信。
```

**目标架构**：

```
                    ┌──────────────────────────────────────────────┐
                    │              消息总线 (Message Bus)            │
                    │  Command / Event 发布-订阅                     │
                    └──┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬──┘
                       │   │   │   │   │   │   │   │   │   │   │
              ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐
              │日志│ │DAG │ │执行│ │Agent│ │匹配│ │场景│ │验证│ │HITL│
              │引擎│ │定义│ │引擎│ │接入│ │引擎│ │引擎│ │引擎│ │引擎│
              └────┘ └────┘ └────┘ └────┘ └────┘ └────┘ └────┘ └────┘
```

**当前架构**（Sprint 98 实际状态）：

```
                    ┌──────────────────────────────────────┐
                    │         server.py (FastAPI)           │
                    │         include_router × 160+         │
                    └──────────┬───────────────────────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
         reins/api/      reins/scheduler/  reins/persistence/
        (~160 路由文件)      (~15 调度文件)     (~25 ORM/迁移)
              │                │                │
         API 层             调度核心            数据层
         高度碎片化           相对集中           相对集中
```

**差距**：当前是"扁平路由 + 集中调度"的单体架构，目标是"独立模块 + 消息总线"的微内核架构。

## 二、核心引擎模块

### 2.1 日志引擎（Log Engine）

**现状**：❌ 分散在各处，没有统一的日志模块。

| 日志类型 | 当前存储位置 | 问题 |
|---------|------------|------|
| 系统操作日志 | `execution_logs` 表 | 只有 Task 级别的执行记录 |
| Agent 通讯日志 | 无独立存储 | 心跳数据在 `agents.last_heartbeat` |
| 执行成功/失败日志 | `task_comments` 表 + `result_verifier` | 验证结果写 comment，非结构化 |
| 调度日志 | `scheduler_logs` 表 | 有但只有调度动作（assign/recover/unlock） |
| 审计日志 | `security.py` + `audit.py` | 有基础实现但未接入所有模块 |
| LLM 调用日志 | 无 | 完全没有记录 |

**目标设计**：

```
┌────────────────────────────────────────────────────────────┐
│                    日志引擎 (Log Engine)                     │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐     │
│  │ 采集层        │  │ 存储层        │  │ 查询层        │     │
│  │              │  │              │  │              │     │
│  │ - 操作日志    │  │ - SQLite     │  │ - 按时间     │     │
│  │ - 通讯日志    │  │ - 可选 ES    │  │ - 按模块     │     │
│  │ - 执行日志    │  │              │  │ - 按 Agent   │     │
│  │ - 审计日志    │  │              │  │ - 按 traceId │     │
│  │ - LLM 日志   │  │              │  │              │     │
│  └──────────────┘  └──────────────┘  └──────────────┘     │
│                                                            │
│  统一接口：                                                  │
│  LogEngine.emit(module, event_type, payload, trace_id)     │
│  LogEngine.query(filters) → [LogEntry]                     │
└────────────────────────────────────────────────────────────┘
```

**统一日志 Schema**：

```json
{
  "id": "log-uuid",
  "timestamp": "2026-05-27T23:00:00Z",
  "trace_id": "trace-uuid",           // 跨模块追踪
  "module": "scheduler|agent|execution|matching|...",
  "event_type": "task_assigned|agent_heartbeat|verification_passed|...",
  "level": "info|warn|error",
  "agent_id": "guzi",                  // 可选
  "task_id": "task-xxx",               // 可选
  "payload": { ... },                  // 结构化数据
  "metadata": {
    "duration_ms": 150,
    "llm_tokens": 2048,                // LLM 调用专属
    "memory_mb": 128                    // 资源使用
  }
}
```

**实现优先级**：
1. **P0**：统一采集接口 + SQLite 存储 + 基础查询
2. **P1**：跨模块 trace_id 透传（所有 Command/Event 携带）
3. **P2**：LLM 调用日志（token 消耗、延迟、模型选择）
4. **P3**：可选 ES 后端（大规模部署时）

### 2.5 基于标签的任务匹配引擎（Task Matching）

**现状**：✅ 可用，四维标签 + 行业标签 + 权重。

| 现有实现 | 位置 | 状态 |
|---------|------|------|
| 匹配引擎 | `reins/scheduler/task_assigner.py` (19KB) | ✅ 核心 |
| 自动标注 | `reins/scheduler/auto_tagging.py` | ✅ 可用 |
| 标签前置检测 | `reins/scheduler/tag_prerequisites.py` (9KB) | ✅ 可用 |
| 任务校验器 | `reins/scheduler/task_validator.py` (2KB) | ✅ 可用 |

**匹配流程**：

```
Task (capability_tags + requires)
    │
    ▼
┌─────────────────────────────────────┐
│          匹配引擎                      │
│                                     │
│  1. 过滤：排除离线/繁忙 Agent          │
│  2. 能力匹配：四维标签交集打分          │
│  3. 前置检查：prerequisites 满足？     │
│  4. 负载评估：当前负载 < 阈值？         │
│  5. 权重衰减：技术 30 天/专业 60 天    │
│  6. 排序：按综合得分排序               │
│                                     │
│  输出：最佳 Agent + 备选列表            │
└─────────────────────────────────────┘
```

**当前打分逻辑**：

```python
score = (
    capability_match_score * 0.4 +    # 能力匹配度
    load_score * 0.3 +                 # 负载情况（越低越好）
    tag_weight_score * 0.2 +           # 标签权重（衰减后）
    reliability_score * 0.1            # 可靠性（成功率）
)
```

**需要补充的点**：
1. **匹配策略可配置**：目前权重是硬编码的，应该支持按行业/场景配置
2. **历史表现反馈**：Agent 在该类任务上的历史成功率应该影响匹配
3. **多 Agent 协作匹配**：目前只匹配单个 Agent，复杂任务需要匹配团队

## 三、消息总线（Message Bus）

**现状**：⚠️ 有 Event Bus，但不是 Command 模式。

| 现有实现 | 位置 | 状态 |
|---------|------|------|
| Event Bus Manager | `reins/eventbus/manager.py` | ✅ 可用 |
| SSE 适配器 | `reins/eventbus/sse_adapter.py` | ✅ 可用 |
| Polling 适配器 | `reins/eventbus/polling_adapter.py` | ✅ 可用 |
| 事件定义 | `reins/events.py` (8.9KB) | ✅ 可用 |
| EventBus 集成 | `reins/eventbus_integration.py` | ✅ 可用 |

**当前 Event Bus 是"发布-订阅"模式，不是"Command 模式"**：

```
当前：
  Publisher ──publish──→ Event Bus ──subscribe──→ Subscriber
  (单向广播)

目标（Command 模式）：
  Sender ──dispatch(Command)──→ Command Bus ──handle──→ Handler
  (命令 → 执行 → 响应)

还需要：
  Sender ──publish(Event)──→ Event Bus ──notify──→ Listener
  (事件 → 通知)
```

**Command 模式设计**：

```
┌────────────────────────────────────────────────────────────┐
│                    消息总线 (Message Bus)                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Command Bus（命令总线）                                     │
│  ├── Dispatch(Command) → Handler                           │
│  ├── 同步执行，返回 Result                                  │
│  ├── 示例：AssignTaskCommand, VerifyResultCommand          │
│  └── 支持事务：Command 失败可回滚                            │
│                                                            │
│  Event Bus（事件总线）← 已有                                │
│  ├── Publish(Event) → Notify(Listeners)                    │
│  ├── 异步通知，无返回值                                     │
│  ├── 示例：TaskAssignedEvent, TaskCompletedEvent           │
│  └── 支持多订阅者                                           │
│                                                            │
│  统一接口：                                                  │
│  Bus.dispatch(command) → Result                            │
│  Bus.publish(event) → void                                 │
│  Bus.subscribe(event_type, handler)                         │
└────────────────────────────────────────────────────────────┘
```

**核心 Command 列表**：

| Command | Handler | 说明 |
|---------|---------|------|
| `AssignTaskCommand` | `TaskAssigner` | 派发任务给 Agent |
| `VerifyResultCommand` | `ResultVerifier` | 验证任务结果 |
| `InstantiateScenarioCommand` | `ScenarioInstantiator` | 实例化场景 |
| `DecomposeGoalCommand` | `GoalDecomposer` | 分解目标 |
| `TriggerHITLCommand` | `HITLHandler` | 触发人机协同 |
| `RulingCommand` | `RulingHandler` | 人工裁决 |

**核心 Event 列表**：

| Event | Listener | 说明 |
|-------|---------|------|
| `TaskAssignedEvent` | LogEngine, AgentSDK | 任务已派发 |
| `TaskCompletedEvent` | Verifier, ProjectExecutor | 任务已完成 |
| `TaskFailedEvent` | TaskRecoverer, LogEngine | 任务失败 |
| `AgentHeartbeatEvent` | HealthManager, LoadCalculator | Agent 心跳 |
| `ScenarioInstantiatedEvent` | LogEngine, ScenarioLibrary | 场景已实例化 |

### 8.2 各模块推荐方案

#### 日志引擎：Loguru

```python
from loguru import logger

logger.add(
    "logs/nexus_{time:YYYY-MM-DD}.log",
    rotation="50 MB",
    retention="30 days",
    format="{time} | {level} | {name} | {message} | {extra[trace_id]}"
)

logger.bind(trace_id="abc-123").info("Task assigned", task_id="task-001", agent_id="guzi")
```

- **优势**：零配置，5 行代码接入，异步写入，自动轮转
- **生产升级**：Fluent Bit（10MB 内存占用）→ Loki（比 ES 轻量 10 倍）

#### DAG 定义：NetworkX + Pydantic

```python
import networkx as nx

# 环路检测
try:
    cycle = nx.find_cycle(G)
except nx.NetworkXNoCycle:
    pass  # DAG 合法

# 拓扑排序 + 并行组
order = list(nx.topological_sort(G))
parallel_groups = list(nx.topological_generations(G))
```

- **不建议用 Airflow/Prefect/Temporal**：太重，面向数据工程/ETL，不是面向 Agent 任务

#### 执行引擎：transitions + 自研

```python
from transitions import Machine

class Task:
    states = ['todo', 'in_progress', 'verifying', 'review_needed',
              'disputed', 'done', 'failed', 'timeout', 'paused']
    transitions = [
        {'trigger': 'assign', 'source': 'todo', 'dest': 'in_progress'},
        {'trigger': 'complete', 'source': 'in_progress', 'dest': 'verifying'},
        {'trigger': 'pass_verification', 'source': 'verifying', 'dest': 'done'},
        {'trigger': 'dispute', 'source': 'review_needed', 'dest': 'disputed'},
    ]
```

- **生产升级**：当需要分布式执行时，引入 Dramatiq（轻量）或 Temporal（可靠但重）

#### Agent 接入：完全自研

```python
class AgentAdapter(ABC):
    @abstractmethod
    async def send_task(self, agent_id: str, task: Task) -> str: ...
    @abstractmethod
    async def get_result(self, agent_id: str, execution_id: str) -> TaskResult: ...
    @abstractmethod
    async def heartbeat(self, agent_id: str) -> AgentStatus: ...

class OpenClawAdapter(AgentAdapter): ...   # 当前
class LangChainAdapter(AgentAdapter): ...  # 未来
class MCPAdapter(AgentAdapter): ...        # 未来

class AdapterRegistry:
    def register(self, platform: str, adapter: AgentAdapter): ...
    def get(self, agent_id: str) -> AgentAdapter: ...
```

- **无现成方案**：市面上没有"多平台 Agent 接入管理器"
- **参考**：LangChain/CrewAI/AutoGen 可作为适配器实现的参考

#### 任务匹配：自研 + scikit-learn 辅助

```python
from sklearn.metrics.pairwise import cosine_similarity

# 向量空间匹配
similarities = cosine_similarity([task_vector], agent_vectors)[0]
scores = similarities * 0.5 + load_scores * 0.3 + reliability_scores * 0.2
```

- **核心自研**：四维标签 + 权重衰减 + 负载评估是 Nexus 差异化
- **辅助**：scikit-learn 提升匹配精度
- **未来**：Agent 数量 > 100 时引入 FAISS（Meta 向量检索库）

#### 消息总线：Redis Pub/Sub + 自研 Command Bus

```python
# Command Bus（不需要额外服务）
class CommandBus:
    def register(self, command_type: str, handler: Callable): ...
    async def dispatch(self, command: Command) -> Result: ...

# Event Bus（基于 Redis）
import redis.asyncio as redis
class EventBus:
    async def publish(self, event: Event): ...
    async def subscribe(self, event_type: str, handler: Callable): ...
```

- **Command Bus**：纯代码封装，基于函数注册表，不需要额外服务
- **Event Bus**：基于 Redis Pub/Sub，轻量，5 分钟部署
