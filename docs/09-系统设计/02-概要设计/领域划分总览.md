## 一、架构分层

Nexus 平台采用 **领域（纵向）+ 技术层（横向）** 的二维架构。

```
                    ┌─────────────────────────────────────────┐
                    │           技术层（跨所有领域）              │
                    │                                         │
                    │  NexusAPI (API 网关 / 路由 / 认证)        │
                    │  SQLite (reins.db) 统一持久化             │
                    │  日志 / 告警 / 配置管理                   │
                    └─────────────┬───────────────────────────┘
                                  │
        ┌─────────────┬───────────┼───────────┬─────────────┐
        ▼             ▼           ▼           ▼             ▼
    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
    │ 认知域 │    │ 驾驭域 │    │ 安全域 │    │ 进化域 │    │ 拓展域 │
    │GrASP │    │Reins │    │Vigil │    │ Evo  │    │Reach │
    └──────┘    └──────┘    └──────┘    └──────┘    └──────┘
```

**设计原则**：
- 五大领域是**业务域**，各自拥有独立的领域模型
- 技术层是**基础设施**，支撑所有领域，不归属任何域
- 领域之间的交互通过技术层的 API 网关路由

## 二、五大领域

### 2.1 认知域（GrASP）

| 属性 | 说明 |
|------|------|
| **一句话定位** | 认知底座，让 Agent"懂" |
| **核心职责** | 知识存储、认知推理、意图理解、认知安全 |
| **类比** | 团队的"知识库 + 智囊团" |
| **对标** | Palantir 的认知图谱 |

**能力清单**：

| 能力 | 说明 |
|------|------|
| 认知存储 | What/How/Why/Lessons/Meta 五类知识 |
| 认知推理 | 基于知识图谱的上下文推导 |
| 意图理解 | 分析用户目标，提取领域上下文 |
| 认知抽取 | 任务派发前抽取相关上下文随任务下发 |
| 认知回流 | 任务完成后将经验反馈给知识库 |
| 认知安全 | 认知投毒检测、知识质量验证 |

**与驾驭域的关系**：
- GrASP 为驾驭域的 Agent 提供行动依据（认知上下文）
- GrASP 不参与任务调度、不管理场景、不直接控制执行
- GrASP 是**被引用的知识库**，不是主动调度的引擎

| **一句话定位** | 协同引擎，让 Agent"动" |
| **核心职责** | 从目标到任务的全生命周期管理、调度执行、人机协同 |
| **类比** | 团队的"项目经理 + 调度中心" |
| **对标** | Jira + Airflow 的组合 |

**子领域**（5 个）：

| # | 子领域 | 职责 | 对应文档 |
|---|--------|------|---------|
| 1 | 任务调度 | Goal → Project → Task 生命周期、DAG 调度、派发 | [任务调度详细设计](../03-详细设计/驾驭领域/任务调度详细设计.md) |
| 2 | 智能体注册 | Agent 注册、心跳、在线状态、负载管理 | [智能体注册详细设计](../03-详细设计/驾驭领域/智能体注册详细设计.md) |
| 3 | 人机协同 | HITL 三种交互模式、审批、通知、权限 | [人机协同详细设计](../03-详细设计/驾驭领域/人机协同详细设计.md) |
| 4 | 验证与仲裁 | 结果验证、争议裁决、方案对比 | [验证与仲裁详细设计](../03-详细设计/驾驭领域/验证与仲裁详细设计.md) |
| 5 | 能力标签 | 四维标签体系、匹配引擎、权重管理 | [能力标签详细设计](../03-详细设计/驾驭领域/能力标签详细设计.md) |

**驾驭域内部交互**：

```
         ┌──────────────┐
         │  能力标签     │  ← 提供匹配结果
         │ (匹配引擎)    │
         └──────┬───────┘
                │ 提供匹配结果
                ▼
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 智能体注册│◄─│  任务调度 │──►│ 人机协同  │
│(Agent信息)│  │ (调度核心)│  │(拦截/审批)│
└──────────┘  └──────┬───┘  └──────────┘
   负载/在线           │ 执行结果
   状态                ▼
              ┌──────────────┐
              │ 验证与仲裁    │
              │ (验证/裁决)   │
              └──────────────┘
```

### 2.3 安全域（Vigil）

| 属性 | 说明 |
|------|------|
| **一句话定位** | 安全卫士，让系统"稳" |
| **核心职责** | 信任评估、行为审计、异常检测、访问控制 |
| **类比** | 团队的"安全官 + 审计员" |
| **对标** | SIEM 系统 + Open Policy Agent |

**四层结构**：

| 层 | 职责 | 核心能力 |
|---|------|---------|
| **外审层** | 对外安全 | RBAC 权限管理、外部访问控制、第三方鉴权 |
| **内审层** | 对内安全 | 行为审计、异常检测、⭐ **信任评估体系** |
| **日志管理层** | 日志管理 | 日志收集、存储、查询、归档 |
| **告警管理层** | 告警管理 | 威胁告警、合规告警、性能告警 |

**信任评估体系（Vigil 核心）**：

三维评分模型（能力 40% + 可靠性 30% + 安全 30%）→ 信任等级 → 动态权限。

> **实现状态**：当前代码实现的是**可用性部分**的 5 因子工程模型（Phase 1），对应能力+可靠性维度。安全维度（Phase 2）待 `vigil/access/` 和审计模块就绪后接入。详见 [`Vigil架构.md`](../03-详细设计/安全领域/Vigil架构.md) §3.2.3.0。

| 维度 | 核心指标 |
|------|---------|
| 能力 | 任务完成率(50%) + 质量分(30%) + 复杂度适应(20%) |
| 可靠性 | 稳定性(40%) + 异常率(30%) + 响应时间(20%) + 恢复能力(10%) |
| 安全 | 越权次数(40%) + 脱敏合规(30%) + 安全事件(20%) + 审计通过率(10%) |

**与驾驭域的关系**：
- Vigil 监控驾驭域的执行过程，记录审计日志
- Vigil 的信任评估影响驾驭域的任务分配（低信任 Agent 不分配敏感任务）
- Vigil 的认知投毒检测已迁移到 GrASP

### 2.4 进化域（Evo）

| 属性 | 说明 |
|------|------|
| **一句话定位** | 进化引擎，让 Agent"成长" |
| **核心职责** | 从执行结果中学习、能力进化、Agent 间通信（A2A） |
| **类比** | 团队的"复盘机制 + 经验传承" |
| **对标** | 持续学习系统 |

**能力清单**：

| 能力 | 说明 |
|------|------|
| 结果学习 | 分析任务执行结果，识别成功/失败模式 |
| 能力进化 | 根据执行结果更新 Agent 能力标签权重 |
| 经验提取 | 从成功执行中提取可复用的经验模式 |
| A2A 通信 | Agent 间传递经验、信念、模式的通信协议 |

**与驾驭域的关系**：
- 驾驭域的验证仲裁判断"对错"，Evo 从"对错"中学习
- Evo 学习结果影响驾驭域的能力标签权重 → 影响下次匹配
- A2A 是 Evo 领域的通信子组件，不是独立模块

---

### 2.5 拓展域（Reach）

| 属性 | 说明 |
|------|------|
| **一句话定位** | 解决方案中心，让能力"复用" |
| **核心职责** | 场景模板管理、解决方案沉淀、行业包交付 |
| **类比** | 团队的"模板库 + 最佳实践库" |
| **对标** | Terraform/Ansible 的配置模板库 |

**子领域**（3 个）：

| # | 子领域 | 职责 | 对应文档 |
|---|--------|------|---------|
| 1 | 场景库 | 场景模板存储、匹配、实例化 | [场景库详细设计](../03-详细设计/拓展领域/场景库详细设计.md) |
| 2 | 行业包 | 行业标准包、定制包、生命周期 | [行业包详细设计](../03-详细设计/拓展领域/行业包详细设计.md) |
| 3 | 系统扩展 | MCP 服务器、Skills、附件、产出物 | [系统扩展接口](../04-接口契约/系统扩展接口.md) |

**与驾驭域的关系**：
- 拓展域提供场景模板 → 驾驭域实例化为 Project/Task
- 场景来源：GrASP 认知总结（一种来源）+ 用户自定义（另一种来源）
- 场景实例化是驾驭域的动作，不是拓展域的动作

## 三、技术层（跨域）

### 3.1 NexusAPI（API 网关）

| 属性 | 说明 |
|------|------|
| **定位** | 所有领域的统一入口，技术基础设施 |
| **职责** | 路由转发、认证中间件、限流策略 |

| **不归属任何业务域** | 是横向切面，不是纵向切片 |

### 3.2 SQLite（reins.db）

| 属性 | 说明 |
|------|------|
| **定位** | 统一数据持久化 |
| **路径** | `D:\work\research\agents-nexus\data\reins.db` |
| **原则** | DB 是唯一真实数据源，禁止双数据源 |

| 属性 | 说明 |
|------|------|
| **定位** | 系统运行日志、审计日志、告警管理 |

## 五、五兄弟一句话总结

| 代号 | 领域 | 一句话 | 层 |
|------|------|--------|---|
| **GrASP** | 认知 | "认知底座，让 Agent 懂" | 知识层 |
| **Reins** | 驾驭 | "协同引擎，让 Agent 动" | 执行层 |
| **Vigil** | 安全 | "安全卫士，让系统稳" | 安全层 |
| **Evo** | 进化 | "进化引擎，让 Agent 成长" | 成长层 |
| **Reach** | 拓展 | "解决方案中心，让能力复用" | 价值层 |

**协作流程**（用户视角）：

```
用户输入目标 → GrASP 注入认知 → Reach 匹配场景 → Reins 实例化并调度
  → Agent 执行 → Vigil 监控审计 → Evo 学习进化 → 结果反馈给 GrASP
```
