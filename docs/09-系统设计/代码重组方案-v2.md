# Nexus 后台代码重组方案 v2

## 设计原则

1. **src/ 根目录 = 5 大领域 + 公共服务**
2. **每个领域 = 自己的 common/ + 子领域目录**
3. **公共服务 = 所有领域共享的基础设施**
4. **API/模型/持久化 = 跨领域基础设施，放在 shared/**

## 迁移步骤（按风险从低到高）

### Phase 1: 创建共享公共服务（低风险）
```
src/shared/database/     ← src/database/
src/shared/eventbus/     ← src/eventbus/
src/shared/auth/         ← src/auth/
src/shared/common/       ← src/common/
src/shared/exception/    ← src/exception/
```
- 更新 import：`from database.*` → `from shared.database.*`
- 更新 import：`from eventbus.*` → `from shared.eventbus.*`
- 更新 import：`from auth.*` → `from shared.auth.*`
- 更新 import：`from common.*` → `from shared.common.*`
- 更新 import：`from exception.*` → `from shared.exception.*`

### Phase 2: 创建 5 大领域目录（中风险）

#### 2a. cognitive/ 认知域
```
src/cognitive/    ← src/grasp/
```
- `reins/grasp/`（caller/circuit_breaker/fallback）→ `src/steering/common/grasp_client/`
- 更新 import：`from grasp.*` → `from cognitive.*`

#### 2b. evolution/ 进化域
```
src/evolution/evo/          ← reins/evo/
src/evolution/common/       ← reins/correction_engine.py
```

#### 2c. extension/ 拓展域
```
src/extension/scenarios/    ← reins/scenarios/
src/extension/industry/     ← reins/industry/
src/extension/common/       ← reins/agent_sdk.py
```

#### 2d. security/ 安全域
```
src/security/vigil/         ← reins/vigil/
src/security/common/        ← reins/audit.py + reins/sanitize.py
```

#### 2e. steering/ 驾驭域（最大改动）
```
src/steering/common/        ← reins/__init__.py 部分功能 + config.py + logging/
src/steering/core/          ← reins/engine.py + workflow_engine.py + assignment.py + ...
src/steering/scheduler/     ← reins/scheduler/
src/steering/tracking/      ← reins/tracking/ + tracker.py + tracker_sync.py
src/steering/messaging/     ← reins/messaging/
src/steering/nexus_log/     ← reins/nexus_log/
src/steering/api/           ← reins/api/（164 个文件）
```

### Phase 3: 跨领域基础设施
```
src/models/                 ← reins/models/（保持不变，提升一级）
src/persistence/            ← reins/persistence/（保持不变，提升一级）
src/services/               ← reins/services/（保持不变，提升一级）
src/api/                    ← reins/api/（提升一级）
src/migration/              ← reins/migration/
```

### Phase 4: 清理
- 删除空的 `reins/data/`、`src/data/`、`src/memory/`、`reins/state/`
- 合并 `src/data/` + `reins/data/` → `src/shared/data/`
- 删除 `reins/` 目录（所有文件已迁移）
- 删除散落测试文件 → 移入 `src/tests/`
- 删除 `fix_startup.py` 等一次性脚本

## 关键决策点

### 1. api/ 放哪里？
- **方案 A**：放在 `steering/api/` — 但 api/ 不只是 steering 的 API
- **方案 B**：放在 `src/api/` 顶层 — 作为跨领域网关（推荐）

### 2. models/ 和 persistence/ 放哪里？
- 它们是跨领域基础设施，不属于任何一个领域
- 建议放在 `src/models/` 和 `src/persistence/` 顶层

### 3. services/ 放哪里？
- services/ 中的 goal_decomposition 调用 cognitive
- task_validator 调用 evolution
- feishu_notification 是通用的
- 建议保持 `src/services/` 顶层

### 4. reins/grasp/ 和 src/grasp/ 的关系
- `src/grasp/` = Grasp 认知服务的完整实现
- `reins/grasp/` = 业务层对 Grasp 的封装（caller/circuit_breaker/fallback）
- 建议：后者移入 `steering/common/grasp_client/`
