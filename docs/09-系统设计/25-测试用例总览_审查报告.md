## 三、逻辑错误（5 项）

### 3.1 TC-V-010：信任评分公式与实际实现不符

- **位置**: 4.2 信任评估 - TC-V-010
- **问题描述**: 测试用例描述的权重为"能力(40%) + 可靠性(30%) + 安全(30%)"，但 `vigil/trust/trust.py` 实际实现为：
  ```
  WEIGHT_SUCCESS_RATE = 0.35
  WEIGHT_CONSISTENCY = 0.20
  WEIGHT_TIMELINESS = 0.20
  WEIGHT_QUALITY = 0.15
  WEIGHT_LONGEVITY = 0.10
  ```
- **建议**: 要么修正测试用例以匹配实际实现，要么修正代码实现以匹配设计文档。建议设计文档优先，保持代码与文档一致。

### 3.2 TC-V-011：动态权限调整的权限限制描述不准确

- **位置**: 4.2 信任评估 - TC-V-011
- **问题描述**: 用例描述"限制敏感任务分配"，但 `access.py` 中的 `DynamicPermission` 并未与 trust 系统集成。RBAC 权限矩阵是静态定义的（`ROLE_PERMISSIONS`），动态权限调整需要额外实现。
- **建议**: 明确动态权限调整的实现边界，或补充 TC-V-011b 作为待实现功能的前置说明。

### 3.3 TC-R-040/041/042：验证结果中的字段名与实际代码不符

- **位置**: 1.5 验证与仲裁 - TC-R-040/041/042
- **问题描述**: 测试用例期望输出 `{"verified": true, "checks_passed": N, "checks_total": N}`，但 `rules.py` 中 `run_verifier_checks` 返回的是 `(bool, str)` 元组（passed, message），且返回消息格式不固定。
- **建议**: 明确 API 端点返回的 JSON 结构定义，确保 `verification/rules.py` 的返回结果与 API 路由层的响应模型一致。

### 3.4 TC-G-031：switch_backend 并发安全测试描述过于简化

- **位置**: 2.4 适配器层 - TC-G-031
- **问题描述**: 用例仅描述"多线程同时调用 switch_backend"，未说明：
  1. 如何模拟并发（threading/asyncio）
  2. 具体的并发数量
  3. 如何验证"无中间态"（断言什么？）
- **建议**: 补充详细的测试步骤和断言条件。

### 3.5 TC-R-032：Agent 离线检测后的任务重新分配逻辑

- **位置**: 1.4 智能体注册与发现 - TC-R-032
- **问题描述**: 用例描述"任务重新排队"，但未说明重新分配的目标（重新派发给其他 Agent？还是仅改变状态等待调度？）实际代码 `agent_registry.py` 中离线处理逻辑与测试描述可能不一致。
- **建议**: 明确离线后任务的状态变更路径。

- **位置**: 1.5 验证与仲裁 - TC-R-040~044
