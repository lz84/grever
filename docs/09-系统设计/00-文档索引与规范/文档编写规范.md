# 文档编写规范

> 所有 Nexus 系统设计文档的编写标准。所有文档必须遵循本规范。

---

## 一、格式规范

### 1.1 文件命名
- 纯中文文件名，不含英文和数字序号前缀
- 示例：`需求总览.md`、`系统架构总览.md`
- 禁止：`01-需求总览.md`、`requirements-overview.md`

### 1.2 标题层级
- 一级标题（`#`）：文档标题
- 二级标题（`##`）：章节
- 三级标题（`###`）：小节
- 最多到三级，避免更深嵌套

### 1.3 强制元数据块
每份文档头部必须包含：

```markdown
# 文档标题

**版本**: v1.0  
**作者**: [姓名]  
**日期**: 2026-05-27  
**状态**: 草稿 | 评审中 | 正式版  
**关联需求**: [需求编号]  
**变更记录**:  
- v1.0 [日期] [作者] [变更内容]
```

### 1.4 表格格式
使用 Markdown 标准表格，对齐清晰：

```markdown
| 字段 | 类型 | 说明 | 必填 |
|------|------|------|------|
| name | TEXT | 名称 | 是 |
```

### 1.5 代码块
- 所有代码示例必须标注语言：` ```python ` 或 ` ```sql `
- SQL 语句使用大写关键字：`SELECT`、`INSERT`
- Python 代码遵循 PEP 8

## 二、内容规范

### 2.1 引用关系
- 引用其他文档使用相对路径链接：`[模块设计](../03-详细设计/任务调度引擎详细设计.md)`
- 引用代码文件使用实际路径：`packages/server/src/reins/api/tasks_crud.py`

### 2.2 交叉引用标记
在文档中使用锚点标记，便于跳转：

```markdown
<span id="tag-lifecycle-state">标签生命周期状态</span>

引用：[\[tag-lifecycle-state\]] 章节定义了...
```

### 2.3 禁止事项
- 禁止"待定"、"TODO"等无意义占位（除非明确标注为"待定需求"）
- 禁止在设计文档中使用"可能"、"大概"等模糊表述
- 禁止截断的代码片段（粘贴的代码必须完整可运行）

## 三、各层文档编写标准

### 3.1 需求分析文档

必须包含：
1. **业务背景**：解决这个问题什么，不解决的代价是什么
2. **用户故事**：谁（角色）、在什么场景、用这个功能、得到什么结果
3. **功能需求列表**：用表格列出每条需求 + 验收标准
4. **非功能需求**：性能、安全、可用性
5. **依赖关系**：依赖哪些其他模块或外部系统

格式：
```markdown
## [UC-XXX] 用户故事标题

**作为** [角色]  
**我希望** [功能]  
**以便** [价值]

### 验收标准

| 条件 | 输入 | 预期输出 |
|------|------|----------|
| 正常 | ... | ... |
| 异常 | ... | ... |
```

### 3.2 概要设计文档

必须包含：
1. **设计决策**：这个模块做了什么选择，为什么不选其他方案
2. **模块划分**：模块职责、数据流向（用 mermaid 或文字描述）
3. **接口清单**：本模块提供和消费哪些接口
4. **关键数据结构**：核心实体的字段定义

### 3.3 详细设计文档

必须包含：
1. **类/函数清单**：每个公共函数签名 + 说明
2. **流程图**：关键业务流程（用 mermaid 或文字）
3. **状态机**：实体状态转换表/图
4. **错误处理**：每类错误的处理策略
5. **边界情况**：空值、并发、超时的处理

### 3.4 接口契约文档

必须包含：
1. **接口清单总表**
2. **每个接口的完整签名**（方法 + 路径 + 请求体 + 响应）
3. **错误码定义**：每种 HTTP 错误码 + 业务错误码

## 四、评审与发布流程

```
草稿 → 内部评审（作者邀请） → 修改 → 正式发布 → 变更记录
```

- 文档发布后禁止直接删除内容，只允许新增"变更记录"
- 所有文档变更必须更新版本号和变更记录
