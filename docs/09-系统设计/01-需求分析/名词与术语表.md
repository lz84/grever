## 战略目标与战术层级术语

### 1.1 Goal（战略目标）

**定义**：用户向 Nexus 输入的最高层任务描述，是整个工作流的起点。

**特征**：
- 描述业务价值而非实现细节
- 由运营人员或外部 API 用户发起
- 经过场景模板匹配后，自动分解为 Projects

**示例**：
```
Goal: "化工罐区发生液氯泄漏，需要启动应急响应"
```

**关联**：
- Goal 分解为 N 个 Project
- Goal 可以选择关联某个 Scenario Template 进行结构化分解

---

### 1.2 Project（战略）

**定义**：Goal 的直接下级执行单元，对应一个业务场景的完整执行计划。

**特征**：
- 归属于某个 Goal
- 包含一个场景模板（Scenario Template）的实例化结果
- 包含 N 个 Task
- 有一个负责人（运营人员或 AI Agent）

**示例**：
```
Project: "液氯泄漏应急响应"
  ├── Task: "识别危化品类型"
  ├── Task: "评估泄漏等级"
  ├── Task: "计算扩散范围"
  └── Task: "制定疏散方案"
```

**关联**：
- Project 分解为 N 个 Task
- Project 对应一个 Scenario Instance（场景实例）

---

### 1.3 Task（战术）

**定义**：Project 的最小执行单元，是派发给 Agent 的实际工作项。

**特征**：
- 不可再分的基本执行单元
- 有明确的输入/输出定义
- 可以声明所需的能力标签（`capability_tags`）
- 可以配置 `condition_type`（执行模式）
- 有状态机：todo / running / waiting_human / done / failed

**示例**：
```
Task: "评估泄漏等级"
  ├── 输入: ["泄漏物质类型", "泄漏量估计", "气象条件"]
  ├── 输出: "响应等级（Ⅰ/Ⅱ/Ⅲ）"
  ├── capability_tags: ["chem:hazmat-identification", "chem:msds-parsing"]
  └── condition_type: "human_decision"
```

---

### 1.4 Goal / Project / Task 三者区分

| 维度 | Goal | Project | Task |
|------|------|---------|------|
| **层级** | 战略目标 | 战略 | 战术 |
| **粒度** | 业务目标（"要解决什么问题"） | 场景（"哪个流程"） | 原子操作（"谁干什么"） |
| **创建方式** | 手动/外部 API | 从场景模板实例化或手动创建 | 从 Project 分解或手动添加 |
| **执行者** | 运营人员 | Project Owner | Agent 或人类 |
| **生命周期** | 完成后归档 | 运行时活跃 | 执行单元，完成即结束 |

## 场景库相关术语

### 2.1 场景库（Scene Library）

**定义**：Nexus 系统中所有场景模板的集合，提供场景的浏览、搜索、安装、管理功能。

**组成**：
- 场景模板列表
- 行业包维度分类
- 搜索和筛选能力
- 场景安装/卸载管理

**与行业包的关系**：场景库从行业包中获取场景模板，行业包是场景的打包发布形态。

---

### 2.2 场景模板（Scenario Template）

**定义**：预定义的业务流程骨架，包含 Task 列表、依赖关系、HITL 配置、能力标签声明。

**特征**：
- 不绑定具体执行数据（参数在实例化时填充）
- 包含完整的业务流程定义
- 可以声明所需的 Agent 角色配置

**数据结构**（简化）：
```yaml
scenario_template:
  id: "sce-leak-response-001"
  name: "危化品泄漏处置"
  description: "化工园区危化品泄漏标准响应流程"
  version: "1.0.0"
  
  tasks:
    - id: "task-001"
      name: "识别危化品类型"
      requires: ["chem:hazmat-identification"]
      condition_type: "none"
      
    - id: "task-002"
      name: "评估泄漏等级"
      requires: ["chem:emergency-response-level"]
      condition_type: "human_decision"
      depends_on: ["task-001"]
```

---

### 2.3 场景实例（Scenario Instance）

**定义**：场景模板在特定上下文（Goal + 参数）下实例化后的执行实体，对应一个 Project。

**特征**：
- 绑定具体执行参数（泄漏物质、泄漏量、气象条件等）
- 包含运行时 Task 实例列表
- 有一个 Project 与之对应
- 状态随执行推进变化

**与场景模板的关系**：
```
场景模板 →（实例化）→ 场景实例（Project）
  抽象流程定义        绑定参数的执行实体
```

---

### 2.4 三者区分

| 维度 | 场景库 | 场景模板 | 场景实例 |
|------|--------|---------|---------|
| **性质** | 场景的集合管理功能 | 业务流程的蓝图/模板 | 模板在运行时实例化的结果 |
| **生命周期** | 持久存储 | 版本化，可发布 | 执行结束后归档 |
| **参数** | 无 | 无（占位符） | 有（实例化时填充） |
| **执行状态** | 无 | 无 | 有（running/done/failed） |

---

## 智能体与技能相关术语

### 3.1 智能体（Agent）

**定义**：Nexus 系统中实际执行 Task 的 AI 实体，通过 API 与外部 AI 服务通信。

**特征**：
- 有一个 Agent Profile（配置和能力描述）
- 声明自己拥有哪些能力标签及权重
- 从任务池领取 Task 并执行
- 通过心跳机制汇报执行状态

**与人的关系**：
- Agent 可以绑定到具体人员（如总指挥 Agent → 张总）
- 绑定后，HITL 节点的通知发送到对应人员

---

### 3.2 技能（Skill）

**定义**：一组相关能力的集合，通常对应一个具体的工具或工作流。

**特征**：
- 有明确的能力边界
- 包含工具（Tools）列表
- 包含验证脚本（Verification）
- 可以声明提供（provides）哪些能力标签

**示例**：
```yaml
skill:
  name: "MSDS Parser"
  provides: ["chem:msds-parsing"]
  tools:
    - name: "msds-parser"
      type: "python"
      entry_point: "msds_parser.py:parse"
  verification:
    - type: "unit_test"
      script: "test_msds_parser.py"
```

---

### 3.3 能力标签（Capability Tag）

**定义**：描述 Agent 能力的标准化字符串标签，用于 Task 与 Agent 的匹配。

**特征**：
- 两层体系：通用四维 + 行业标签
- 格式：`{维度前缀}:{能力标识符}`
- 可带权重（Agent 级别）

**示例**：
```
chem:hazmat-identification     # 行业标签（化工：危化品识别）
biz:chemical-park-operations  # 业务标签（化工园区运营）
tech:msds-parsing            # 技术标签（MSDS 解析）
mgmt:incident-command        # 管理标签（事故指挥）
```

---

### 3.4 三者区分

| 维度 | 智能体（Agent） | 技能（Skill） | 能力标签（Capability Tag） |
|------|---------------|-------------|------------------------|
| **定义** | 执行 Task 的 AI 实体 | 能力的工具化封装 | 能力的标准化描述 |
| **粒度** | 大（包含多个技能） | 中（一组相关能力） | 小（单一能力点） |
| **运行时** | 领取并执行 Task | 被 Task 调用 | 匹配引擎使用 |
| **是否可选** | 必须（有才能执行） | 可选（无则纯 AI 推理） | 必须（有才能匹配） |

## 行业包相关术语

### 4.1 行业包（Industry Pack）

**定义**：Nexus 的核心商业交付形态，包含场景、技能、知识、Agent 方案、能力标签的完整行业解决方案。

**组成**：
```
行业包 = 能力标签库 + 场景 + 技能 + 知识 + Agent方案
```

**示例**：化工应急行业包
- **能力标签**：chem:hazmat-identification、chem:diffusion-modeling 等
- **场景**：危化品泄漏处置、储罐火灾、有毒气体扩散
- **技能**：MSDS 解析器、扩散计算器
- **知识**：《危险化学品事故应急救援预案编制导则》
- **Agent 方案**：现场评估 Agent、决策支持 Agent、预案生成 Agent

---

### 4.2 定制包（Customization Pack）

**定义**：基于标准行业包，为特定客户定制的行业包扩展。

**特征**：
- 基于某个标准行业包（base_pack）扩展
- 定制内容：新标签、新场景、新知识、新参数
- 不修改标准包内容，共享标准包的通用能力
- 有自己的版本体系（独立于标准包）
- 格式：`.nexus`（实际是 ZIP 压缩包）

**与标准包的关系**：
```
标准行业包（化工应急）
  ├── 通用场景（泄漏/火灾/扩散）
  ├── 通用标签（chem:xxx）
  └── 通用知识（通用应急指南）
       ↓ 定制
定制行业包（XX化工园区）
  ├── 新增场景（XX园区特殊预案）
  ├── 新增标签（chem:xxpark:xxx）
  └── 设施参数（储罐坐标、敏感目标）
```

---

### 4.3 两者区分

| 维度 | 行业包（Industry Pack） | 定制包（Customization Pack） |
|------|----------------------|---------------------------|
| **性质** | 通用行业解决方案 | 客户专属定制扩展 |
| **发布者** | Nexus 官方/合作伙伴 | 咨询团队/合作伙伴 |
| **版本管理** | 官方版本号（语义化） | 客户定制版本号 |
| **依赖** | 独立存在 | 依赖某个标准行业包 |
| **分发方式** | 安装包形式 | .nexus 文件导入 |
| **定价** | ¥5-10 万/年（标准包） | ¥10-50 万（定制包） |

## HITL（人在回路）术语

### 5.1 HITL 概述

**定义**：Human-in-the-Loop，人在回路。AI 执行过程中，关键节点交由人类决策或输入，确保 AI 不会在重要环节走偏。

**核心价值**：不是拖慢效率，是让 AI 不会走偏。

---

### 5.2 待裁决（human_decision）

**定义**：Task 执行前需要人类审批/拍板，AI 给出方案，人类决定是否执行或如何调整。

**典型场景**：
- "是否启动 Ⅰ 级应急响应？"
- "AI 建议方案 A 或 B，由人拍板"
- "代码重构方案需要技术负责人审批"

**执行流程**：
```
Task 创建 → 检查 condition_type = human_decision
  → Task 状态 = waiting_human
  → 飞书通知审批人
  → 人类审批（同意/拒绝/修改）
  → Task 状态 = todo，继续执行
  → DependencyResolver 解锁下游任务
```

---

### 5.3 待审批（human_input）

**定义**：Task 执行过程中需要人类提供关键数据/信息，AI 无法自主获取。

**典型场景**：
- "当前下风向风力等级是多少？"
- "泄漏点附近学校有多少学生？"
- "这个接口改动你们 team 是否认可？"

**执行流程**：
```
Task 执行中 → 遇到 condition_type = human_input 的节点
  → Task 状态 = waiting_human
  → 创建 human_input_request（含 branches/选项）
  → 飞书通知相关人员
  → 人类输入数据
  → Task 继续执行，使用人类提供的数据
  → DependencyResolver 解锁下游任务
```

---

### 5.4 待输入（human_input）vs 待裁决（human_decision）

| 维度 | 待裁决（human_decision） | 待输入（human_input） |
|------|------------------------|---------------------|
| **介入时机** | 执行前（AI 给出方案） | 执行中（AI 需要数据） |
| **人类动作** | 审批（同意/拒绝/修改方案） | 输入（提供数据/信息） |
| **AI 状态** | 暂停等待 | 暂停等待 |
| **典型示例** | "是否启动 Ⅰ 级响应？" | "当前风速几级？" |
| **数据结构** | 单次决策 | 可含分支选项（branches） |

## 验证与仲裁相关术语

### 6.1 验证（Verification）

**定义**：Task 执行完成后，通过预设的验证脚本自动检查输出是否正确。

**触发时机**：Task 状态变为 done 后，验证器自动运行。

**验证方式**：
- 单元测试脚本（unit_test）
- 规则匹配（regex / JSON schema）
- AI 自动评估（AI eval）
- 人工确认（manual）

**验证结果**：
- `passed`：输出符合预期
- `failed`：输出不符合预期，触发重试或升级

---

### 6.2 仲裁（Ruling）

**定义**：Task 验证失败后，由仲裁机制决定后续处理方式。

**仲裁决策**：
- `retry`：重新执行 Task（最多 N 次）
- `skip`：跳过该 Task，继续执行下游
- `escalate`：升级给人工处理

**仲裁依据**：
- 验证失败的原因类型
- 重试次数统计
- Task 的关键程度

---

### 6.3 裁决（Adjudication）

**定义**：人工对异常 Task 做最终处理决定，是最高级别的人工干预。

**与仲裁的区别**：
- **仲裁（Ruling）**：自动化决策，基于规则
- **裁决（Adjudication）**：人工决策，人工拍板

**典型场景**：
- AI 执行严重偏离预期，需要人工接管
- 多次重试均失败，需要人工决定是否继续
- 涉及合规/法律责任，必须人工确认

---

### 6.4 三者区分

| 维度 | 验证（Verification） | 仲裁（Ruling） | 裁决（Adjudication） |
|------|--------------------|---------------|---------------------|
| **执行者** | 机器（自动） | 机器（自动） | 人类 |
| **时机** | Task 完成后 | 验证失败后 | 仲裁无法处理后 |
| **动作** | 检查输出正确性 | 决定后续处理方式 | 人工最终决策 |
| **可配置** | 是（验证脚本） | 是（仲裁规则） | 降级场景 |

## 执行模式术语

### 7.1 探索模式（Exploration Mode）

**定义**：Agent 在相对宽松的约束下自主探索解决方案，允许尝试多种路径。

**特征**：
- 不强制按预设流程执行
- Agent 可以自主规划执行路径
- 验证较为宽松，主要关注结果而非过程
- 适用于：需求不明、方案待定、创新类任务

**示例**：
```
Goal: "帮我分析一下这个代码库的安全风险"
  → Agent 自由探索代码结构
  → 多种分析路径并行尝试
  → 输出综合分析报告
```

---

### 7.2 验证模式（Verification Mode）

**定义**：Agent 严格按照预设的场景流程执行，每一步都在验证控制之下。

**特征**：
- 必须按 DAG 顺序执行（依赖关系强制）
- 每个 Task 有明确的输入/输出规范
- HITL 节点强制暂停
- 验证失败可能阻断后续执行
- 适用于：合规要求高、流程标准化、执行确定性要求高的任务

**示例**：
```
化工应急场景
  → Task 1: 识别泄漏物质（自动）
  → Task 2: 评估泄漏等级（HITL 审批）
  → Task 3: 计算扩散范围（自动，但依赖 Task 1/2 结果）
  → Task 4: 制定疏散方案（HITL 输入：风向风力）
  → Task 5: 启动应急响应（HITL 审批）
  每一步都有记录，可追溯
```

---

### 7.3 两者区分

| 维度 | 探索模式 | 验证模式 |
|------|---------|---------|
| **执行路径** | Agent 自主规划 | 按预设 DAG 执行 |
| **灵活性** | 高（可动态调整） | 低（严格按流程） |
| **适用场景** | 创新、探索、研究 | 合规、生产、标准化流程 |
| **可追溯性** | 低（过程灵活） | 高（每步有记录） |
| **HITL 依赖** | 弱（可绕过） | 强制（HITL 节点必须处理） |

## 能力标签维度

Nexus 采用四维能力标签体系，每个维度代表一种类型的能力。

### 8.1 business（业务维度）

**定义**：与特定业务领域相关的知识和经验，不涉及具体技术实现。

**标签前缀**：`biz:`

**示例**：
| 标签 | 名称 | 说明 |
|------|------|------|
| `biz:chemical-park-operations` | 化工园区运营 | 化工园区的日常运营管理知识 |
| `biz:emergency-management` | 应急管理体系 | 应急管理体系的建立和运作知识 |
| `biz:software-engineering` | 软件工程 | 软件开发全流程管理知识 |

---

### 8.2 professional（专业维度）

**定义**：需要专业训练才能获得的技能，通常与行业实践密切相关。

**标签前缀**：`prof:` 或直接行业前缀

**示例**：
| 标签 | 名称 | 说明 |
|------|------|------|
| `chem:hazmat-identification` | 危化品识别 | 识别危险化学品类型、危害类别 |
| `chem:diffusion-modeling` | 扩散建模 | 气体/液体扩散范围计算和建模 |
| `chem:evacuation-planning` | 疏散规划 | 应急疏散方案制定 |
| `dev:security-audit` | 安全审计 | 代码和系统的安全漏洞审查 |

---

### 8.3 technical（技术维度）

**定义**：具体技术工具、框架、系统的操作能力。

**标签前缀**：`tech:`

**示例**：
| 标签 | 名称 | 说明 |
|------|------|------|
| `chem:msds-parsing` | MSDS 文档解析 | 解析 Material Safety Data Sheet |
| `chem:leak-rate-calculation` | 泄漏速率计算 | 使用泄漏计算工具 |
| `tech:github-actions` | GitHub Actions | CI/CD 流水线配置 |
| `tech:pytest` | Pytest 测试框架 | 编写和运行 pytest 测试 |

---

### 8.4 management（管理维度）

**定义**：组织协调、资源调配、人员管理的综合能力。

**标签前缀**：`mgmt:` 或行业前缀

**示例**：
| 标签 | 名称 | 说明 |
|------|------|------|
| `chem:incident-command` | 事故指挥 | 应急事故现场指挥能力 |
| `chem:resource-allocation` | 资源调配 | 应急资源调度和分配 |
| `chem:multi-agency-coordination` | 多部门协调 | 跨部门/跨机构协调沟通 |
| `dev:tech-debt-management` | 技术债管理 | 技术债务识别和优先级管理 |

---

### 8.5 四维总览

| 维度 | 前缀 | 含义 | 示例 |
|------|------|------|------|
| `business` | `biz:` | 业务领域知识 | 化工园区运营、应急管理 |
| `professional` | 行业前缀如 `chem:` | 专业技能/行业专长 | 危化品识别、扩散建模 |
| `technical` | `tech:` | 技术工具/框架能力 | MSDS 解析、CI/CD 配置 |
| `management` | `mgmt:` | 管理/协调能力 | 事故指挥、资源调配 |

## 其他核心术语

### 9.1 场景包文件（.nexus）

**定义**：行业包的打包分发格式，实际是一个 ZIP 压缩包。

**内部结构**：
```
custom-pack-v1.0.nexus（ZIP）
├── manifest.json          # 包元数据
├── tags.yaml              # 能力标签定义
├── scenarios/             # 场景文件
├── skills/                # 技能配置
├── knowledge/             # 知识库
├── agent-profiles/        # Agent 方案
├── facilities/            # 设施参数
└── org-chart/             # 组织架构映射
```

---

### 9.2 能力标签匹配引擎

**定义**：Nexus 的核心调度组件，负责将 Task 与最合适的 Agent 进行匹配。

**匹配逻辑**：
```
匹配度 = (Agent 拥有的标签 ∩ Task 需要的标签) 的权重之和 / 总权重
```

**匹配优先级**：
1. 行业标签优先于通用四维标签
2. 权重高的 Agent 优先
3. 标签交集越大，匹配度越高

---

### 9.3 DependencyResolver

**定义**：Nexus 的依赖解析和任务解锁组件。

**职责**：
- 监听 Task 完成事件
- 检查下游 Task 的依赖是否全部满足
- 满足则解锁（状态从 blocked → todo）
- 支持手动解锁和自动解锁

---

### 9.4 能力标签四维体系与行业标签的关系

```
┌──────────────────────────────────────────┐
│           通用四维标签（所有行业共用）        │
│  business / professional / technical /      │
│  management                               │
└────────────────────┬─────────────────────┘
                     │ 映射到
┌────────────────────┴─────────────────────┐
│           行业能力标签库（每个行业一套）       │
│  chem:hazmat-identification               │
│  dev:python-code-review                   │
│  medical:diagnosis-analysis               │
└──────────────────────────────────────────┘

行业标签必须映射到四维之一，保证跨行业 Agent 可以部分匹配。
```
