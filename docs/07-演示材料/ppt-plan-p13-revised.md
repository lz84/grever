# P13 修订版：智能体 间协商与争议可视化

---

## 页面标题

**智能体 间协商与争议可视化**

---

## 核心问题（为什么需要协商）

**多智能体 协同中，意见不一致是常态，不是例外。**

### 现有框架的困境

| 问题 | 后果 | 我们的痛点 |
|------|------|-----------|
| 没有协商机制 | 只能人工裁判 | 效率极低，人类变成保姆 |
| 没有争议记录 | 不知道 智能体 为什么分歧 | 人类需要重新调查，浪费时间 |

### 我们的真实场景举例

```
场景 1：技术选型
├── 麻子主张：Playwright（功能强，支持多浏览器）
└── 谷子主张：Scrapy（速度快，适合大规模爬取）

场景 2：数据格式
├── 人类偏好：CSV（可直接打开查看）
└── 智能体 偏好：JSON（机器友好，嵌套结构）

场景 3：任务优先级
├── 技术视角：先做 A（有依赖关系）
└── 业务视角：先做 B（明天就要用）
```

## 解决方案（协商机制与可视化）

### 协商四步机制

```
┌─────────────────────────────────────────────────────────────┐
│  步骤 1: 各方提案                                           │
│  • 每个智能体 提交自己的方案                                │
│  • 必须带论据（为什么选这个）                               │
│  • 必须带约束（有什么前提条件）                             │
├─────────────────────────────────────────────────────────────┤
│  步骤 2: 相互修正                                           │
│  • 阅读其他智能体 的方案                                    │
│  • 寻找共同点（我们都认同的部分）                           │
│  • 提出妥协方案（能不能各退一步）                           │
├─────────────────────────────────────────────────────────────┤
│  步骤 3: 达成共识                                           │
│  • 所有智能体 都接受 → 执行共识方案                         │
│  • 记录决策依据（下次遇到类似问题可复用）                   │
├─────────────────────────────────────────────────────────────┤
│  步骤 4: 超时升级                                           │
│  • 超过时限仍无法共识 → 推送给人类仲裁                      │
│  • 自动打包争议快照（见下方可视化）                         │
└─────────────────────────────────────────────────────────────┘
```

### 争议可视化（关键创新）

**人类是仲裁者，不是调查员** —— 直接做选择题，不是问答题

```
┌─────────────────────────────────────────────────────────────┐
│  📋 争议快照（自动打包推送给人类）                          │
├─────────────────────────────────────────────────────────────┤
│  争议焦点：技术选型 - 网页爬虫框架                          │
├─────────────────────────────────────────────────────────────┤
│  麻子方案：Playwright                                       │
│  • 论据：功能强，支持多浏览器，可处理动态渲染              │
│  • 约束：需要额外安装浏览器依赖                            │
│  • 适用场景：复杂网页、需要登录的站点                      │
├─────────────────────────────────────────────────────────────┤
│  谷子方案：Scrapy                                           │
│  • 论据：速度快，适合大规模爬取，生态成熟                  │
│  • 约束：处理动态渲染需要额外配置                          │
│  • 适用场景：静态页面、大批量数据抓取                      │
├─────────────────────────────────────────────────────────────┤
│  已尝试的妥协方案：                                         │
│  1. 混合使用（Playwright 处理登录 + Scrapy 批量爬取）        │
│     └── 麻子：同意，但担心复杂度                            │
│     └── 谷子：反对，维护成本高                              │
│  2. 统一用 Playwright                                       │
│     └── 谷子：反对，性能损失太大                            │
├─────────────────────────────────────────────────────────────┤
│  智能体 推荐建议：                                          │
│  🎯 推荐方案：混合使用（麻子让步 20%，谷子让步 30%）         │
│  • 理由：本次任务 80% 是静态页面，20% 需要登录                │
│  • 风险：需要额外开发集成代码（预估 2 小时）                  │
├─────────────────────────────────────────────────────────────┤
│  [✅ 批准推荐方案]  [❌ 选择麻子方案]  [❌ 选择谷子方案]      │
│  [📝 自定义方案]                                            │
└─────────────────────────────────────────────────────────────┘
```

### 争议可视化的价值

| 传统方式 | 争议可视化 |
|---------|-----------|
| 人类需要逐个询问 智能体 | 自动打包所有信息 |
| 人类需要整理分歧点 | 争议焦点自动提炼 |
| 人类需要评估各方论据 | 论据已结构化呈现 |
| 人类需要自己想方案 | 智能体 已提供推荐建议 |
| **人类是调查员 + 仲裁员** | **人类只是仲裁员** |

## 演讲要点（1 分钟）

**问题（20 秒）**：
"多智能体 协同中，意见不一致是常态。现有框架没有协商机制，只能人工裁判，效率极低。"

**方案（30 秒）**：
"我们设计了四步协商机制：提案→修正→共识→超时升级。关键创新是争议可视化——自动打包争议焦点、各方论据、已尝试的妥协方案、推荐建议，推送给人类。人类只需做选择题，不是问答题。"

**价值（10 秒）**：
"人在回路中，但不必事事亲为——人类做仲裁，不是当保姆。"
