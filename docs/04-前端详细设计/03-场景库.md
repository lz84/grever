# 场景库 - 详细设计

> 路由前缀：`/scenarios`
> 对应：达(Reach) — 场景实例化、数据背书、反馈回收
> 核心理念：每个场景是活的，带有运行数据背书。实例化 → 修改 → 执行 → 反馈 → 优化 → 回收为新版本或新分支。
> 子页面：场景列表、场景详情、收藏

---

### 1.2 字段定义

| 字段 | 数据来源 | 显示格式 | 交互 |
|------|----------|----------|------|
| 场景名称 | `scenario.name` | 文本，最多 20 字 | 点击 → `/scenarios/:id` |
| 分类 | `scenario.category` | 标签：地震(红)/火灾(橙)/化学品(紫)/防汛(蓝)/通用(灰) | |
| 状态 | `scenario.status` | 标签：活跃(绿)/归档(灰)/草稿(橙) | |
| 版本 | `scenario.version` | 文本 `v2.1` | |
| 成功率 | 从执行记录计算：`成功次数/总执行次数` | 百分比 | 90%+ 绿色，70-89% 蓝色，50-69% 橙色，<50% 红色 |
| 平均耗时 | 从执行记录计算：平均 `completed_at - started_at` | `x 分钟` | |
| 使用次数 | `scenario.usage_count` 或执行记录计数 | 数字 + `次` | |

| `chemical` | 化学品 | 紫色 |
| `flood` | 防汛 | 蓝色 |

### 1.4 筛选与排序

| 筛选项 | 选项 | 说明 |
|--------|------|------|
| 分类 | 全部 + 从场景数据中提取 | |
| 状态 | 全部/活跃/归档/草稿 | |
| 排序 | 成功率/使用次数/平均耗时/更新时间 | 默认成功率降序 |
| 搜索 | 输入关键词 | 300ms 防抖，过滤名称、描述 |

### 1.5 API 调用

| 操作 | 方法 | 端点 | 说明 |
|------|------|------|------|
| 加载场景列表 | GET | `/api/v1/scenarios` | |
| 执行统计数据 | GET | `/api/v1/workflows?scenario_id={id}` | 批量获取，前端按 `scenario_id` 分组计算 |

> **关键**：成功率和平均耗时通过执行记录计算。一次性获取所有相关 workflow 数据，前端按 `scenario_id` 分组统计。**禁止对每个场景单独调用 API。**

### 2.4 收藏功能

**存储**：`localStorage`，Key: `nexus_starred_scenarios`

**结构**：`{ [scenarioId]: { name, category, status, version, description, starredAt } }`

**描述长度限制**：最多存储前 100 字符，完整描述从 API 获取

**操作**：
- 收藏：写入 localStorage → 图标变 ⭐
- 取消：删除 → 图标变 ☆
- 页面加载时检查 localStorage

### 2.5 实例化路径

```
场景列表/详情 → [实例化此场景]
    ↓
跳转 /coordination/goals/new?scenario_id={id}
    ↓
CreateGoal 页面预填：
  - 目标名称 = 场景名称
  - 描述 = 场景描述
  - 协作步骤 = 场景协作步骤（可修改）
  - 顶部显示「📋 来源场景：{场景名称} [修改来源]」
    ↓
用户根据实际情况修改 → 提交 → 创建成功 → 跳转目标详情
    ↓
执行完成后，反馈数据回写到场景库：
  - 成功率更新
  - 平均耗时更新
  - 如有用户修改，生成新版本或新分支
```

**CreateGoal 页面 URL 参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `scenario_id` | string | 场景 ID |

**预填行为**：
- 加载时调用 `GET /api/v1/scenarios/{scenario_id}`
- 预填目标名称、描述
- 预填协作步骤到目标详情页面的「项目区块」（每个 step 创建一个子项目）
- 如果场景不存在（404）：显示 Toast `场景不存在或已被归档`，不预填

**协作步骤映射**：
场景的 `steps[]` → 目标详情页面的「项目区块」：
- 每个 `ScenarioStep` 创建一个默认项目
- 项目名称 = step.name
- 项目描述 = `agent_type` + `required_capabilities` 的组合描述
- 用户可修改/删除/新增项目

## 6. API 返回结构定义

### 6.1 场景列表 API

**端点**：`GET /api/v1/scenarios`

**查询参数**：
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `category` | string | ❌ | 按分类过滤 |
| `status` | string | ❌ | 按状态过滤 |
| `q` | string | ❌ | 搜索关键词 |
| `sort` | string | ❌ | 排序字段：`success_rate`/`usage_count`/`avg_duration`/`updated_at` |
| `order` | string | ❌ | `asc` 或 `desc`，默认 `desc` |
| `limit` | number | ❌ | 默认 20 |
| `offset` | number | ❌ | 默认 0 |

**返回结构**：
```typescript
interface ScenarioListResponse {
  total: number;
  items: ScenarioSummary[];
}

interface ScenarioSummary {
  id: string;
  name: string;
  category: 'earthquake' | 'fire' | 'chemical' | 'flood' | 'general';
  status: 'active' | 'archived' | 'draft';
  version: string;
  success_rate: number;      // 0-100
  avg_duration_ms: number;   // 毫秒
  usage_count: number;
  scenario_desc: string;     // 适用场景描述，最多 100 字
  created_at: string;        // ISO 8601
  updated_at: string;        // ISO 8601
}
```

### 6.2 场景详情 API

**端点**：`GET /api/v1/scenarios/{id}`

**返回结构**：
```typescript
interface ScenarioDetail {
  id: string;
  name: string;
  category: string;
  status: string;
  version: string;
  description: string;
  scenario_desc: string;
  triggers: string[];                    // 触发条件列表
  steps: ScenarioStep[];                 // 协作步骤
  metrics: ScenarioMetrics;              // 数据背书
  versions: string[];                    // 版本号列表，如 ["v1.0", "v1.5", "v2.0", "v2.1"]
  created_at: string;
  updated_at: string;
}

interface ScenarioStep {
  order: number;            // 步骤序号
  name: string;             // 步骤名称
  agent_type: string;       // 智能体类型
  required_capabilities: string[];  // 所需能力
}

interface ScenarioMetrics {
  total_executions: number;     // 总执行次数
  success_count: number;        // 成功次数
  failed_count: number;         // 失败次数
  avg_duration_ms: number;      // 平均耗时
  min_duration_ms: number;      // 最短耗时
  max_duration_ms: number;      // 最长耗时
  avg_conflicts: number;        // 平均冲突数
  avg_step_completion: number;  // 平均步骤完成率 0-100
}
```

### 6.3 反馈回收 API

**端点**：`POST /api/v1/scenarios/{id}/feedback`

**触发时机**：执行完成后，**系统自动调用**（前端不主动触发）。协同中心在执行完成时调用此端点。

**触发条件**：`workflow.status === 'completed' || workflow.status === 'failed'`

**请求体**：
```typescript
interface FeedbackRequest {
  workflow_id: string;
  status: 'completed' | 'failed';
  duration_ms: number;
  steps_completed: number;
  steps_total: number;
  conflicts_count: number;
  user_modifications?: Array<{   // 用户对场景的修改
    field: string;        // 修改的字段名
    old_value: any;
    new_value: any;
  }>;
}
```

**返回结构**：
```typescript
interface FeedbackResponse {
  success: boolean;
  new_version_suggested?: boolean;   // 是否建议保存为新版本
  updated_metrics?: ScenarioMetrics;  // 更新后的数据背书
  message?: string;
}
```

**错误处理**：
| HTTP 状态码 | 返回 | 处理 |
|-------------|------|------|
| 404 | `{ "error": "Scenario not found" }` | 记录日志，不重试 |
| 400 | `{ "error": "Invalid feedback data" }` | 记录日志，不重试 |
| 500 | `{ "error": "Internal error" }` | 记录日志，稍后重试 |

## 4. 场景反馈回收机制

### 4.1 回收流程

```
目标执行完成
    ↓
系统自动调用 POST /api/v1/scenarios/{id}/feedback
    ↓
请求体：
{
  "workflow_id": "xxx",
  "status": "completed",
  "duration_ms": 1920000,
  "steps_completed": 5,
  "steps_total": 5,
  "conflicts_count": 1,
  "user_modifications": [...]  // 用户对场景的修改记录
}
    ↓
场景库更新：
  - 成功率重新计算
  - 平均耗时重新计算
  - 如果有用户修改，提示"是否保存为新版本？"
```

### 4.2 版本管理

| 场景 | 处理方式 |
|------|----------|
| 执行成功，无修改 | 仅更新统计数据（成功率、耗时） |
| 执行成功，有用户修改 | 提示用户"是否保存为新版本？"→ 用户确认则生成新版本 |
| 执行失败 | 记录失败原因，不改变版本，仅更新失败率 |

### 4.3 分支管理

**触发条件**：用户修改了场景的协作步骤或触发条件

**规则**：
- 微调（修改参数值、智能体分配）→ 合入主线，更新版本号
- 重大修改（增删步骤、修改触发条件）→ 生成新分支，用户可选择是否合并回主线
